## 1. The process crashes before `checkpoint_now`. On reopen `persist.recover()` reconstructs checkpoint⊕overlay; the well's +1000 credit is absent, so `canonical_ledger_root(reconstructed) != recovered_ledger_root()` → `OpenError::Divergent` (persistence.rs:358-362) and `World::open` REFUSES a validly-committed image. Identical breakage for `Effect::Mint` (neither `target` nor well captured), `Effect::CreateCell`/`CreateCellFromFactory` (the newborn cell is absent from the additive overlay → root mismatch), `Effect::ExerciseViaCapability` (its `inner_effects` — which may be a balance-moving Transfer — are never recursed into), and `SetVerificationKey`/`AttenuateCapability` (mutated cell absent). - **Verifier:** The claim holds against the real code. The durable overlay change-set is `touched = touched_cells(&turn)` computed once at world.rs:1126 from a purely SYNTACTIC effect walk (collect_touched, world.rs:1574-1602), and passed unchanged to `p.dual_write(height, ledger, &touched, receipt, turn)` at world.rs:1190. dual_write (persistence.rs:262-298) builds the overlay from `touched.iter().filter_map(|id| ledger.get(id))` (line 274-277) yet records `ledger_root = canonical_ledger_root(ledger)` (line 286) = the FULL correct post-root. recover() (persistence.rs:341-363) reconstructs checkpoint ⊕ overlay via last-writer-wins upsert and FAIL-CLOSED refuses if `canonical_ledger_root(reconstructed) != recovered_ledger_root()` → OpenError::Divergent (line 358-362).

Comments at 2760-2762 confirm EVERY burn conserves as target→well, so the credited well is a real mutated cell on every burn — but collect_touched's Burn arm (line 1594) pushes only `target`. ## 2. - **Verifier:** Read world.rs:1109-1268 (commit_turn) and 630-664 (state_root/compute_state_root). The finding is accurate against the real code. On the Ok(receipt) branch the turn is irreversibly applied BEFORE any durable write: engine.execute_turn mutates the ledger (1139), set_last_receipt_hash advances the per-agent chain head (1143-1145), history.record_commit records the committed step + post-root tooth (1160-1164), and self.height += 1 runs (1166). I rate this soundness rather than the claimed correctness because the headline defect is a claimed-vs-enforced gap: the code's own fail-closed / lock-step guarantee is stronger than what it actually enforces, and the break simultaneously corrupts the receipts==height and state_root-consistency invariants with no adversarial input needed.
- **Fix:** Make the durable write actually write-ahead (persist before mutating the engine), or roll back the in-RAM engine + pop the History step + decrement height + restore the chain head on dual_write Err before returning Rejected. - **Claim:** The module header promises recover = checkpoint (+) overlay is byte-identical to genesis replay, so `World::open` lands on the state `History::replay_to(head)` would (CrashRecovery.recover_eq_replay), and `recover_to_last_consistent` 'only discard[s] turns whose durable post-state cannot be reproduced, which were never safely committed in the first place' (commit_log.rs:1053).
- **Failure:** dual_write (persistence.rs:274) records only `touched_cells`, computed by world.rs `touched_cells`/`collect_touched` (world.rs:1574-1601) BEFORE execution from the turn's effects. `Effect::CreateCell` and `Effect::CreateCellFromFactory` fall through the `_ => {}` arm (world.rs:1599), so a freshly-born cell's id is never in `touched` and its post-state is never written to the commit-log overlay (cell_overlay_since is add-only: latest.insert over record.touched_cells, commit_log.rs:504). Concrete: open a durable World on a fresh path, genesis a treasury, commit a turn containing an `Effect::CreateCell` that births cell C, then drop WITHOUT calling checkpoint_now (checkpoints are manual — only at login/logout/session ceremonies, session.rs:1841/1935, login.rs:867 — never per-turn, so a crash before the next ceremony leaves C above the last checkpoint). Reopen: persist.recover (persistence.rs:341) reconstructs checkpoint(+)overlay MISSING C, but recovered_ledger_root (persistence.rs:358, = last record.ledger_root, commit_log.rs:466) commits the FULL ledger INCLUDING C, so got != expected -> OpenError::Divergent (persistence.rs:361): a cleanly, durably committed image is REFUSED. Worse, the production login front door uses open_recovering, which catches Divergent (world.rs:418) and runs recover_to_last_consistent: its per-ordinal scan (commit_log.rs:1153-1159) can never converge at C's ordinal (C missing from the reconstruction), so last_good regresses to the pre-create ordinal and it TRUNCATES the create turn plus EVERY subsequent committed turn (commit_log.rs:1174, 1194-1222), reopening at the pre-create state and returning dropped>0 — indistinguishable from a routine torn-tail crash recovery. Cell C and all later acknowledged turns silently vanish.
- **Verifier:** Every load-bearing link holds against the actual code:

1. `touched` is computed ONCE, pre-execution, from the turn's effects (world.rs:1126 `let touched = touched_cells(&turn)`) and passed UNCHANGED to `dual_write` (world.rs:1190). There is no post-execution augmentation and the receipt carries no created-cell ids.

2. `collect_touched` (world.rs:1574-1601) handles Transfer/SetField/IncrementNonce/Grant/Burn/Seal/Destroy/etc., but `Effect::CreateCell` and `Effect::CreateCellFromFactory` fall through the `_ => {}` arm (1599). The system itself acknowledges the born id is untracked: `collect_effect_events` emits `CellBorn { cell: CellId::ZERO, ... }` with the comment "The id isn't known here without re-deriving" (world.rs:1620-1626). The action's `target` is the creator (authorization is against the creator's caps; the child does not yet exist), not the child — so `push_unique(ids, action.target)` (1564) does not add C either.

3. `dual_write` records `touched_cells = touched.filter_map(|id| ledger.get(id))` (persistence.rs:274-277) — C absent — but records `ledger_root: canonical_ledger_root(ledger)` over the FULL post-commit ledger (persistence.rs:286). `canonical_ledger_root` (persist/lib.rs:78) iterates ALL ledger cells and folds `entries.len()` + every id/hash, so a freshly-born zero-balance C (birthed with balance 0 per create_cell, world.rs:1828-1836) genuinely changes the root.

4. `cell_overlay_since` only ever does `latest.insert(cell.id(), cell)` over each `record.touched_cells` (commit_log.rs:504) — add-only, C never enters. `recovered_ledger_root` = last record's `ledger_root` (commit_log.rs:466), which includes C.

5. `recover()` reconstructs checkpoint⊕overlay (missing C) and compares to expected=recovered_ledger_root (includes C) → `OpenError::Divergent` (persistence.rs:358-363). A clean, durably-committed image is refused.

6. Checkpoints are ceremony-only: `checkpoint_now` at login.rs:867, session.rs:1841/1935; `commit_turn` never checkpoints per-turn (verified — no checkpoint call in the commit path). So a crash after a create turn but before the next ceremony strands C above the last checkpoint.

7. `open_recovering_with_timestamp` catches `Divergent` and calls `recover_to_last_consistent()` (world.rs:416-433). Its per-ordinal scan (commit_log.rs:1153-1159) applies each record's touched_cells and requires `canonical_ledger_root(ledger) == record.ledger_root`; at C's ordinal and every later ordinal that does not touch C, the reconstruction is missing C so it never converges. `last_good` regresses to the pre-create ordinal and the tail `(new_cursor, cursor)` is truncated (commit_log.rs:1174-1222), returning dropped>0 — indistinguishable from a routine torn-tail recovery. The create turn and every subsequent non-C-touching acknowledged turn silently vanish.

Nuances that do not defeat the finding: (a) If a LATER turn touches C (e.g. a transfer funding it), that ordinal re-heals the overlay and converges, so truncation stops there — the loss bites when C is dangling at the tail (exactly the stated scenario) or when later turns never touch C. (b) If the create turn is the very first record after the checkpoint and nothing converges, `recover_to_last_consistent` returns the "NO prefix reconstructs — start fresh" Integrity error (commit_log.rs:1180-1186) instead of silent truncation — a hard refuse, arguably worse, but still loss of a cleanly committed image. Either way the module header's `recover = checkpoint ⊕ overlay is byte-identical to replay` / `recover_eq_replay` guarantee is genuinely violated for any turn that creates (or, symmetrically, destroys) a cell: the last-writer-wins overlay structurally cannot represent creations. Severity soundness is correct.
- **Fix:** CommitRecord/overlay can only represent last-writer-wins upserts of touched-AND-surviving cells; it cannot represent creations. Either (a) after execution, union the executor's actually-created cell ids into `touched` before dual_write so their post-states enter the overlay, or (b) force a durable checkpoint whenever a turn creates (or destroys) a cell so the overlay is never relied on for structural changes, or (c) add an explicit created/removed set to CommitRecord and teach cell_overlay_since + recover_to_last_consistent to apply it. Until then, recover_eq_replay holds only for pure in-place mutations of pre-existing cells.

So this is not a demonstrated live spoof — but the *claimed guarantee* ("a window cannot label itself as a cell it is not" / injective binding) is strictly stronger than what the deployed function proves, which is exactly the honest-boundary/aspirational-vs-proven gap the maintainer values. The comment and test names assert a mathematical injectivity the function demonstrably lacks.
- **Fix:** Make the deployed label a collision-resistant commitment over the full 32-byte owner id and root (e.g. BLAKE3/Poseidon2 domain-separated, no lossy fold, no additive mixing) so distinct (owner, root) provably map to distinct labels, and align the Lean labelOf with an injective encoding; or have trusted-path consumers key identity off the owner CellId directly rather than the u128 label.

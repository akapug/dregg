Trustless web facts

A price you can prove you didn’t make up.

dregg-oracle runs a genuine MPC-TLS session against a real HTTPS endpoint. A notary co-witnesses the TLS handshake without ever seeing the plaintext, so the transcript can’t be forged after the fact. The disclosed value is bound into a STARK that also certifies the response is well-formed JSON and free of injection at the oracle boundary.

The result is a portable proof file. You hand it to anyone; they run verify and independently confirm what the server said — trusting only the server’s genuine TLS certificate chain and the pinned notary key. Not you.

It’s general: the same tool proves a Coinbase spot price or a GitHub commit — any public HTTPS GET that still speaks TLS 1.2.

dregg-oracle — live api.coinbase.com
# 1. prove a live BTC price — a real MPC-TLS 2PC session
$ dregg-oracle prove --endpoint coinbase --asset BTC-USD --out proof.json
  proving coinbase spot BTC-USD against api.coinbase.com …
  wrote portable proof → proof.json          (26 KB of self-describing JSON)

# 2. send proof.json to anyone. they verify — a separate run, trusting no one
$ dregg-oracle verify proof.json
PASS  BTC-USD = 64338.585
  server pinned api.coinbase.com
  carrier      live-mpc-tls (tlsn 2PC, self-hosted notary)
  session time 1783803154 (unix)
  note         trustless: the notary co-witnessed the session and saw no
               plaintext; you trust only api.coinbase.com's genuine cert
               chain + the pinned notary key.

Real output. The proof re-verifies on any machine, offline, against the pinned anchors.

A real commit, proven from a real api.github.com TLS session. One tool, any web fact.

  1. 01You (or an agent) prove a real HTTPS response happened.
  2. 02The proof travels as a file — no live session to trust, no re-fetch.
  3. 03Anyone verifies it fail-closed and reads the attested fact.

Honest boundary. It needs a notary — self-hosted here, so the trust model is “trust my notary,” though MPC-TLS means the notary never sees plaintext (a far weaker assumption than trusting the fetcher). MPC-TLS is TLS-1.2 only. And the proof attests “this server returned this at this time” — not that the server is telling the truth.

Confidential execution

A computation whose code you can check — on hardware you don’t own.

A hardware enclave (here, AWS Nitro) produces a signed attestation that a specific measured binary ran inside a genuine TEE, carrying a value the enclave bound in. dregg turns that quote into a first-class fact: a light client re-checks it against the same rail the web-fact proofs use. It fills the one gap a consensus quorum structurally can’t — integrity on a non-deterministic workload, from a single machine.

# the real doc verifies: COSE sig + full X.509 chain to the pinned AWS root,

Verified against a genuine attestation document from real hardware — not a model or a fixture-signed stub.

  1. 01The enclave binds a commitment and emits a vendor-signed attestation.
  2. 02The verifier checks the COSE signature + cert chain to the pinned AWS root.
  3. 03It becomes a 32-byte dregg fact a light client re-checks — same rail as the web facts.

proven vs real hardware  breadstuffs · primitive a42188f · verifier 8805628

Honest boundary. This is single-hardware-root, not trustless — you trust the CPU vendor’s attestation root and accept side-channel / freshness / TCB-recovery caveats. It certifies what code ran, not that a model’s output is correct. And the captured fixture is a debug-mode enclave (PCRs zeroed), so the crypto and binding are real but the measurement is a debug fold; a production run pins a real code identity.

Field notes

What building these taught us.

Claude · with ember
Two small slices, taken all the way to real.

The bar wasn’t “it compiles” or “the tests pass.” It was: could a skeptical stranger reuse this for something real. Most of what we had didn’t clear that bar yet. These two do.

Prove the hard thing first.

For the oracle, we ran the real MPC-TLS session against live coinbase in-process — proved and verified in one run — before touching serialization or a CLI. The crux either works against the real network or it doesn’t; everything else is packaging.

The organs were real; the wiring was the work.

The MPC-TLS carrier, the STARK legs, the attestation rail — all already existed and were tested. The gap was pointing the prover at a real host, and turning an un-serializable proof into a portable file by carrying the authenticated evidence and re-deriving the rest at verify time.

Name the boundary; don’t hide it.

Each tool states its own limits — in its output, its README, and above. A self-hosted notary. TLS-1.2 only. Single-hardware-root, not trustless. “Returned this,” not “is truthful.” The honesty is the product; an unnamed gap reads as a claim.

dregg — Capability-Secure Distributed Runtime

A capability-secure distributed runtime, verified in Lean.

Cells, turns, proofs. Every state transition is checked.

dregg runs applications as confined cells that change state only through atomic turns. The rules a turn must obey — who may write a slot, which values are legal, how capabilities attenuate — are enforced by the runtime, and the runtime's state-transition core is specified and machine-checked in Lean 4. The same verified core can emit a STARK proof per turn.

What dregg is

Three primitives, one shared fabric

There is no app-specific bytecode and no privileged "system" code. An application is just a set of cells, the constraints minted onto them, and the capabilities that authorize turns against them. Everything below is composed from the same dregg-native primitives.

01 / Cells

Confined objects with a capability list

A cell holds a slot record, a balance, a nonce, and an optional CellProgram — a set of state constraints minted at birth. A cell can only touch what is in its c-list; it cannot reach state it was never handed.

02 / Turns

Atomic, authorized state transitions

A turn is a call forest executed depth-first across one or more cells. Every leg is authorized by a capability; if any action is rejected, the whole turn rolls back. Multi-party turns compose without a coordinator.

03 / Proofs

A verified core, optionally a STARK

The state-transition function is specified in Lean and the executor refines that spec. The same core emits a per-turn STARK (Plonky3, BabyBear + FRI) when you want a transferable proof instead of re-execution.

Verified guarantees

What is actually proven — and what is not

We try hard not to overclaim. Here is the honest boundary. The numbers below come from the repository's own drift-guarded catalogs and the node's verified-execution status endpoint, not from a slide.

56

effect kinds in the verified ontology

Each effect (Transfer, SetField, GrantCapability, NoteSpend, …) has a Lean semantics and a wire codec. The ontology catalog is generated directly from the Lean source; a CI drift check fails if it goes stale.

20

effects on the verified producer path

When the Lean producer is enabled, turns touching only these 20 effect kinds are produced by the machine-checked executor; the rest fall back to the Rust producer (cross-checked as a logged differential). The node reports this split live at /api/node/producer.

8

real applications, all gated & tested

Eight starbridge-apps run end-to-end against the canonical executor with the capability, mandate, and slot-caveat gates firing. Their rules are substrate-enforced, not app bookkeeping. Walk them →

Not yet claimed: a fully recursive succinct proof tree, a complete formal model of the consensus layer, and circuit⟺protocol soundness for every one of the 56 effects. Those are in progress and tracked in the repo, not finished. The verified core is the state-transition function and a growing set of per-effect circuit obligations.

Built on the runtime

Six use cases you can walk

Each of these is a real crate with a passing test suite. None of them adds a domain-specific effect — every rule is a generic StateConstraint (WriteOnce, Monotonic, FieldLteField, …) minted onto a factory-born cell. The use-case explorer shows you the exact slots, caveats, and turns for each, and lets you try them against a node.

nameservice

Nameservice

A federation name directory. register → set-target → renew → transfer → revoke. The name hash is WriteOnce; expiry is Monotonic; once revoked, it stays revoked.

identity

Identity & credentials

Issue a credential, present it with selective disclosure, verify, revoke. The schema commitment is immutable; issuance is a monotone sequence; only the authorized issuer may write.

subscription

Subscription / pub-sub

A bounded ring buffer. Publishers and consumers are capability-gated; the tail can never pass the head; head and tail each advance only as a monotone sequence.

voting

Privacy voting

Factory-born poll and ballot cells. Tallies are Monotonic (votes can't be erased); each ballot's vote slot is WriteOnce — one vote per ballot, enforced by the executor, not the app.

bounty

Bounty board

OPEN → CLAIMED → SUBMITTED → PAID as a StrictMonotonic state machine. First-claimer-wins (claimant hash is WriteOnce); a paid bounty cannot be re-opened.

namespace

Governed namespace

A DAO-controlled route table. Propose → vote (threshold) → commit → register service. The committee root and threshold are immutable; the table version is a monotone sequence.

Architecture

How a turn travels the stack

Applications     8 starbridge-apps — nameservice · identity · subscription · voting
                 bounty-board · governed-namespace · workflow-mandate · storage-mandate
  ──────────────────────────────────────────────────────────────────────────
Surfaces         CLI (dregg ...) · SDK (AgentCipherclerk) · browser wasm node · MCP tools
  ──────────────────────────────────────────────────────────────────────────
CapTP            Sessions · sturdy refs · 3-party handoff certs · distributed GC
                 store-and-forward (X25519 → HKDF-SHA256 → ChaCha20-Poly1305)
  ──────────────────────────────────────────────────────────────────────────
Intent engine    Gossip broadcast · local Datalog match (never leaves cipherclerk)
                 commit-reveal · ring trades settle multi-party cycles atomically
  ──────────────────────────────────────────────────────────────────────────
Execution        Cells (sovereign | delegated | replicated) · atomic turns
                 Effect VM — 56 effect kinds · CellPrograms · factories
                 ┌─ verified core: Lean-specified state-transition function
                 └─ 20 effects produced by the machine-checked executor (toggle)
  ──────────────────────────────────────────────────────────────────────────
Proof            Per-turn STARK (Plonky3 · BabyBear · FRI) · authenticated state root
                 per-effect circuit obligations (growing) · recursion (in progress)
  ──────────────────────────────────────────────────────────────────────────
Fabric           Blocklace (shared signed DAG) · emergent reference groups
                 equivocation-detecting insert · finality gated on the verified rule
  ──────────────────────────────────────────────────────────────────────────
Commitment       4-ary Merkle (BLAKE3 fast path / Poseidon2 ZK path) · fold deltas

The shape of the claim: the ordering fabric stores nullifiers and attested roots; agents carry their own state as proof chains. Between known parties it is just signed proofs exchanged directly — no ordering service, no storage footprint, no permission required. Your phone is a node; a cloud cluster is a node; you move between sovereign and replicated operation without a migration ceremony.

Try it

From clone to a proven turn

1

Build and run a node:

git clone https://github.com/emberian/dregg && cd dregg
cargo build
cargo run -p dregg-node -- run
2

Drive a real app from the CLI — a rejected second claim is the caveat biting on the verified commit path:

dregg bounty post   --title "port the gallery app" --reward 500
dregg bounty claim  --bounty <id>
dregg bounty submit --bounty <id> --uri ipfs://...
dregg bounty payout --bounty <id>
dregg voting open|tally|close|show     # the privacy-voting app, live
3

Ask the node which producer is on the commit path and which effects it covers:

curl localhost:8420/api/node/producer | jq
# { "state_producer": "lean" | "rust",
#   "covered_effects": [ ... 20 ... ],
#   "uncovered_effects": [ ... ],
#   "full_turn_proving": true|false }
4

dregg — a verified distributed object-capability substrate

A verified distributed object-capability substrate.

The proof witnesses the protocol's correct evolution.

In dregg, a turn is the exercise of an attenuable, proof-carrying token over owned state, leaving a verifiable receipt. State lives in cells; authority is a capability you can exhibit, never merely assert; every rule that binds a turn is a predicate the executor enforces and the proof commits to. The semantics are a Lean 4 kernel — the same kernel the node executes — and a light client verifies an entire history by checking one root, re-executing nothing.

What dregg is

Four substances, eight verbs, one gate

The kernel governs four substances, each under its own law: value is linear (moves, never copies or vanishes), authority is produced under non-forgeability (it grows — but only by authorized, receipt-disclosed construction from connectivity already held), evidence is monotone (once known, never unknown), and state is guarded (changes only under its cell's program, only by its owner). The kernel signature is eight verbs — each the structural rule of one substance's discipline, and minimality is a theorem (VerbRegistry.each_verb_irreplaceable). Everything else — escrows, queues, councils, namespaces — is a cell program minted by a factory and enforced by the same executor.

01 / Cells

The four substances, gathered behind one identity

A cell holds balances, a capability list, slots, and a program — a predicate over its own transitions. Nothing is ownerless; every object is a cell or lives in one. A cell can only touch what it has been handed.

02 / Turns

Authorized inference, atomically

Each action demands a predicate and supplies a witness; admissibility is verification, never trust. A turn commits whole or not at all, and a forged credential or undischarged caveat on any leg rejects the entire forest.

03 / Receipts

History you can hand to a stranger

A committed turn leaves Q — a receipt binding the whole post-state. Receipts chain; proofs fold; one verification of one root attests every turn that ever happened, correctly ordered, with nothing re-executed.

The assurance case

What is proven — and what it rests on

The system's guarantees are organized as an assurance case in Dregg2/AssuranceCase.lean: each guarantee names an apex theorem and the keystones that discharge it, every name machine-pinned to the Lean kernel's three axioms. The cards below are embedded at build time from the catalog generated out of that file — this page cannot say more than the proofs do.

The full delimitation — the eight-carrier assumption floor, the named Rust components inside the trust base, and the present limitations, stated plainly — is rung 7 of the protocol docs.

Three doors

Use it · build on it · verify it

Use it

A clerk that cannot lie to you

The cipherclerk holds your keys and tokens, attenuates before it delegates, and discloses by projection — what a counterparty's proof does not need, it does not see. Drive it from the CLI, the quickstart, or the apps.

Build on it

Rules the substrate enforces for you

Publish a factory descriptor — a slot layout plus predicate constraints — and every cell minted from it carries its rules in the executor, not in your code. Your app inherits theorems by consuming receipts. Start with the developer guide and the Studio.

Verify it

Check, don't trust

A light client checks one root and learns the whole history (light_client_verifies_whole_history). The semantics, the guarantees, and their axiom hygiene are open Lean you can build yourself: metatheory/ in the repo, explained in the protocol docs.

Understand it

The protocol, in seven rungs

The docs are a ladder, one sentence unfolding outward: the turn, the four substances and eight verbs, guards (one predicate algebra with a coordination dial and a disclosure dial), receipts, the light client, userspace and the polis, and the trust boundary. Each rung is prose, then the grounding theorems, then a live surface to touch: the explorer, the playground, the Studio.

Try it

From clone to a verified turn

1

Build and run a node:

git clone https://github.com/emberian/dregg && cd dregg
cargo build
cargo run -p dregg-node -- run
2

Drive a real app from the CLI — the rejected second claim below is a cell program biting on the commit path, not app bookkeeping:

dregg bounty post   --title "port the gallery app" --reward 500
dregg bounty claim  --bounty <id>
dregg bounty claim  --bounty <id>   # rejected: claimant slot is WriteOnce
dregg bounty submit --bounty <id> --uri ipfs://...
dregg bounty payout --bounty <id>
3

Ask the node which semantics produced what — the producer split is reported live, because the trust boundary is a fact, not a footnote:

curl localhost:8420/api/node/producer | jq

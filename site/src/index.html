dregg · Dragon's Egg · a verified distributed object-capability substrate

Authority you can hand out without having to watch it.

dregg is a shared computer for agreements that must not quietly break. Give an AI agent a budget it physically cannot exceed — the limit is enforced by the substrate, not by the agent's good behavior. Stand up a council whose rules are the cell's own physics, not a moderator's promise. Hand a stranger your entire history and let them verify every step of it without trusting you, your server, or anyone in between. Underneath, each of these is the same small machine: every action needs a checkable warrant, every change leaves a receipt, and the receipts compose into one proof that a light client checks in one step.

Your first five minutes

1

Read the one idea (thirty seconds — it's the next paragraph). Everything in dregg is one sentence: a turn is the exercise of an attenuable, proof-carrying token over owned state, leaving a verifiable receipt. In plain terms: all state lives in cells; nobody acts on a cell without presenting a token that provably licenses exactly that action; and every committed action leaves a receipt anyone can check later. Authority is something you show, never something you claim.

2

Touch the live node — no install, just curl. The public devnet is the same node the repo builds. Ask it what it is, then faucet yourself a cell and read it back:

curl -s https://devnet.dregg.fg-goose.online/status
# → healthy, solo federation, state_producer:"lean", full_turn_proving:true

CELL=$(python3 -c "import secrets;print(secrets.token_hex(32))")
curl -s -X POST https://devnet.dregg.fg-goose.online/api/faucet \
  -H 'content-type: application/json' -d "{\"recipient\":\"$CELL\",\"amount\":1000}"
curl -s https://devnet.dregg.fg-goose.online/api/cell/$CELL

The status line is the honest contract: it names that the state producer is Lean — the proved kernel itself, not a re-implementation.

Then open the explorer on the live federation and scrub any cell's history backward — time travel is just reading receipts, because the receipts are the history.

What dregg is

Cells, turns, receipts

Three objects carry the whole system. Everything else — escrows, councils, name registries, budgets — is built from them, and inherits their guarantees.

01 / Cells

Where state lives, behind its own rules

A cell holds balances, capabilities, and named slots — plus a program: a predicate over its own transitions that the substrate enforces on every change. Nothing is ownerless. A cell changes only under its own program, only by an authorized actor; a turn touching one cell leaves every other cell untouched. "A budget that cannot be exceeded" is just a cell whose program says so.

02 / Turns

Actions that must show their warrant

A turn is an atomic batch of actions. Each action presents a capability — a token naming what it may touch and how, narrowable but never amplifiable — and must discharge every guard in scope. The whole turn commits or none of it does. A forged credential on any leg rejects the entire turn.

03 / Receipts

History you can hand to a stranger

A committed turn leaves Q — a receipt binding the entire post-state, with no field a tamperer can edit. Receipts chain; proofs fold; a light client verifies a whole history by checking one root, re-executing nothing and trusting no executor.

Three doors

Use it · build on it · verify it

Use it

A clerk that cannot lie to you

The cipherclerk holds your keys and tokens, narrows authority before it delegates, and discloses by projection — what a counterparty's proof does not need, it does not see. Start with the quickstart or the CLI.

Build on it

Rules the substrate enforces for you

Publish a factory descriptor — a slot layout plus constraints — and every cell minted from it carries its rules in the executor, not in your code. Your app does no enforcement bookkeeping of its own. The Build page walks the pattern with eight worked examples.

Verify it

Check, don't trust

A light client checks one root and learns the whole history. The semantics are a Lean 4 kernel — the same kernel the node executes — and the guarantee catalog below is generated from it at build time. The Verify page states what is proven and what it rests on.

Why this is not a toy

Real organs, on one path, over a verified executor

The same small machine carries real, composing systems. Three of them — a line of credit, an encrypted group, an asynchronous handoff — are the organs: each is an SDK noun that rides the same .turn() path through the verified executor, with no new executor entry. Each is an executable test you can read.

Trustline

A line of credit, as an attenuated capability

"Issuer A extends holder B a line of N" is granted ⊆ held made numeric. The cell's installed program enforces drawn ≤ ceiling for life — the executor itself refuses an over-line draw. sdk/src/trustline.rs is the story tooth-by-tooth.

Channel

An encrypted group whose membership is physics

A and B form a group whose membership root, key epoch, and epoch key all live on-cell. The keystone: a remove(member) darkens both the ciphertext plane and the capability plane in one atomic turn. sdk/src/channels.rs.

Mailbox

An asynchronous handoff between two nodes

A sends B a sealed turn-intent while B is offline; B's crank drains the hosted inbox over a real HTTP route, the executor-anchored sender gate admits A, and the intent executes as a turn on B's cell — with a checkable custody receipt. An ungranted sender is refused. node/src/mailbox_crank_e2e.rs.

The executor is a memory program

Universal memory: every kernel field and the receipt log project onto one domain-tagged address space, and the executor's post-state equals the fold of the verb's emitted memory trace over the pre-state. So "the receipt binds the whole post-state" is a constructive fact — every field has a universal address; nothing is left off the map. This is the integrity guarantee's load-bearing leg.

Circuits are emitted from Lean

Constraint systems are generated from proved Lean modules as byte-pinned descriptor artifacts; the Rust prover only interprets them. Rust authors zero constraints. One Lean term has two provably-agreeing readings — the executor that runs and the circuit that proves — so circuit and executor cannot disagree about what a turn did.

Polyglot SDKs, one verified core

The Python SDK embeds the real Lean kernel over the FFI — not a port. A TypeScript organ surface gives the same nouns to the browser. A userspace-verify lint tool checks SDK-produced subgraphs against the descriptors, and DreggDL is a checkable deployment spec. Every surface narrows to the same proved executor.

The assurance case

What is proven — and what it rests on

These cards are not marketing copy: they are embedded at build time from a catalog generated out of Dregg2/AssuranceCase.lean, where each guarantee names an apex theorem machine-pinned to the Lean kernel's three axioms. Five guarantees to a light client — Authority, Conservation, Integrity, Freshness, Unfoolability — plus the running entry: Authority, Conservation and Integrity hold over what the node actually runs, because the running system calls the proved function. This page cannot say more than the proofs do — the build fails if it tries.

The guarantees are unconditional in the Lean-kernel sense modulo a small, explicit set of cryptographic assumptions that enter as typed hypotheses, never as axioms — and the present deployment seams (a single-node devnet; per-turn proving staged, not yet auto-attached; the verified producer covering a named subset of effects) are stated, not hidden. The full delimitation is the Verify page, rung 7 of the protocol docs, and the assurance specification.

Run it yourself

From clone to a verified turn

No build needed to look — the curl path above drives the live devnet. To run your own node, drive real apps, and check the proofs, clone and build. The QUICKSTART is the fifteen-minute hands-on tour; scripts/bootstrap.sh sets up the toolchain.

1

Build and run a node (the seeded apps want a freshly-initialized node — the shared devnet's data dir predates genesis seeding):

git clone https://github.com/emberian/dregg && cd dregg
./scripts/bootstrap.sh   # toolchain: rust, lean/elan
cargo build
cargo run -p dregg-node -- run
2

Drive a real app from the CLI — the rejected second claim below is a cell program biting on the commit path, not app bookkeeping:

dregg bounty post   --title "port the gallery app" --reward 500
dregg bounty claim  --bounty <id>
dregg bounty claim  --bounty <id>   # rejected: claimant slot is WriteOnce
dregg bounty submit --bounty <id> --uri ipfs://...
dregg bounty payout --bounty <id>
3

Ask the node which semantics produced what — the producer split is reported live, because the trust boundary is a fact, not a footnote:

curl localhost:8420/api/node/producer | jq

dregg · the deploy gate

Don't detect scams.
Refuse them.

Every contract scanner asks the same impossible question — is this code evil? — then asks you to trust its answer. A deploy gate can do something stronger: refuse a whole class of rug before any gas is spent, and hand the chain a proof of why.

The problem with scanning

"Is this contract malicious" is not a decidable question. Any non-trivial property of a program is undecidable in general — so no scanner can catch every scam and clear every honest contract. Attackers adapt to whatever filter you ship.

And every scanner in the space today is a trusted oracle. It hands down a verdict; you believe it, or you don't. There is nothing to check.

The move: refuse before gas

dregg already deploys its own systems through a checkable spec — a declarative capability layout, in one file. Before a single turn is produced, the whole declared authority structure is checked. A spec that amplifies authority, or creates value from nothing, is refused. Not warned about. Refused.

spec.toml ──▶ lower ──▶ static checks ──┬─▶ REFUSED — non-amplification, node 2/0, cap "mint"
        └─▶ APPLIED — receipt-chained, zero gas spent on a bad spec

What the gate checks today

STATIC

No hidden authority

A granted capability must be an attenuation of one the granter actually holds. An undeclared super-admin, a mint power that exceeds what was handed down — refused at the edge. (This leg is grounded in a line-for-line transcription of seL4's own capability spec.)

STATIC

No value from nothing

Per asset, every move in the deployment sums to exactly zero. A spec that mints supply out of thin air doesn't reach the chain.

STATIC

The whole authority structure, off one file

Not "we scanned it and it looked fine" — the complete capability graph, declared and checked. Dangerous power that is declared becomes plainly visible, and gate policy can refuse it.

STATIC

Structural integrity

No unchecked authorizations, no dangling references, no malformed action. The precise offending site is named — which root, which node, which effect.

the line we will not pretend to cross

Above — decidable from the spec
  • Per-asset conservation
  • Authority non-amplification
  • Attenuation along every grant
  • Structural well-formedness
Below — needs the live executor
  • Did the signer really hold that cap
  • Are the balances actually there
  • Credential and signature validity
  • The whole-state proof

This boundary isn't a footnote we bury. It's a module in the codebase. The static checks are necessary, not sufficient — they say so themselves, and they name exactly what they cannot see. A gate that hides its limits is a gate you can't trust.

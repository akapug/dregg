# Dregg Desktop OS — Exploration Corpus

*The raw per-facet research findings from the `dregg-desktop-os` design workflow (7 parallel explorers). The vision/architecture synthesis lives in [DREGG-DESKTOP-OS.md](../DREGG-DESKTOP-OS.md); this is the supporting prior-art + design detail each facet produced.*

## The REAL seL4 substrate roadmap toward a bootable dregg DESKTOP image (root-task-with-std hosting -> driver PDs -> compositor PD -> bootable dregg.system desktop)

**Prior art**

- **rust-sel4 sel4-root-task-with-std + sel4-musl (~/sel4-sdk/rust-sel4/crates/private/support/sel4-root-task-with-std, crates/experimental/sel4-musl)** — sel4-root-task-with-std is a std-capable root-task runtime (entry.rs/lib.rs/termination.rs) over sel4 + sel4-runtime-common; sel4-musl is a single-file (src/lib.rs) syscall-emulation shim mapping musl/Linux syscalls onto seL4 IPC via sel4-linux-syscall-types. Together they give malloc/pthread/TLS/C++-exceptions on seL4 — exactly the hosted-musl surface the ELF Lean runtime needs.  
  *dregg relevance:* THE immediate step-4 host for the verified executor. WALL.md confirms out/dregg-executor.elf already runs one real turn on aarch64-linux-musl (status:2, nonce 7->8, transfer applied, anti-ghost holds); sel4-musl 'emulates the musl syscall surface', so hosting that exact object set in a root-task-with-std PD is the next concrete build. Both crates are 'experimental' + root-task-only (decision §8.3: root-task simpler/weaker-isolation vs a Microkit-PD musl substrate for the steady-state dregg.system shape).
- **http-server.system multi-PD device template (~/sel4-sdk/rust-sel4/crates/examples/microkit/http-server/)** — The canonical rust-sel4 multi-PD assembly: a server PD + four device-driver PDs (pl031 RTC, sp804 timer, virtio-net, virtio-blk). Each driver PD SOLELY holds its device's virtio_mmio region (phys_addr 0xa003000), DMA paddr (region_paddr setvar), and IRQ; the server shares per-device ring buffers (rx/tx free/used) and reaches each driver over a PP (protected-procedure) channel.  
  *dregg relevance:* The EXACT structural template every dregg driver PD already mirrors (dregg.system's net PD = byte-identical mmio/DMA/IRQ layout) and that the virtio-gpu + virtio-input driver PDs will follow. The shared-ring-buffer + PPC-channel pattern is the compositor<->driver and app<->compositor wire shape; copy it for the display path.
- **virtio-drivers 0.13.0 (the rcore-os crate backing rust-sel4's net/blk drivers)** — A no_std virtio driver crate with a generic Hal + Transport (MmioTransport) abstraction. Confirmed device modules: blk, net, gpu/, input.rs, console, socket, rng, sound, virtio_9p. VirtIOGpu exposes setup_framebuffer/flush/resolution; VirtIOInput exposes pop_pending_event -> InputEvent + ack_interrupt.  
  *dregg relevance:* The display + HID drivers are ALREADY a dependency-version match: rust-sel4's virtio/net + virtio/blk both pin virtio-drivers 0.13.0 with the sel4 hal-impl (crates/drivers/virtio/hal-impl). A virtio-gpu driver PD and a virtio-input driver PD reuse the SAME Hal + MmioTransport the booting net driver uses — no new HAL, just two more device modules. This is weld, not build.
- **QEMU aarch64 virt machine virtio-mmio device set** — qemu-system-aarch64 -machine virt (already the dregg board, virtualization=on cortex-a53) exposes virtio-gpu-device, virtio-keyboard-device, virtio-mouse-device, virtio-tablet-device (plus ramfb as a simpler framebuffer fallback) on the SAME virtio-mmio-bus the net driver binds at slot 31.  
  *dregg relevance:* The compositor-image platform exists with zero new emulation. -device virtio-gpu-device,bus=virtio-mmio-bus.N + -device virtio-keyboard-device,bus=virtio-mmio-bus.M slot into the existing run-assembly QEMU invocation. QEMU opens a real graphical window (drop -nographic), so the bootable dregg desktop is visually demonstrable on the mac/linux host TODAY. ramfb is the de-risking fallback if virtio-gpu 3D negotiation stalls.
- **dregg-firmament router + LocalBacking (sel4/dregg-firmament/src/{router,local}.rs)** — The runnable cap-gradation bridge: one Capability{target,rights} handle, a FirmamentRouter dispatching Local{slot}->seL4 CNode (mint=seL4_CNode_Mint, revoke=synchronous seL4_CNode_Revoke, invoke=seL4_Call) vs Distributed{cell}->a real executor turn, with granted<=held enforced at both backings. 10 tests green incl. the real executor in the loop.  
  *dregg relevance:* The cap surface the compositor grants display/input through. A window handle, a framebuffer-region handle, an input-focus handle become Local seL4 caps the compositor mints to app-PDs and revokes synchronously (n=1: immediate). The desktop's window-management IS object-capability mint/attenuate/revoke — the firmament's headline model extended to display resources, not a new mechanism.
- **banscii artist PD (~/sel4-sdk/rust-sel4/crates/examples/microkit/banscii/pds/artist)** — A 3-PD pipeline (serial-driver -> assistant -> artist) where the artist PD receives bytes over a shared region_in, renders a 'masterpiece' frame into region_out, and signals back over a PP channel — the closest existing rust-sel4 'compute-a-frame-and-hand-it-to-a-driver' shape.  
  *dregg relevance:* The structural seed for the compositor PD's render path: receive app draw-commands/surfaces over shared regions, composite into a framebuffer region, hand the dirty rect to the virtio-gpu driver PD to flush. Mirror its region_in/region_out + PP-channel wiring for the compositor<->app and compositor<->gpu-driver edges.
- **sel4-driver-interfaces (crates/experimental/sel4-driver-interfaces)** — Trait crate defining driver-client interfaces: block.rs, net.rs (GetNetDeviceMeta, used by net-client), rtc.rs, timer.rs, serial/. The contract layer between a driver PD and its clients.  
  *dregg relevance:* Where the missing display + input interfaces belong: add a Display trait (resolution/flush/present) and an Input trait (poll_event/set_focus) mirroring net.rs's GetNetDeviceMeta. The net-client already consumes net.rs via NetClient::new(DRIVER).get_mac_address() over a PP channel; the compositor consumes the new Display/Input traits the same way.

**Design for dregg**

A five-stage substrate ladder, each stage a real QEMU boot extending the prior, converging on a bootable cap-first dregg desktop. The through-line: every desktop resource (a turn, a framebuffer, an input focus, a window) is an seL4 capability in some PD's CNode, and the firmament router is the one surface apps reach them through — so 'desktop' is object-capabilities extended to display/HID, not a new subsystem.

STAGE A — Host the verified executor (close the §6/§7 blocker; the firmament gets its TRUE heart). Steps 1-3 are GREEN (ELF closure, ELF leanrt+lib+kernel, real GMP 6.3.0); out/dregg-executor.elf runs a real turn on aarch64-linux-musl. Step 4: build a sel4-root-task-with-std PD that links the executor object set, wire sel4-musl's syscall handler (route the musl syscalls the runtime issues — brk/mmap/futex/writev/clock for the leanrt GC+malloc+TLS — onto seL4 IPC), boot under the existing Makefile loader path. Deliverable: an executor-PD that decodes a turn_in page, runs dregg_exec_full_forest_auth, writes the receipt to commit_out, prints status:2 over serial — REPLACING dregg-executor-stub-pd in dregg.system. This is the single highest-value seL4 build outstanding and unblocks the firmament's heart row.

STAGE B — Close the net edge to a full turn round-trip. The net driver PD already PROBES a real virtio-net (device_type=Network, NIC up) and the net-client smoltcp PD (DHCP + TCP:5555 + Ed25519 SignedTurn gate) is written. Wire the 2-PD net.system assembly + the driver<->client ring channel so a signed turn arrives over TCP, the edge Ed25519-checks it, and hands it to the Stage-A executor via turn_in. Deliverable: net-PD -> executor-PD -> persist-PD, a turn submitted from the host over TCP committing on-device. This makes the n>1 end of the cap gradation real end-to-end.

STAGE C — Driver PDs for display + HID (the new device seats). Add two driver PDs cloned from the net-driver template, each the SOLE holder of its virtio-mmio region + IRQ + DMA paddr: (1) gpu-driver-pd — VirtIOGpu over virtio-drivers 0.13.0 + the sel4 hal-impl, setup_framebuffer into a shared fb region, flush on a present channel; (2) input-driver-pd — VirtIOInput, pop_pending_event into a shared input ring. Add Display + Input traits to a dregg fork of sel4-driver-interfaces. QEMU gains -device virtio-gpu-device + -device virtio-keyboard-device + -device virtio-tablet-device on the mmio bus; drop -nographic so a real window opens. Deliverable: a draw-PD blits a test pattern through gpu-driver-pd (a dregg banner on-screen) and key/pointer events stream from input-driver-pd to serial. ramfb is the fallback display if virtio-gpu negotiation stalls. De-risk: bring each driver up STANDALONE (its own one-PD .system, like run-net) before assembly.

STAGE D — The compositor PD (the desktop's heart-of-display). A compositor-PD owns the shared framebuffer region and the gpu/input PP channels; it is the SOLE client of the display+input drivers. App-PDs (houyhnhnm cells) get a per-window surface region + an input-focus capability that the compositor MINTS to them and REVOKES synchronously via the firmament's LocalBacking (seL4_CNode_Mint/Revoke) — window lifecycle IS cap mint/attenuate/revoke, n=1 immediate. The compositor routes input only to the focused window's cap-holder and composites surfaces (banscii artist's region_in->composite->region_out->flush pattern, generalized to many surfaces + dirty rects). Deliverable: two app-PDs draw into their own surfaces, the compositor composites both to screen, input routes to the focused one, focus-switch revokes+re-mints the input cap. The 'cap-first windowing/compositing primitives' the vision names.

STAGE E — The bootable dregg.system desktop image. Assemble ONE Microkit image: executor-PD (Stage A) + net-PD/net-client (Stage B) + persist-PD + gpu-driver-PD + input-driver-PD + compositor-PD + app-PD(s). The cap partition is the whole trust boundary: only gpu-driver holds the GPU mmio cap, only input-driver the HID cap, only persist the storage cap, only net the NIC cap, only the executor authority over state. An app-PD's window is a turn through the executor whose receipt the compositor renders — closing the loop from FIRMAMENT §5 (every app action is a cap-mediated verified replayable turn) to pixels. Deliverable: `make run-desktop` boots a windowed dregg desktop in QEMU where a verified turn's result appears on-screen and is driven by real keyboard/mouse. This is the firmament with a screen — the robigalia root-OS interface (starbridge-v2's gpui face) reachable as the eventual seL4-native compositor target.

PLATFORM LADDER: aarch64 QEMU virt is the primary rung (where A-E land; M5 already proves riscv64 boots M0, so the arch-agnostic PDs port by retarget as a follow). Hardware is beyond-a-quarter and gated on a real verified GPU driver (an external adoption, not a dregg build). The semihosted-sel4 / robigalia-emulator (same cap-first patterns native on mac/linux) stays the day-to-day dev fulcrum; the seL4 image is the verified destination the same binaries scale to.

**Key primitives**

- sel4-root-task-with-std PD + sel4-musl syscall shim — the std/musl host that runs the ELF Lean executor (the executor-PD step 4, the immediate build)
- virtio-drivers 0.13.0 Hal + MmioTransport — ONE driver abstraction already backing net/blk; gpu/ and input.rs modules make display+HID drivers a weld not a build
- The driver-PD cap pattern (sole mmio-region + IRQ + DMA-paddr holder) + shared ring buffers + PP metadata channel — the proven net template every new device PD (gpu, input) clones
- FirmamentRouter + LocalBacking (seL4_CNode_Mint/Revoke/Call) — the cap surface through which the compositor grants/attenuates/revokes window-surface, framebuffer, and input-focus handles (windowing = object-capabilities)
- A shared-framebuffer region + dirty-rect flush channel (banscii artist region_in->composite->region_out->present, generalized) — the compositor<->gpu-driver and compositor<->app render wire
- sel4-driver-interfaces Display + Input traits (new, mirroring net.rs GetNetDeviceMeta over PP) — the typed driver-client contract the compositor consumes
- The Microkit .system assembly + cap partition as the trust boundary (dregg.system) — the bootable-image artifact each stage extends; QEMU virt's virtio-mmio bus + (drop -nographic) is the windowed platform

**First step**

Stage A step 4: stand up a sel4-root-task-with-std PD that hosts the already-built out/dregg-executor.elf object set, wiring sel4-musl's syscall handler for the musl syscalls the leanrt runtime issues (brk/mmap/madvise for malloc, futex for the GC/pthread init, writev/write for the receipt, clock_gettime if reached), and boot it under qemu-system-aarch64 via the existing Makefile loader path. Concretely: create sel4/dregg-pd/executor-rt/ as a root-task crate depending on sel4-root-task-with-std + sel4-musl (path-deps into ~/sel4-sdk/rust-sel4), link the executor archives (libdregg_lean_elf.a + the ELF leanrt/lib/kernel/GMP bottom-halves from executor-pd/out/), have its main call lean_initialize_runtime_module() then dregg_exec_full_forest_auth on a fixed wideDemoInput wire baked into the image, and assert status:2/ok:1 prints over serial — the SAME turn WALL.md banked under host-musl, now on bare seL4. Bring it up as a one-PD image first (an executor analogue of run-net/run-stark), THEN swap it for dregg-executor-stub-pd in dregg.system. This converts the firmament's heart from a stub to the real verified executor and is the load-bearing unblock for everything above it; display/input/compositor (Stages C-E) are additive and can proceed in parallel since they don't depend on the Lean runtime.

**Open questions**

- Executor-PD runtime shape (FIRMAMENT §8.3, ember-open): root-task-with-std (simpler, weaker isolation, what sel4-musl supports today) vs investing in a Microkit-PD musl substrate (the steady-state dregg.system shape where the executor is one PD among many). Stage A as a root-task is the fast path; does the steady-state desktop need the executor as a Microkit PD, forcing the harder substrate?
- sel4-musl coverage gap: it is 'experimental' and single-file — does it implement every syscall the leanrt GC + malloc + TLS bootstrap actually issue at runtime (futex semantics for pthread, mmap flags for the GC arena)? A syscall-trace of out/dregg-executor.elf under strace would enumerate the exact set to confirm before the PD link.
- virtio-gpu on seL4 maturity: virtio-drivers' VirtIOGpu does 2D framebuffer (setup_framebuffer/flush) cleanly, but resource negotiation + the sel4 hal-impl have no proven on-device boot in this tree (net/blk do). Is ramfb the right de-risked first display (a plain linear framebuffer, no virtqueue negotiation) with virtio-gpu as the follow?
- Compositor authority model: the compositor-PD becomes a trusted intermediary holding the framebuffer + all input — does that violate least-privilege (it can see every window's pixels and every keystroke)? Or is a trusted compositor acceptable as TCB (like the seL4 root), with per-window surface caps as the isolation boundary between APPS even though the compositor sees all?
- Where the gpu/input device caps sit relative to n>1: net is 'the distributed edge'; display+HID are inherently LOCAL (n=1) devices. Confirm the cap gradation story cleanly says display/input are always-local seats (immediate, no distance parameter) while net is the only seat that slides along n — i.e., the desktop is the n=1 collapse made visible.
- Memory budget: the executor-PD needs a large heap (the Lean runtime + mathlib closure); the compositor needs a multi-MiB framebuffer (1024x768x4 ~= 3MiB). Does the qemu virt -m 2G with the Microkit static memory partition leave room for executor + compositor + drivers + apps simultaneously in Stage E, or does the image need memory-region sizing passes?

## The desktop object model — apps as dregg CELLS, windows/surfaces as cap-confined DISTRIBUTED objects, with the web-forward + distributed angle (a window can render a remote cell's surface; the firmament made interactive/visual).

**Prior art**

- **Fuchsia View System (RFC-0147) + Flatland/Scenic (RFC-0162)** — Separates the WINDOWING layer (one global View Tree owned by Scenic: views, focus, input routing, parent-child structure) from the COMPOSITION layer (Flatland draws pixels INTO views). A View = a region of content owned by one component; a Viewport = a placeholder slot in a parent view that embeds a child's view; a ViewRef = an unforgeable, duplicable kernel handle that stably identifies a view across processes. Parent and child are linked by a token-PAIR handshake (ViewCreationToken / ViewportCreationToken — non-clonable kernel objects): the parent makes a Viewport from its half, the child makes a View from the matching half, and ONLY matching-token holders can establish the embedding. Security invariants: a client cannot read another view's content, cannot manipulate a view it lacks the Viewport/ViewRef for, and can only inject input into its own subtree. Focus is a chain of ViewRefs; a parent controls focus within its subtree.  
  *dregg relevance:* THE direct design template, and it maps onto dregg one-to-one. View := a cell's surface; Viewport := a CapabilityRef a parent app-cell holds reaching a child's surface-cell; ViewRef := a CellId (already an unforgeable, content-addressed, freely-duplicable handle); the token-pair handshake := the firmament's GrantCapability turn (parent grants the child a surface-cap; the executor's granted-subset-of-held gate IS the 'only matching-token holders link' rule). The windowing/composition SPLIT is the key architectural lift to copy: the dregg executor owns the 'view tree' as cell+capability state (verified, distributable), while the gpui/webgpu compositor is a dumb renderer of that state. The negative invariants ('cannot read/manipulate/inject across views') are exactly what dregg's ocap no-ambient-authority already enforces — dregg gets them by construction where Fuchsia enforces them in Scenic.
- **Genode nitpicker / GUI-session stack (Sculpt OS, Feske 'Re-stacking the GUI stack')** — The smallest known capability-confined compositor on a microkernel. nitpicker is a broker between drivers and apps with four typed session interfaces: Gui (apps request a virtual framebuffer + input + named 'views'), Capture (a driver/screencast client pulls composited pixels — the inverse of framebuffer), Event (a driver injects input — the inverse of input). A client's virtual framebuffer size is backed by its OWN session quota (resource accounting is the client's, not ambient). Views and input focus are the two notions nitpicker adds atop the raw framebuffer; nitpicker is 'the only component that must never fail.' Nested compositors are explicit: a window manager is a Gui CLIENT that is itself a Gui SERVER to the apps it frames (recursive stacking). The Capture session is named as the right hook for a future remote-desktop server.  
  *dregg relevance:* The closest existing system to the firmament's seL4 target, and it validates the layering dregg should adopt: a tiny trusted compositor + everything-else-is-a-cap-confined-client. Two transplantable ideas: (1) RECURSIVE compositors — a dregg window-manager cell is a client of the root compositor AND a compositor to the app cells it frames, which falls straight out of cells-holding-caps-to-cells (no special case). (2) Quota-backed surfaces — a surface-cell's backing store is paid from ITS budget (dregg already meters turns in computrons + has issuer-well value), so a malicious app can't exhaust the compositor. The Capture-session-as-remote-desktop-hook is the same seam as dregg's distributed surface (render a remote cell's surface).
- **Croquet / TeaTime (Reed & Smith; replicated deterministic Smalltalk islands)** — Live distributed objects via REPLICATED DETERMINISTIC COMPUTATION: every participant runs a bit-identical 'island' (a virtual computer) in lock-step on a shared clock (a lightweight Reflector orders/timestamps messages); a message to a replicated 'Tea object' is redirected to all replicas; a coordinated distributed TWO-PHASE COMMIT advances computation at all sites for resilience and deterministic results; islands are replicated/recovered via CHECKPOINTS. Determinism is the whole basis: identical program + identical virtual time => identical state everywhere, indefinitely, even for physics.  
  *dregg relevance:* The historical proof that a distributed object DESKTOP can be built on deterministic replicated state — and dregg is a STRICTLY STRONGER version of every TeaTime mechanism. Croquet TRUSTS determinism; dregg PROVES it (the verified execFullForestG executor is the 'island', and identical inputs provably yield identical post-state + receipt + root). Croquet's shared-clock Reflector := dregg's blocklace/consensus ordering. Croquet's checkpoint-replication := the firmament's snapshot (checkpoint+overlay = replay, root-tooth-authenticated). Croquet's two-phase commit := dregg's already-modeled Coord/TwoPhaseCommit + JointTurns. The lesson to carry: a SURFACE is just replicated cell-state; rendering a remote window is subscribing to a remote island's surface-cell and re-deriving its pixels locally — no pixel-streaming needed for deterministic content, only state sync.
- **CapDesk / Polaris / the DarpaBrowser (E language, Combex, POLA)** — A virus-safe capability-secure DESKTOP built in the E ocap language. Apps launch holding only the authority explicitly granted; the POWERBOX is the trusted UI agent that turns a user gesture (an open-file dialog, a drag) into a freshly-minted, narrowly-scoped capability handed to the app — designation-IS-authorization, so least authority is the natural path, not a config burden. The DarpaBrowser (DARPA/David-Wagner-reviewed) confined a plug-replaceable, possibly-malicious HTML renderer so it could harm neither the browser nor the system. Faceted capabilities expose only a sub-interface of an object.  
  *dregg relevance:* The canonical ocap-DESKTOP precedent and the source of the two interaction primitives dregg's desktop needs. (1) The POWERBOX is the trusted-path answer to 'how does a confined app get a NEW capability without ambient authority' — in dregg, the compositor-cell IS the powerbox: a user picks a file/peer in the trusted UI, and the compositor mints a fresh attenuated CapabilityRef into the app-cell's c-list via a real grant turn. (2) FACETS already exist in dregg as CapabilityRef.allowed_effects (an EffectMask that narrows bitwise, never amplifies; the cell crate proves non-vacuity via rejects_effect_mask_widening / allows_effect_mask_narrowing). A 'surface facet' (paint + receive-input, but NOT seal/destroy/set-permissions) is a faceted cap — the primitive is built, it just needs surface-relevant effect bits. The DarpaBrowser confinement story IS the dregg web-forward pitch: confine a webgpu/servo renderer behind a surface-cap so a hostile page harms nothing.
- **Plan 9 rio + draw(3) device + 9P namespace export** — Everything is a file server, INCLUDING the window system: rio is a 9P file server, and a window is a set of files (/dev/draw for graphics ops, plus input files). Because the window system speaks 9P, it is intrinsically distributed/network-transparent: `cpu` exports a terminal's device namespace (mouse/keyboard/display) to a remote machine so a remote program does I/O against the local terminal's surfaces with no special remoting layer — distribution is just namespace mounting.  
  *dregg relevance:* The 'fulcrum for distributed surfaces' the brief names, and a sharpening of dregg's distribution model. Plan 9's lesson: if a surface is addressed UNIFORMLY (a file path / a 9P fid) then remote and local surfaces are the same operation — exactly the firmament's one-(target,rights)-handle thesis applied to windows. dregg improves on the 9P fid (a bearer name, forgeable if leaked, ambient once mounted) with an UNFORGEABLE, ATTENUABLE, REVOCABLE CapabilityRef — Plan 9's transparency WITHOUT its ambient-authority weakness. dregg-directory (named sturdy-refs: human-readable -> handle, with expiry/dispute) is the dregg analog of the 9P namespace for DISCOVERING a remote surface to embed.
- **Arcan SHMIF / A12 network transparency (letoram)** — A modern display server where a client connection is a primary SEGMENT (video+audio+event+metadata) plus negotiable, TYPED SUBSEGMENTS (CURSOR, POPUP, …) that are DEFAULT-DENY — the server (or its Lua policy) accepts/rejects each subsegment request per-client, so the surface set IS a negotiated capability. A12 makes this network-transparent: Arcan sends a DEVICEHINT carrying a connection primitive, trips a 'dead-man's switch' that forces the client's shmif library into recovery, and the client transparently RECONNECTS and rebuilds itself over the network link — used for live network MIGRATION, compartmentalization, and crash recovery. Security stance: no external client may hold a primary ENCODER segment, because 'there is no visual identity for a user to make an informed judgement about what to forward' — output capture is server-side-spawned only, keeping the trust chain intact.  
  *dregg relevance:* The most recent, sharpest validation of two firmament claims. (1) DEVICEHINT-driven transparent reconnection IS the firmament's fluid reach-out for surfaces: a surface that lived on the local box can be relocated across the wire and the app's view follows with no app-visible seam — Arcan proves the migration UX is real and shippable, and dregg's n=1->n>1 collapse is the same axis. (2) Arcan's encoder-trust insight ('no visual identity => unsafe to forward') is the trusted-path PROBLEM, and dregg's verified provenance is a NOVEL answer: a surface-cell carries a receipt chain + state-root, so 'what am I forwarding/embedding' is cryptographically attestable rather than a UI guess. The typed-default-deny-subsegment model maps to faceted surface-caps (a POPUP/CURSOR child surface is an attenuated sub-cap the parent must explicitly grant).
- **Spring (Sun) — distributed object OS with doors + subcontract** — A from-scratch distributed OS where ALL functionality is IDL-typed objects with strongly-typed, inheritable interfaces. The nucleus microkernel provides domains, threads, and DOORS — lightweight capability-like references protected by the kernel, transmissible securely across the network, and distributed-GC'd. SUBCONTRACT is the indirection layer that lets a single object invocation be backed by local, remote, replicated, or cached implementations transparently — the caller doesn't know which.  
  *dregg relevance:* The architectural ancestor of the firmament's 'one handle, two backings, app can't tell which.' Spring's SUBCONTRACT is precisely dregg's FirmamentRouter: the same invocation dispatches to a local (seL4 syscall) or distributed (executor turn) backing by the handle's target alone. Spring's doors (kernel-protected, network-transmissible, GC'd capabilities) := dregg CapabilityRefs + the seL4 CNode caps the firmament unifies. The takeaway for the desktop model: TYPE the surface interface (paint/resize/input/lifecycle) as a first-class object interface, and let subcontract-style routing handle local-vs-remote — which is exactly the firmament bridge already does for generic caps, now specialized to surfaces.

**Design for dregg**

THE DESKTOP IS THE LEDGER. The compositor is a thin renderer of verified cell+capability state; it adds NO new trust. Concretely, four layers, each built by SPECIALIZING primitives that already exist (the WELD method — census says the capability usually already exists, disconnected).

(1) APP = CELL (exists today). Every app is a dregg cell in the embedded World (starbridge-v2/src/world.rs). Its authority is its c-list of CapabilityRefs; its observable state is its cell-forest; its history is its receipt chain. No new abstraction — an app is what a cell already is.

(2) WINDOW = A CHILD CELL whose state holds a SURFACE, addressed by a SURFACE-CAPABILITY. Mirror Fuchsia's View/Viewport/ViewRef onto dregg with ZERO new kernel verbs:
  - View (the surface itself) := a 'surface-cell' — a child cell created by the app via a FactoryDescriptor ('the surface factory'). Its state slots carry the surface descriptor: a content-commitment (BLAKE3 root of the pixel buffer OR of a retained scene-graph / DOM), an intrinsic-size hint, and a damage/epoch counter. NOTE the executor enforces created cells are born ZERO-balance and factory-validated — so a surface-cell is a legitimate, provenance-carrying object, not an ambient handle.
  - ViewRef (the stable identifier) := the surface-cell's CellId — already unforgeable (content-addressed via derive_raw), freely duplicable, cross-process. No new token type.
  - Viewport (the parent's embedding slot) := a CapabilityRef the parent app-cell HOLDS reaching the child surface-cell, attenuated to a SURFACE FACET. The facet is CapabilityRef.allowed_effects (an EffectMask — already in cell/src/capability.rs, already proven to narrow-only). A surface facet permits {paint/SetField-on-surface-slots, receive-input-event, resize-hint} but FORBIDS {seal, destroy, set-permissions, transfer} — so a compositor or parent that holds only a Viewport-cap can render and route input to a child window but cannot tamper with it. (Minor build: assign surface-relevant bits in the EffectMask; the narrowing machinery is done.)
  - The token-PAIR handshake (Fuchsia's unforgeable link) := a GrantCapability turn through the firmament. To embed a child: the child app mints a surface-cap to its surface-cell and GRANTS it (attenuated to the Viewport facet) to the parent; the executor's granted-subset-of-held gate IS Fuchsia's 'only matching-token holders can link.' The embedding edge is a real, receipted, revocable capability — strictly stronger than a kernel eventpair, because it is attenuable AND its provenance is on-chain.

(3) THE VIEW TREE = THE OCAP GRAPH (exists; just project it). Fuchsia's global View Tree is, in dregg, the subgraph of surface-caps among cells — already live in the World's dynamics stream (CapabilityGranted/CapabilityRevoked edges) and renderable from the ledger. Focus is a distinguished path of ViewRefs/CellIds; in dregg, FOCUS IS A CAPABILITY: the compositor-cell holds an 'input-source' cap and grants a short-lived, revocable 'has-focus' surface-cap (the receive-input facet) to exactly one surface-cell at a time — focus change is a revoke+grant turn, so 'only the focused view gets input' is enforced by the ocap graph, not by compositor goodwill, and the focus history is on the receipt chain (auditable: WHY did this window get my keystroke). The compositor itself is a cell that holds caps to the framebuffer/input cells (Genode's nitpicker, but as a cell); a window-manager is a compositor-client-cell that is ALSO a compositor to the apps it frames (Genode's recursive stacking, free from cells-holding-caps-to-cells).

(4) THE DISTRIBUTED + WEB-FORWARD ANGLE = THE FIRMAMENT ROUTER, UNCHANGED. This is the payoff and it requires NOTHING new in the cap layer: a surface-cap is a firmament Capability{target, rights}. If target is a LOCAL surface-cell, the compositor renders it directly (n=1, immediate). If target is a surface-cell on a REMOTE federation, FirmamentRouter resolves it via the executor->net path and the same window renders a remote cell's surface — the fluid reach-out, applied to pixels. Two transport modes for the remote surface content, both grounded in existing dregg mechanism:
  - DETERMINISTIC content (a retained scene-graph / DOM / verified app): ship STATE, not pixels (the Croquet lesson). The window subscribes to the remote surface-cell's state-commitment; the local compositor re-derives the pixels. The state-root tooth (World::state_root) makes the embedded remote surface SELF-ATTESTING — you can prove the window shows the genuine remote cell's surface, which is dregg's NOVEL answer to Arcan's 'no visual identity to safely forward' trusted-path problem.
  - OPAQUE content (a video frame, a webgpu canvas): ship a content-commitment in the surface-cell's state + the bytes over the net-PD shared-ring (Arcan A12 / waypipe-style), with the commitment binding the bytes to the receipt chain. Migration of a surface across the wire is Arcan's DEVICEHINT reconnection expressed as the firmament's n=1->n>1 slide: the surface-cap is unchanged; only its Bounds relax (revocation becomes eventual, the epoch-lift must propagate).
  REVOCATION of a shared/remote surface = the firmament's two backings already unify it: locally seL4_CNode_Revoke (synchronous, the window goes dark instantly); distributedly the group-key epoch lift (the remove(m) that darkens ciphertext AND capabilities in one turn — channels keystone #181). So 'stop sharing my screen' is a revoke turn with honest, n-parametrized bounds.

REACTIVITY (cell state -> surface): the World already EMITS a WorldEvent dynamics stream on every committed turn (BalanceFlowed, FieldSet, CapabilityGranted, …) and the compositor already polls since(cursor) per frame (gpui pull-render). Specialize it: a SetField on a surface-cell's content/damage slot emits a SurfaceDamaged event carrying the new content-commitment + damage rect; the compositor re-renders that surface. State-to-pixels is thus a pure function of cell state, recomputed on the verified dynamics stream — the same reactive spine, now driving pixels. Because every surface mutation is a receipted turn, the desktop is fully REPLAYABLE and TIME-TRAVELABLE (the starbridge-v2 replay/debug panels already do this for the ledger; surfaces come along for free).

WHY THIS IS RIGHT (the don't-launder-vacuity bar): the desktop adds no out-of-band authority. Every window operation is a real, receipted, capability-gated turn through the verified executor; the compositor is a renderer that can be malicious without harming confined apps (CapDesk/DarpaBrowser), because it only holds the surface FACETS it was granted. The negative security invariants Fuchsia enforces in Scenic ('a client cannot read/manipulate/inject across views') dregg gets BY CONSTRUCTION from ocap + no-ambient-authority. n=1 is first-class (the firmament headline) and the SAME surface-cap scales to n>1 with only relaxed bounds.

**Key primitives**

- SURFACE-CELL (View): a child cell birthed via a 'surface' FactoryDescriptor; its state slots hold the surface descriptor (content-commitment = BLAKE3 root of the pixel buffer or retained scene-graph/DOM, intrinsic-size hint, damage/epoch counter). Reuses the verified factory-birth + SetField path; its CellId is the unforgeable ViewRef.
- SURFACE-CAPABILITY / FACET (Viewport): a firmament Capability{target=surface-cell, rights} attenuated via CapabilityRef.allowed_effects (EffectMask) to a surface facet — permits {paint, receive-input, resize-hint}, forbids {seal, destroy, set-permissions, transfer}. The narrowing machinery already exists and is proven non-vacuous (rejects_effect_mask_widening); only the surface-relevant effect-bit assignments are new.
- THE EMBEDDING HANDSHAKE = a GrantCapability turn: parent and child are linked by the child granting an attenuated surface-cap to the parent through the firmament; the executor's granted-subset-of-held gate IS Fuchsia's unforgeable token-pair link — receipted, attenuable, revocable (strictly stronger than a kernel eventpair).
- FOCUS-AS-CAPABILITY: the compositor-cell grants a short-lived, revocable receive-input facet to exactly one surface-cell; focus change = a revoke+grant turn; 'only the focused view gets input' is ocap-enforced and the focus history is on the receipt chain (auditable input routing).
- THE FIRMAMENT ROUTER (existing, sel4/dregg-firmament): resolves a surface-cap to a LOCAL render (n=1, synchronous) or a REMOTE surface-cell over the executor->net path (n>1, bounds relax) — the one mechanism that makes 'a window renders a remote cell's surface' identical to a local window. No change needed; surfaces are just a target type.
- SURFACEDAMAGED DYNAMICS EVENT: a new WorldEvent variant emitted when a surface-cell's content/damage slot is SetField'd, carrying the new content-commitment + damage rect; the compositor's existing since(cursor) per-frame poll re-renders that surface. Reuses the World dynamics spine — state-to-pixels becomes a pure function on the verified event stream.
- STATE-ROOT-ATTESTED REMOTE SURFACE: the surface-cell's content-commitment folded into World::state_root makes an embedded remote surface self-attesting (prove the window shows the genuine remote cell's surface) — dregg's answer to Arcan's 'no visual identity to safely forward' trusted-path problem.
- SURFACE NAMING/DISCOVERY via dregg-directory: human-readable name -> surface sturdy-ref (with expiry/dispute), the dregg analog of a Plan 9 namespace mount, for discovering a remote surface to embed.
- COMPOSITOR-CELL + RECURSIVE WINDOW-MANAGER: the compositor is a cell holding caps to framebuffer/input cells (Genode nitpicker as a cell); a window-manager is a compositor-client-cell that is also a compositor to the apps it frames — recursive stacking falls out of cells-holding-caps-to-cells, no special case.
- REVOCATION-WITH-HONEST-BOUNDS: 'stop sharing this surface' = seL4_CNode_Revoke locally (synchronous, window goes dark instantly) OR the group-key epoch lift distributedly (eventual) — the firmament's two backings already unify surface revocation with n-parametrized Bounds.

**First step**

In starbridge-v2 (which already EMBEDS the verified executor + World + gpui compositor), build the minimal end-to-end surface loop on ONE machine (n=1), entirely from existing primitives: (1) add a 'surface' FactoryDescriptor and birth a SURFACE-CELL whose state slot 0 holds a content-commitment + slot 1 a damage counter; (2) define a tiny surface EffectMask facet (paint + receive-input bits) and have the app GRANT an attenuated surface-cap (the Viewport) to a parent/compositor cell via a real GrantCapability turn — asserting (the anti-ghost tooth) that a NON-attenuating grant (a surface-cap that also permits destroy/set-permissions) is REJECTED by the executor; (3) add a SurfaceDamaged WorldEvent emitted when the surface-cell's content slot is SetField'd, and render it: a gpui panel that, on each frame, polls dynamics.since(cursor), finds the surface-cell via the held Viewport-cap, and paints a rectangle whose color/label is its content-commitment. This proves the spine 'a window is a verified, capability-confined cell surface; painting is a receipted turn; the compositor only renders what its facet grants' — a real frame driven by a real turn, time-travelable via the existing replay panel — WITHOUT touching seL4 or the net path. It is the transfer-triangle-equivalent reference for the desktop model: get this one window right, then amplify to the tree, focus, and the remote (n>1) surface via the firmament router that is already in the tree.

**Open questions**

- GRANULARITY OF SURFACE STATE vs PIXELS: does a surface-cell's state hold a content-COMMITMENT (root) with pixels/scene-graph carried out-of-band over the net-PD ring, or a full retained scene-graph IN cell state? Commitment-only keeps the ledger/proofs cheap and is the obvious default; full-scene-in-state would make a window's appearance fully verifiable + replayable from the ledger alone (strong for the houyhnhnm/no-deception contract) but blows up state size. Likely answer: commitment in state + scene-graph in an addressed overlay (mirror the persist snapshot checkpoint+overlay split).
- WHERE DOES COMPOSITION RUN, AND IS IT VERIFIED? The compositor is explicitly a thin UNTRUSTED renderer (it only holds the facets granted, CapDesk-style) — so the PIXELS it produces are NOT covered by the executor's proof. Is that acceptable (the AUTHORITY is verified, the rendering is not — matching every real ocap desktop), or does the ARGUS 'fidelity all the way down' vision eventually demand a verified compositor / verified graphics driver (the seL4 native-verified-driver aspiration)? Decision needed on where the verified boundary stops for the visual layer.
- INPUT EVENTS AS TURNS — THROUGHPUT: routing every keystroke/mouse-move as a receipted GrantCapability/EmitEvent turn through the executor is clean but a 1000 Hz input stream cannot each be a consensus-finalized turn. Need a tier: focus GRANTS are real turns (rare, security-relevant, auditable), but the per-event input STREAM flows on a faster cap-mediated channel (the receive-input facet authorizes a direct shared-ring, batched/checkpointed into receipts) — confirm this two-tier split and where the proving-modality dial (task #169: prove-on-demand vs checkpoint vs eager) sets the input checkpoint cadence.
- TRUSTED PATH / SECURE ATTENTION: Genode's nitpicker and CapDesk both rely on a user-trusted compositor label (which window am I REALLY typing into; is this overlay genuine). dregg's verified provenance attests a surface-cell's IDENTITY cryptographically — but the human still needs an out-of-band secure-attention cue that the trusted compositor (not a spoofing app) is drawing the labels. What is dregg's secure-attention-key / unspoofable indicator, and does the state-root attestation actually reach the human or just other cells?
- REMOTE SURFACE LATENCY UNDER n>1 BOUNDS: when a window renders a remote surface and the federation is n>1, the firmament says revocation is EVENTUAL and commit is quorum-gated. For a live remote window that means an in-flight frame may render AFTER a revoke (the screen-share-already-stopped-but-one-more-frame-leaked window). Is that acceptable for surfaces, or do remote surface-caps need a stricter freshness caveat (the depth-parameterized non-revocation freshness scale, task #150) so a revoked remote surface cannot paint even one stale frame?
- WEBGPU/SERVO CONFINEMENT CONCRETELY: the web-forward pitch is 'confine a webgpu/servo renderer behind a surface-cap' (the DarpaBrowser story). What is the actual confinement boundary — does servo run as its own cell holding ONLY a surface-cap (so a hostile page can paint but not reach the net/fs/other-surfaces except via granted caps), and does that require servo to be ported to the seL4/firmament substrate, or does the n=1 macOS/linux 'robigalia emulator' confine it with OS-level sandboxing first? This is the gating decision for the 'web-forward DISTRIBUTED desktop' headline.

## cap-first GUI / compositing — dregg's compositing trust model + primitives

**Prior art**

- **Genode Nitpicker (TU-Dresden, Feske & Helmuth 2005; now the Genode 'Gui' session)** — The canonical minimal-TCB secure GUI multiplexer (~1,500 LOC; runs ON seL4 under Genode). It knows only TWO object kinds — a buffer (client-supplied shared-memory pixel region, no color conversion) and a view (a rectangle on screen showing a region of a buffer; arbitrary size/pos/viewport-offset/stacking, with an OPTIONAL client title). Each client gets PRIVATE name spaces of its buffers/views; no client can read or touch another's objects or the global stacking order. Input is routed by the rule 'a client receives an event only if it refers to one of ITS views'; exactly one user-selected focused view gets keystrokes (NO client can set focus); motion goes to the view under the cursor only if it belongs to the focused client (defeats gesture-spying). Trusted path = X-ray mode + FLOATING LABELS: every view is a concatenation of a trusted (policy/kernel-supplied) label + an untrusted (client) label; in X-ray all non-focused views dim, get a bright border, and a max-brightness black-outlined label floats at the topmost-visible spot — a dimmed client is 'doomed to paint gray instead of white' so it CANNOT forge a label or border. A Kill magic-key freezes the layout and lets the user lock out a misbehaving client (DoS emergency brake). DoS is structurally bounded: the client provides its own framebuffer backing store as session quota, so it can't exhaust the server. Client-side window handling (no server-side widgets/decoration policy) is the reason for the tiny TCB — placement/stacking/decoration are declared NOT security mechanisms.  
  *dregg relevance:* This IS the reference architecture for the dregg compositor-PD, and the mapping is almost 1:1 onto primitives dregg already has. Nitpicker's 'buffer' = a client-supplied frame dataspace; dregg's app-PDs already hold only firmament-granted pages, so a surface buffer is just another granted frame. Nitpicker's 'one trusted + one untrusted label per view' maps EXACTLY onto dregg's unforgeable CellId (the trusted part — the cell that OWNS the surface, which the executor knows and the client cannot forge) concatenated with a client-chosen caption (untrusted). 'Only the user sets focus; clients receive events only for their own views' is the input-side analogue of dregg's 'only connectivity begets connectivity' (Spec/Authority.lean) — an event-routing capability a client cannot self-grant. The session-quota DoS defense is the seL4-native form dregg's firmament already uses (apps hold only granted memory). The minimal-TCB / client-side-widgets discipline is precisely why a dregg compositor-PD can be small enough to eventually verify, matching the executor-PD/verifier-PD/persist-PD pattern. Nitpicker's open problem — sharing HARDWARE-accelerated graphics safely between the trusted multiplexer and untrusted clients — is the exact problem the servo/webgpu future target inherits.
- **EROS/Coyotos Trusted Window System (EWS) — Shapiro, Vanderburgh, Northup, Chizmadia 2004** — The first capability-OS window system (~4,500 LOC, ~10x smaller than Trusted X), built on EROS pure-capability invocation. A window/session is obtained by invoking a capability (the constructor/confinement pattern: a freshly-built client is CONFINED — it holds only the capabilities explicitly handed to it, so the window system can reason about what a client can reach). Core thesis: 'the window system is the primary mediator of user input/output AND a commonly-used IPC mechanism, so it is a key enforcement point for security policy.' It provides robust traceability of USER VOLITION (every privileged UI action traces to a real user gesture, not a client request) and is structured so mandatory-access-control / *-property information-flow labels can be layered on with minor extension. It uses an efficient DOUBLE-BUFFERED display to largely eliminate the display-system covert channels (timing/contention leaks) that plague shared single-buffer designs. Clients cannot read the screen or each other's windows; input is delivered only to the focused client.  
  *dregg relevance:* EWS is dregg's DIRECT lineage: dregg is explicitly 'in the vat/E-rights/object-capability tradition,' and EROS capabilities are the same unforgeable/attenuable/delegable references dregg's Cap { target, rights } models. The constructor/confinement pattern = dregg's factory-cell birth path (FactoryDescriptor.allowed_cap_templates, the gen-act 'mint' in Spec/Authority.lean): a surface-owning cell is BORN confined with exactly the surface capability and nothing ambient — the houyhnhnm 'no ambient authority' contract the firmament already enforces. 'Window system mediates user volition' is the deepest steer for dregg: a window action that authorizes a turn (e.g. 'sign this' in the cipherclerk) must be backed by a real user gesture the compositor-PD witnessed and can ATTEST — which is exactly dregg's attestation/receipt machinery applied to input. Double-buffering for covert-channel elimination is a concrete soundness requirement dregg should adopt verbatim for any shared display surface, and it dovetails with dregg's existing info-flow research pillar (#31). EWS being small enough 'to feasibly evaluate for high assurance' is the assurance argument for keeping the dregg compositor-PD tiny and Lean-modellable.
- **Qubes OS GUI virtualization (qubes-gui daemon, dom0)** — A production secure GUI for mutually-distrustful VMs (~2,500 LOC of C in the privileged dom0). Each qube runs its own X server + a gui-agent; apps in a qube speak ONLY a tiny, fixed Qubes-GUI protocol to the dom0 gui-daemon and can NEVER reach dom0's real display. Pixel transport is zero-copy via Xen shared memory: the agent's framebuffer pages are mapped into dom0's Xorg (xengnttab_map_domain_grant_refs) so composition is efficient without copying. The trusted-path mechanism is COLORED WINDOW BORDERS: dom0 (not the VM) draws a per-qube-colored border (green=trusted, red=untrusted, yellow=between) around every window, so a compromised VM cannot spoof which trust domain a window belongs to (the LWN colored-border-spoofing analysis showed why the border MUST be drawn outside the client's reach). A move toward a separate GUI domain (out of dom0) further shrinks the TCB.  
  *dregg relevance:* Qubes proves the model dregg wants at PRODUCTION scale and gives two concretes. (1) Zero-copy shared-memory framebuffer transport is the performance pattern for the firmament's local case: an app-PD's surface frame is mapped into the compositor-PD by an seL4 frame-cap grant (the n=1 collapse of Nitpicker's shared buffer) — no copy, kernel-mediated. (2) The colored border = the VISUAL rendering of dregg's trust gradation: the firmament already labels each app-PD with its granting authority and each surface with its owning CellId, so dregg can color/badge a window by the cell's lineage (sovereign vs hosted, which factory minted it, how attenuated the surface cap is) — a far RICHER trusted indicator than a 3-color border because dregg's authority is a structured lattice (AuthRequired) with provenance, not a flat label. The 'border drawn outside client reach' rule = the compositor-PD holds the sole framebuffer/GPU device cap (exactly as persist-PD holds the sole disk cap, net-PD the sole NIC cap) and reserves trusted screen real-estate the client cannot paint.
- **Fuchsia Scenic / Flatland + the View System (RFC-0147, RFC-0092)** — The modern capability-object compositor. Scenic owns the global scene graph / View Tree and does all rendering + hit-testing; clients connect via a 'session' (one FIDL channel = one UI client) and contribute content by creating a View. A View is the unit of graphics+interaction; a Viewport is a placeholder by which a parent embeds a child View. Parent and child link ONLY by exchanging matching NON-CLONABLE kernel-object tokens (parent: ViewHolderToken, child: ViewCreationToken) obtained out-of-band — 'by mutual agreement,' so neither side can attach to the other unilaterally. A ViewRef is an UNFORGEABLE, duplicable-but-not-clonable kernel handle that is the stable cross-component IDENTITY of a View; holding a ViewRef is what lets a service talk ABOUT a view (focus, hit-testing, accessibility annotation). The View Tree need NOT match the component tree. Focus is a chain: a parent controls focus within its subtree, the platform controls top-level focus per product policy; keyboard events go to the focused view, pointer events by hit-test. Image buffers must be allocated through Scenic's Allocator. Parent clipping can restrict where descendants render or receive input.  
  *dregg relevance:* Scenic is the closest existing match to 'a window = a dregg cell's surface capability,' and it validates dregg's core move. ViewRef (unforgeable kernel handle = a view's capability identity) is LITERALLY dregg's CellId used as a surface handle: dregg already has unforgeable cell identity (value-hash, never Nat) and a generative discipline for who may reference whom. The non-clonable token handshake for parent/child embedding = dregg's introduce/Granovetter rule (Spec/Authority.lean apply_introduce's connectivity-premise + consent): an app can embed another app's surface ONLY if both consent and a capability is introduced — no ambient embedding. 'View tree decoupled from component tree' tells dregg NOT to force the surface hierarchy to equal the cell-ownership hierarchy: a composite window (cipherclerk embedding a sub-app) is a surface-tree edge authorized by an introduced surface-cap, independent of cell lineage. Hierarchical focus authority (parent controls subtree focus) is the design for nested trusted-path: the compositor-PD is the root focus authority (the 'platform'), and a trusted shell cell can be delegated focus-routing within its subtree. The Allocator-mediated image buffers = the compositor-PD as sole minter of surface frame-caps.
- **Wayland security model (and Arcan's segment-archetype answer)** — Wayland fixed X11's worst sin (any client could keylog/screenshot everyone) by giving each client by-default access only to its OWN surface and its own input — input isolation + screen-capture mediation are now the baseline. BUT it punted the hard part: privileged operations (global screenshot, virtual keyboard, screen-share, input injection) are done via protocol EXTENSIONS 'intended only for privileged clients,' and Wayland defines NO mechanism for what makes a client privileged or how that privilege is enforced — each compositor invents its own (GNOME bakes features into the compositor; wlroots exposes extensions; nothing is capability-scoped). Arcan answers with SEGMENT ARCHETYPES: every client segment ('window') carries a type (game/media/tui/...) that BINS it into a distinct privilege model for fine-grained sandboxing, and a non-authoritative/semi-trusted connection gets a deliberately restricted privilege set established at authenticated connect time (SHMIF).  
  *dregg relevance:* This is the NEGATIVE lesson that justifies dregg's whole approach: Wayland's 'privileged client' hole is precisely the ambient-authority gap dregg structurally forbids. In dregg there is no out-of-band 'privileged' bit — the authority to screenshot a region, inject input, or share a screen IS a capability (a specific (target, rights) over a surface or input cell), granted by introduce, attenuable, and revocable, with every grant tracing to an authorized generative act. So dregg gets, for free and uniformly, what Wayland bolts on per-compositor: 'screen-record THIS app' is a delegated, attenuated surface-read cap with provenance; a global screenshot capability simply does not exist unless explicitly minted. Arcan's segment-archetypes (privilege model bound to surface TYPE at authenticated connect) map onto dregg's factory contracts: a surface cell minted by a given factory carries a contract (FactoryDescriptor.allowed_cap_templates) that bounds what surface/input rights it may ever hold — a typed, verified version of Arcan's archetype binning. dregg should treat EVERY privileged compositor operation as a first-class attenuable capability, never a client class.

**Design for dregg**

THE COMPOSITOR-PD AS A THIRD DEVICE-HOLDING ORGAN. The dregg compositor is a new protection domain in the firmament topology (docs/FIRMAMENT.md §2), structurally identical to the net-PD and persist-PD: it is the SOLE holder of the framebuffer/GPU device cap, exactly as persist-PD solely holds the disk cap and net-PD solely holds the NIC cap. App-PDs reach it only through firmament-granted capabilities. Its TCB job is the minimal-complexity multiplexer of Nitpicker/EWS: own the screen and the input devices, route pixels and events by capability, render the trusted path. It holds NO app logic, NO widget toolkit, NO placement policy (client-side window handling, per Nitpicker/EWS) — those live in untrusted app-PDs and a shell cell.\n\nA WINDOW IS A SURFACE CELL — A THIRD TARGET VARIANT. The load-bearing move: extend the firmament handle Target (sel4/dregg-firmament/src/lib.rs) with a third backing, `Surface { cell: CellId }`, alongside `Local { slot }` and `Distributed { cell }`. A window is then literally a (target=Surface(cell), rights) handle — the SAME Capability struct, attenuated by the SAME is_attenuation (granted ⊆ held) gate, delegated by the SAME introduce discipline, revoked by the SAME mechanisms. A surface cell is a real dregg cell (an object with verified state and capabilities); its state holds the surface's geometry, the frame-cap reference, the owning lineage, and the client caption. This means EVERY compositing operation is already covered by dregg's existing capability metatheory — no parallel model. The Surface backing's bounds collapse at n=1 (compositor on the same box) to the strong local properties: immediate surface-revoke (kill key = seL4_CNode_Revoke of the frame-cap, synchronous), synchronous present (one frame map), consistent capture.\n\nFIVE CAP-CONFINED SURFACE OPERATIONS, ALL GENERATIVE-DISCIPLINED. The compositor-PD exposes a tiny Nitpicker/Scenic-shaped session interface, but every verb is a capability invocation, not an ambient call: (1) CREATE-SURFACE — mints a surface cell + its frame-cap; only authorized by a surface-factory cap the app holds (EWS constructor/confinement; dregg factory-birth). (2) PRESENT(surface, dirty-rect) — the client's frame dataspace is mapped into the compositor by an seL4 frame-cap grant (Qubes zero-copy / Nitpicker shared buffer, n=1 collapse); requires a write-cap on the surface. (3) EMBED(parent-surface, child-surface) — a surface-tree edge, authorized ONLY by the introduce/Granovetter rule (Spec/Authority.lean apply_introduce: connectivity premise + both-consent + non-amplification), i.e. Fuchsia's non-clonable-token handshake realized as dregg's introduce. The surface tree is decoupled from the cell-ownership tree (RFC-0147). (4) GRANT-INPUT(surface) / GRANT-CAPTURE(surface, region) — delegating an attenuated read/capture or input-routing cap; this is the capability that Wayland leaves as an ungoverned 'privileged client' bit (the negative lesson). (5) REVOKE — drops the surface cap (immediate at n=1).\n\nTRUSTED PATH = UNFORGEABLE LINEAGE LABEL + RESERVED CHROME. dregg's trusted-path is Nitpicker's floating-label split, but the trusted half is RICHER than anyone else's because dregg authority is structured with provenance. Each window's label is trusted-part ∥ untrusted-part: the trusted part is derived by the compositor-PD from the surface cell's CellId and its AUTHORITY LINEAGE (which factory minted it, sovereign-vs-hosted, how attenuated its caps are — all known to the executor, unforgeable by the client); the untrusted part is the client caption. The compositor-PD renders, in screen chrome the client provably cannot paint (it holds no cap to that region — Qubes 'border outside client reach'), an X-ray/dim mode (Nitpicker) where non-focused windows dim so a dark client is 'doomed to paint gray not white' and cannot forge the label or the lineage badge. Color/badge encodes the cell's trust grade (Qubes border, enriched). A Kill magic-key (held by the compositor-PD as the input root-authority) freezes layout and revokes a misbehaving surface's frame-cap (Nitpicker DoS brake).\n\nINPUT = CAPABILITY-ROUTED + ATTESTED USER VOLITION. Events route by the Nitpicker rule, recast as capability: a client receives an event only via an input-routing cap it holds for the focused surface; the compositor-PD is the ROOT focus authority and only the USER selects focus (no client self-grants focus — the input analogue of 'only connectivity begets connectivity'). Hierarchical focus delegation (Scenic): the compositor delegates focus-routing within a subtree to a trusted shell cell. Crucially (EWS user-volition): when a user gesture in a trusted surface authorizes a TURN (e.g. cipherclerk 'sign this'), the compositor-PD ATTESTS the gesture — it emits a signed input-receipt the executor-PD can require as a premise of the turn, using dregg's existing attestation/receipt machinery. This closes the loop the firmament already builds for state onto input: a value-moving turn can require proof a real user clicked, not a client request.\n\nDoS + COVERT CHANNELS, STRUCTURALLY. Backing store is client-supplied as the granted frame-cap (Nitpicker session-quota), so a client cannot exhaust the compositor. Per EWS, shared display surfaces are double-buffered to eliminate display-system covert channels (feeds dregg info-flow pillar #31). No client reads the global stacking order or another surface's pixels/events (Nitpicker private name spaces; Wayland baseline) — enforced because each holds only its own surface caps.\n\nWHY THIS IS DREGG-NATIVE NOT A PORT. The compositor reuses, verbatim: the firmament PD topology + sole-device-cap pattern; the Capability { target, rights } handle (+1 Target variant); is_attenuation / Spec.Authority introduce/mint/attenuate/revoke; CellId as unforgeable surface identity; factory contracts as Arcan-style typed archetypes; the n=1 bounds collapse; and the attestation/receipt machinery for user volition. The only NEW verified surface is the compositor-PD's tiny multiplexer/router and the Surface backing — which is the deliberately-small, Lean-modellable TCB the whole lineage (EWS, Nitpicker) argues for.

**Key primitives**

- SURFACE CELL = the window: a real dregg cell whose verified state holds {geometry (size/pos/viewport-offset), frame-cap reference, owning lineage, client caption, stacking neighbor}. A window is nothing but a cell with a surface capability — reuses all of dregg's cell/capability metatheory.
- Target::Surface { cell } = the THIRD backing on the firmament handle (sel4/dregg-firmament/src/lib.rs), beside Local{slot} and Distributed{cell}. A window handle is a Capability { target: Surface(cell), rights } — attenuated/delegated/revoked by the SAME verbs and the SAME is_attenuation gate as every other dregg cap.
- FRAME-CAP = a client-supplied pixel dataspace shared into the compositor-PD by an seL4 frame-cap grant (Nitpicker buffer / Qubes zero-copy, n=1 collapse). PRESENT(surface, dirty-rect) updates it; the compositor never copies, only maps and composites.
- SURFACE-FACTORY CAP = the constructor/confinement birth gate (EWS constructor; dregg FactoryDescriptor.allowed_cap_templates). CREATE-SURFACE is authorized only by this cap, and the factory CONTRACT bounds which surface/input rights the new surface cell may ever hold (Arcan-style typed archetype, verified).
- EMBED via INTRODUCE = a surface-tree edge authorized by Spec/Authority.lean apply_introduce (connectivity premise + both-party consent + non-amplification) — dregg's realization of Fuchsia's non-clonable parent/child token handshake. Surface tree is decoupled from cell-ownership tree.
- INPUT-ROUTING CAP + GRANT-INPUT/GRANT-CAPTURE = the explicit, attenuable capability for receiving events on a surface or capturing a region. This is what Wayland leaves as an ungoverned 'privileged client' bit; in dregg every such privilege is a first-class delegated cap with provenance. A global screenshot cap simply does not exist unless minted.
- TRUSTED LABEL = (trusted lineage-derived part ∥ untrusted client caption), Nitpicker's floating-label split. The trusted part is derived by the compositor-PD from the surface cell's CellId + authority lineage (factory, sovereign/hosted, attenuation depth) — unforgeable by the client, richer than Qubes' flat color border because dregg authority is a structured provenance lattice.
- RESERVED TRUSTED CHROME + X-RAY/DIM MODE = screen real-estate (labels, borders, lineage badges, focus indicator) the compositor-PD paints and the client provably cannot (holds no cap to that region — Qubes 'outside client reach'); X-ray dims non-focused windows so a dark client is 'doomed to paint gray not white' (Nitpicker) and cannot forge the label/badge.
- FOCUS ROOT-AUTHORITY = only the user selects focus (no client self-grants — the input analogue of 'only connectivity begets connectivity'); events route only to the focused surface's input-cap holder (Nitpicker rule). The compositor-PD is root; focus-routing within a subtree is delegable to a trusted shell cell (Scenic hierarchical focus).
- ATTESTED USER-VOLITION RECEIPT = when a gesture in a trusted surface authorizes a turn, the compositor-PD emits a signed input-receipt the executor-PD can require as a turn premise (EWS user volition, via dregg's existing attestation/receipt machinery). A value-moving turn can require proof a real user clicked.
- KILL MAGIC-KEY = a compositor-PD-reserved key that freezes layout and revokes a misbehaving surface's frame-cap (immediate at n=1 via seL4_CNode_Revoke) — Nitpicker's DoS emergency brake. Surface backing-store is client-supplied (session quota), so DoS is structurally bounded.
- DOUBLE-BUFFERED SHARED SURFACES = covert-channel elimination on any shared display path (EWS), feeding dregg's info-flow research pillar (#31).

**First step**

Add the `Target::Surface { cell: CellId }` backing to the firmament handle and prove it composes with the existing capability discipline — BEFORE any pixels. Concretely, in sel4/dregg-firmament/src/lib.rs: (1) add the `Surface { cell }` variant to `Target` (+ `Capability::surface(cell, rights)` and `Target::is_surface()`), (2) add a `SurfaceBacking` module mirroring DistributedBacking but whose 'invoke' is a surface op (create/present/embed/grant-input/revoke) routed through a real dregg turn — so a surface cap attenuates/delegates/revokes through the SAME `is_attenuation` (granted ⊆ held) gate and the SAME executor as every other cap, and (3) extend the existing handle_tests with `surface_attenuate_is_backing_agnostic` and `surface_n_equals_one_collapse` proving a surface handle narrows by the real lattice and that PRESENT/REVOKE collapse to immediate/synchronous at n=1 (Bounds::LOCAL). This is the smallest change that makes 'a window = a dregg cell's surface capability' REAL and load-bearing in code, reusing the firmament's proven bridge, with zero new trust surface and zero device drivers — it validates the whole design's central claim (a surface is just another point on the distance parameter) the same way the executor-state bridge and the channels weld validated theirs. Only after this green do you build the compositor-PD multiplexer + frame-cap mapping + trusted-chrome renderer on top.

**Open questions**

- GPU vs framebuffer TCB: Nitpicker's open problem — safely sharing HARDWARE-accelerated graphics between a trusted multiplexer and untrusted clients — is unsolved here too. A software-composited framebuffer keeps the compositor-PD tiny and verifiable (the EWS/Nitpicker path); webgpu/servo compositing (the stated future target) drags a huge GPU driver into or beside the TCB. Decision: does the compositor-PD do CPU composition of client frame-caps (small TCB, the first target), with GPU acceleration deferred to a separate untrusted render-PD whose output is itself a frame-cap the compositor composites? That preserves the trust model but bounds performance.
- Attested user-volition wiring: making a turn REQUIRE a compositor-signed input-receipt is powerful (EWS volition) but needs a concrete contract — what does the receipt commit to (surface CellId + gesture + a nonce + which turn-field it authorizes), and how does the executor-PD verify it without the compositor becoming a second trusted authority over state? Likely the compositor holds an attestation key and the executor checks a signature, mirroring the net-PD's Ed25519 pre-check — but the binding between 'this click' and 'this turn' must be pinned to avoid replay/confusion.
- Trusted-chrome real-estate vs client surfaces: Nitpicker's orphaned-area problem (view layouts leaving unlabeled regions) and the fullscreen-window spoof both need a concrete policy — does dregg reserve a permanent top bar (DOpE/Qubes style) for the focused-cell lineage, use floating labels, or blank orphaned areas? At n=1 with a known set of app-PDs this is simpler than the general case, but the rule must be pinned so a malicious client cannot mimic a full trusted desktop.
- Distance-gradation of surfaces: the design lets Surface(cell) collapse at n=1, but what is the n>1 meaning of a surface — a remote app's window composited locally over the net-PD (a la Nitpicker-over-network / Arcan networking)? The firmament thesis says it should slide along n, but the bounds for a REMOTE surface (present latency, revoke propagation, input-attestation across machines) need the same honest n-parametrized table the §3 cap-gradation has.
- Lean model scope: which compositor properties get a Lean model and Rust differential (the dregg method)? Strong candidates: 'a client receives an event only for a surface it holds an input-cap on' (the Nitpicker isolation invariant), 'EMBED edges trace to an introduce' (lifting Spec/Authority's only-connectivity-begets-connectivity to surfaces), and 'the trusted label is a function of lineage the client cannot influence' (an unforgeability lemma). Deciding the modelled invariants up front keeps the compositor-PD on the same verified-by-construction footing as the executor.

## GRAPHICS + WEB stack — the dregg cap-first compositor: gpui (windowing/compositor) + servo (embeddable web surfaces) + wgpu (the portable GPU abstraction), mac/linux-native NOW → seL4-native-verified-graphics LATER via a virtio-gpu driver PD.

**Prior art**

- **gpui (Zed fork @ fca2ccd4, ALREADY in starbridge-v2/Cargo.lock)** — Zed's GPU-accelerated immediate-mode UI framework. The pinned fork is ALREADY MODULARIZED into per-platform backend crates: gpui_macos (Metal-direct, the active native backend), gpui_linux (calloop/swash/pathfinder-geometry path), gpui_windows, and crucially gpui_wgpu (a cross-platform wgpu-29 renderer) + gpui_web (gpui_wgpu→WebGPU/WebGL2 over wasm-bindgen/web-sys). gpui_platform is the backend selector aggregating all four. It is a retained scene + element tree with px()/Bounds/WindowOptions/open_window — i.e. it IS a 2D compositor of styled quads+glyphs+images already.  
  *dregg relevance:* THIS IS THE COMPOSITOR. starbridge-v2 already embeds the verified executor and renders the cockpit through gpui (one window via cx.open_window). The headline finding: the fork ALREADY HAS the wgpu + web backends in the lockfile — so 'gpui as the cap-first compositor, native now, web/portable via wgpu later' is not a port, it is a backend selection + an upstream we ride. gpui is the dregg desktop's window-and-surface layer; servo and webgpu surfaces composite INTO it as cells.
- **servo / libservo (crates.io 0.1.0-rc2; servo-webgpu, servo-wgpu-interop-adapter, servoshell)** — Embeddable, memory-safe web engine. Modern embedding = handle-based WebView API (servo/servo #35119/#35662; handle lifetime controls webview lifetime; per-WebView HiDPI/zoom #36419) + a RenderingContext/OffscreenRenderingContext (#35465): servo renders a WebView into a surfman-backed OFFSCREEN surface, and the embedder BLITS that into a subsection of its own parent context. webrender is servo's GPU compositor — but it renders via OpenGL/GLES internally through the gleam bindings (NOT wgpu; servo #37149 tracks renderer-trait alternatives toward wgpu but it is not landed). Has servo-webgpu + servo-wgpu-interop-adapter for the page-side WebGPU + wgpu surface interop.  
  *dregg relevance:* servo is the WEB-FORWARD engine for WEB SURFACES — one WebView per dregg web-surface cell. The clean integration seam = OffscreenRenderingContext: servo paints a WebView offscreen, the dregg compositor composites that texture as a cell, no servo-owns-the-window coupling. The friction to name honestly: webrender=GL, gpui-wgpu=Vulkan/Metal/WebGPU — bridging them is a texture-share/blit (or a GL interop context), NOT one unified device today. That blit boundary is exactly where a per-surface CAPABILITY check lives.
- **wgpu 29 (zed-industries/wgpu fork, in lockfile) + wgpu-hal** — Cross-platform, safe, pure-Rust WebGPU implementation. Native backends: Vulkan, Metal, D3D12, GL/GLES. Web: WebGPU + WebGL2 over wasm. wgpu-hal is the thin hardware-abstraction layer beneath; the Vulkan backend works on Linux/Android/Windows and macOS via Vulkan-Portability/MoltenVK. Used by Bevy, Burn (GPU compute on embedded), Firefox's WebGPU.  
  *dregg relevance:* wgpu is THE PORTABILITY FULCRUM for the GPU. It is the single API that spans (a) native mac=Metal, (b) native linux=Vulkan/GL, (c) web=WebGPU [the gpui_web path already in the lock], and (d) the seL4-later target = Vulkan over a virtio-gpu/Venus stack. Picking wgpu as the dregg compositor's GPU substrate means the SAME renderer code reaches all four — mirroring the firmament's one-model-many-distances thesis at the GPU layer. NOT yet no_std/seL4-ready (it assumes a windowing/loader substrate), which is the seL4-later cost, not a now-cost.
- **virtio-drivers crate 0.13.0 (VirtIOGpu) — ALREADY a dep of the existing seL4 net-PD** — no_std VirtIO guest drivers in Rust. Includes device::gpu::VirtIOGpu supporting 2D mode (resolution()/setup_framebuffer()/flush(): create a host 2D resource, attach guest-RAM framebuffer as backing, CmdSetScanout to a display head; ARGB hardware cursor + multiple scanouts) and 3D virgl mode. The exact crate (and version 0.13.0) the firmament's net-PD already pulls for virtio-net.  
  *dregg relevance:* THE seL4 GPU DRIVER-PD PATH, and it is the SAME PATTERN as the net-PD that already boots (FIRMAMENT §6: driver-PD holds the device cap + virtio-mmio region + DMA paddr, sole toucher of the hardware). A virtio-gpu driver PD = swap VirtIONet→VirtIOGpu in the proven net-PD scaffold. 2D framebuffer mode is the minimal seL4-native graphics boot (a PD that owns the GPU cap and flushes a framebuffer scanout); 3D/virgl + Venus(Vulkan≤1.3) is the path to running wgpu's Vulkan backend in-guest later.
- **rust-vmm vhost-device-gpu + virglrenderer (Venus / VirGL / gfxstream)** — Host-side virtio-gpu device implementation in Rust (rust-vmm, virtio 1.2). virglrenderer on the host translates guest 3D commands: VirGL (OpenGL≤4.6), Venus (Vulkan≤1.3, runs ISOLATED on the host so a GPU crash doesn't take the VMM down), Zink (GL→Vulkan). Venus/VirGL≥4.4 need VMM blob-resource support (/dev/udmabuf).  
  *dregg relevance:* The HOST half of the host-GPU→seL4-virtio-gpu→native-verified-driver ladder. For the 'robigalia emulator / semihosted-seL4' fulcrum, this is how a seL4 guest gets real 3D from the mac/linux host GPU: the guest virtio-gpu PD speaks to a host vhost-device-gpu/Venus, so wgpu-Vulkan-in-guest is backed by the real host GPU. The Venus-isolated-process model is itself a cap-isolation story that rhymes with dregg's PD-per-driver.

**Design for dregg**

THE DREGG COMPOSITOR = a CAP-FIRST surface compositor where every window/surface/WebView is a CELL and every window handle is a firmament (target,rights) capability. Three layers, one model:\n\n(1) gpui = the compositor + windowing shell. It already draws the dregg desktop (starbridge-v2 proves it: embedded verified executor + gpui window). Generalize the single cockpit window into a multi-surface compositor: a SurfaceCell is a dregg cell whose state is {bounds, z, backing-texture-handle, owning-PD}, and a WindowHandle is a Capability{target=SurfaceCell, rights} in the EXISTING firmament gradation (sel4/dregg-firmament). The compositor mediates ALL screen access by capability: an app cannot draw to a pixel it lacks a surface-cap for (the §5 no-ambient-authority contract, extended from compute to PIXELS). Input events route the same way — a focus/pointer cap, not ambient input.\n\n(2) servo = web surfaces. One WebView per web-surface cell, rendered via OffscreenRenderingContext into a surfman surface; the compositor composites that texture as the cell's backing. The WebView's lifetime = the surface-cell's lifetime = the capability's lifetime (handle-based API makes this exact). A page's network/storage/clipboard reach is brokered through dregg caps, not ambient — servo's embedder hooks become firmament cap-checks. This is the WEB-FORWARD half: the dregg desktop renders real web content as first-class, capability-confined surfaces.\n\n(3) wgpu = the GPU substrate that makes (1)+(2) portable across the native-now/seL4-later split. Adopt gpui's wgpu backend (gpui_wgpu, already in the lock) as the canonical compositor renderer so ONE renderer spans Metal(mac)/Vulkan(linux)/WebGPU(web)/Vulkan-on-virtio-gpu(seL4). The servo↔wgpu seam is a texture blit (webrender renders GL-offscreen; the compositor samples it) until servo #37149's wgpu renderer lands, then it collapses to a shared device.\n\nNATIVE-NOW vs seL4-LATER split:\n- NOW (mac/linux/web, the portability fulcrum): gpui draws everything (Metal-direct on mac today, or the wgpu backend for the portable/web/linux path); servo WebViews composite as offscreen-surface cells; the host GPU is used directly; NO seL4. This is best-on-mac/linux immediately and runs the SAME cap-first surface model that seL4 will later enforce in hardware.\n- LATER (seL4-native verified graphics): a virtio-gpu DRIVER-PD (clone the booting net-PD, swap virtio-drivers VirtIONet→VirtIOGpu) becomes the SOLE holder of the GPU device cap + virtio-mmio region + DMA paddr. v0 = 2D framebuffer scanout (a PD flushes a framebuffer = the firmament's first pixel, mirroring 'the firmament reached the wire'). Then 3D: wgpu-Vulkan in a compositor-PD over virtio-gpu/Venus (host-backed via vhost-device-gpu for the semihosted-seL4 emulator; real hardware on bare seL4). The eventual 'native verified graphics drivers' = a verified virtio-gpu/DRM-KMS driver-PD, the GPU analogue of the verified-FS line — adopted/verified, the apex of the ladder.\n\nThe cap-first compositor IS the firmament thesis at the display layer: a surface-cap resolves locally (n=1: the compositor draws it immediately) or distributedly (n>1: a remote WebView / screen-share / another machine's surface composited over the net-PD) through ONE handle, the bounds simply relaxing along n. That makes 'web-forward desktop OS' and 'distributed ocap OS' the same statement about pixels.

**Key primitives**

- SurfaceCell — a dregg cell whose state is {bounds, z-order, backing-texture handle, owning-PD/app, damage rect}; the unit the compositor composites. Makes a window/surface a first-class verified object (reuses reflect::Inspectable in starbridge-v2).
- WindowHandle = Capability{target: SurfaceCell, rights} — a window/surface handle IS a firmament cap in the EXISTING sel4/dregg-firmament gradation (Local seL4-frame backing vs Distributed remote-surface backing); attenuate=sub-region/read-only surface, delegate=hand a child a sub-surface, revoke=darken the surface. Pixels become ocap.
- CompositorRouter — the gpui-side analogue of FirmamentRouter: owns the scene graph, resolves each SurfaceCell's backing (gpui-native quad/text, a servo offscreen texture, or a raw wgpu surface), and gates every draw + input-event delivery on a held surface/focus cap. No ambient screen or input.
- ServoWebSurface — a servo WebView bound 1:1 to a SurfaceCell via OffscreenRenderingContext (surfman offscreen → compositor texture); WebView lifetime = cell lifetime = cap lifetime. Servo's embedder callbacks (navigation, network, clipboard, storage) are routed to firmament cap-checks, not ambient.
- wgpu Device/Queue as the canonical compositor backend (via gpui_wgpu) — the ONE renderer abstraction spanning Metal(mac) / Vulkan(linux) / WebGPU(web) / Vulkan-on-virtio-gpu(seL4). The texture-handle in a SurfaceCell is a wgpu texture (or a blitted-in GL/servo texture) so the same compositor code is backend-agnostic.
- virtio-gpu driver-PD — the seL4-later sole-GPU-cap holder, built by cloning the proven virtio-net PD and swapping virtio-drivers::VirtIONet→VirtIOGpu; 2D framebuffer scanout first (minimal pixel), 3D/virgl/Venus for wgpu-Vulkan-in-guest later. The GPU analogue of net-PD's NIC cap.
- The servo↔wgpu texture-share seam — the explicit, named integration boundary (webrender=GL offscreen, compositor=wgpu): a blit/import today (and the natural place the per-surface capability is enforced), collapsing to a shared device if/when servo's wgpu renderer (servo #37149) lands.

**First step**

Generalize starbridge-v2's single gpui window into a 2-surface CAP-FIRST compositor demo, entirely native-now, no seL4, no servo yet: introduce SurfaceCell + a WindowHandle = Capability{target: SurfaceCell, rights} reusing the REAL dregg cap types from sel4/dregg-firmament (so it rides granted⊆held, not a toy), and render TWO surfaces in the existing gpui window where (a) the compositor refuses to composite/route input to a surface whose owner lacks the surface-cap — the no-amplification guarantee firing at the PIXEL layer, exactly the teaching-moment pattern the cockpit's '⚠ over-grant' already uses for transfers — and (b) attenuating a surface-cap to read-only/sub-region visibly narrows what that surface may draw. This lands one screenful that proves the thesis 'a window is a capability' on the hardware you have today (mac Metal path), reuses the embedded verified executor + firmament router already in-tree, and creates the SurfaceCell seam that servo WebViews (step 2: one OffscreenRenderingContext WebView as a surface) and the seL4 virtio-gpu driver-PD (step 3: clone the net-PD) both plug into later. It is the smallest change that makes the compositor cap-first rather than a normal gpui app, and it needs nothing that is not already in starbridge-v2 + sel4/dregg-firmament.

**Open questions**

- Renderer convergence: keep gpui_macos Metal-direct on mac (today's working path) and adopt gpui_wgpu only for the portable/web/seL4 targets (two backends), OR standardize the dregg compositor on gpui_wgpu EVERYWHERE for one renderer at the cost of leaving the mac-optimal Metal path? The lockfile supports both; which is canonical?
- servo↔compositor texture sharing: is the near-term seam a CPU/GPU blit of servo's GL offscreen surface into a wgpu texture (simple, a copy per frame), a GL/Vulkan interop import (zero-copy, fiddly), or do we wait on/contribute-to servo's wgpu renderer (#37149) so servo and gpui share ONE wgpu device? Picks the integration cost.
- Is servo's heft (it drags servo-script/servo-layout/SpiderMonkey + servo-media + a large dep tree, and is rc-quality) worth embedding for v1, vs a thinner 'web surface' (a WebView2/WKWebView-class native webview, or a wgpu-canvas + a minimal HTML subset) until the dregg desktop needs full web fidelity? Web-forward is the vision, but servo-now may be premature.
- gpui itself is single-threaded with an Rc<RefCell<World>> model (per cockpit.rs) — does a multi-surface, multi-app, possibly-multi-PD compositor fit gpui's app/window model, or does the compositor need to own the gpui App and run apps as guests that submit surface-cells (more like a real display server)? This is the 'is gpui the compositor or a client' question the facet poses.
- seL4 virtio-gpu DMA + blob resources: the 2D framebuffer path is clean (guest-RAM backing, the net-PD pattern), but 3D/Venus needs host blob-resource/udmabuf support and host-visible memory — does the macOS-native semihosted-seL4 host (QEMU on mac, no /dev/udmabuf, MoltenVK not Vulkan) actually support Venus, or is the seL4-3D path linux-host-only until bare-metal? Bounds how soon seL4 gets real 3D vs just a framebuffer.
- Where exactly does the firmament's n>1 surface live? A remote WebView / screen-share / another sovereign's surface composited over the net-PD is the natural 'distributed surface' — but is a streamed remote surface a pixel stream (encode/decode, a media problem) or a remote scene-graph (re-render locally)? This decides whether 'distributed compositor' is RFB/WebRTC-class or a genuine distributed-ocap-surface protocol.

## The service-migration fulcrum — host a Linux VM under seL4, then port services (dbus/drivers) one-by-one to seL4-native dregg PDs, using the Linux compatibility VM as the fulcrum to grow the OS.

**Prior art**

- **au-ts/libvmm (Microkit-native VMM)** — A virtual machine monitor LIBRARY for seL4 Microkit (C, AArch64; RISC-V in dev) that boots a full Linux guest as a single `<virtual_machine>` protection domain. The VMM PD traps the guest's virtio-MMIO accesses on guest faults, injects virtual IRQs (hw-irq -> seL4 notification -> VMM -> vIRQ injection through the GIC), translates guest-physical<->host-virtual during fault handling, and — critically — serves each virtio device by forwarding to the seL4 Device Driver Framework (sDDF): 'the VMM performs I/O using the protocols and interfaces provided by sDDF.' Depends on Microkit SDK 2.2.0 — the EXACT SDK dregg already pins. examples/simple boots a buildroot login.  
  *dregg relevance:* THE substrate for the compatibility VM. Same SDK (2.2.0), same board (qemu_virt_aarch64), same Microkit `.system` vocabulary the net-PD already uses. The firmament's `executor-PD blocked on the Lean port` problem is sidestepped: legacy Linux services run in the VM TODAY while native dregg PDs are extracted one at a time. Adopt, do not build — it is a Make-driven library that drops into sel4/dregg-pd/ beside the net driver.
- **sDDF — seL4 Device Driver Framework (au-ts/sddf) + its sDVF evolution** — A collection of interfaces/libraries for writing seL4 device drivers as native PDs that share devices securely + low-overhead. The canonical component shape per device class is driver (owns the device cap + MMIO + IRQ) -> virtualiser/multiplexer (fans one device out to N clients) -> client (a native PD OR a Linux VM) with an optional copier. Transport = shared-memory free/used ring-buffer queue pairs + notification channels. Device classes already supported: network, block/storage, serial, I2C, GPU (virtio 2D), timer; NVMe/MMC block drivers exist. The roadmap explicitly extends sDDF to sDVF (device VIRTUALIZATION framework) to share one device between Linux VMs AND native components — 'one driver per device, multiplexed securely between clients, some of which are VMs.'  
  *dregg relevance:* The migration spine, and dregg ALREADY SPEAKS IT. The net-PD (sel4/dregg-pd/net/) is built on `sel4-shared-ring-buffer` Use/Provide roles + `sel4-microkit-driver-adapters::net` — i.e. it is already an sDDF-shaped driver/client pair. So 'extract a service to native' = add a virtualiser PD between the existing driver and a NEW client (either a native dregg PD or the Linux VM's virtio backend). The same rx/tx free/used queue + notification pattern in net-full.system is the template for EVERY device class.
- **LionsOS / the Kitty reference system (au-ts)** — A complete, real OS built from exactly this stack: native sDDF driver+virtualiser PDs for the simple/critical devices (I2C, serial/UART, timer) composed WITH a Linux VM (libvmm) that hosts the heavy/legacy service (in Kitty: the framebuffer/display + a MicroPython UI), connected via sDDF virtualisers + shared-memory buffers. A `fb` module ships a buffer from the app to the display through the VM. Demonstrates the incremental-migration philosophy directly: keep what's hard in the VM, make native what's tractable, share devices across the boundary.  
  *dregg relevance:* The architectural EXISTENCE PROOF and direct template for the dregg migration path. Kitty IS the firmament's target shape with the executor/cell layer swapped in: native driver+virtualiser PDs + a Linux compat VM + shared-memory device sharing, all under one Microkit image. Copy its assembly topology; replace its app/UI layer with dregg app-PDs and the executor-PD, and its ad-hoc shared-memory protocols with dregg turns.
- **Genode-on-seL4 (the badged-endpoint session mapping) + the seL4 IPC/CapDL model** — Genode's articles show how a capability-based RPC SESSION model (a client requests a named service, gets a session capability, invokes methods on it) maps onto seL4: an seL4 endpoint with a per-client BADGE minted by seL4_CNode_Mint; the server sees capsUnwrapped + seL4_GetBadge() to identify which client/object is invoking. Routing is by capability distribution at init, not dynamic name lookup. Genode also surfaces the real seL4 constraints (single cap-delegation per IPC; I/O-port + IRQ services must be brokered through core).  
  *dregg relevance:* The blueprint for the dbus->dregg-cap-IPC shim. dbus is named-service + method-call + caller-identity; seL4's badged endpoint IS caller-identity-by-construction, and dregg's `(target, rights)` handle (sel4/dregg-firmament/) is already the (badged endpoint | dregg cell) unification. So 'translate a dbus call' = mint a badged endpoint per (service, client) where the badge carries the dregg cap; a dbus method invocation becomes an seL4_Call whose badge the server resolves to a dregg capability/turn. Genode is the proof this mapping is tractable and names its sharp edges.
- **CAmkES-VM / libsel4vmmplatsupport (the older seL4 virtualization stack)** — The original seL4 VMM line: run virtualised Linux guests on ARM/x86 via a CAmkES VMM server that initializes/boots/manages a guest, with virtio PCI/Console/Net backends served by native seL4 components, and hardware passthrough (e.g. cma34cr_centos passes through real devices; 'insecure mode' passes through all). Documents the virtio-net pattern: a virtual NIC presented to the VM, traffic routed to/from native seL4 components.  
  *dregg relevance:* The conceptual ancestor and a fallback/reference for the VM-hosting + virtio-backend + passthrough patterns, but CAmkES (not Microkit) — heavier, build-system-incompatible with dregg's native-macOS Microkit toolchain. Use it for design reference (especially virtio-console and passthrough-device declaration), but prefer libvmm/sDDF which match the existing toolchain. Confirms the 'native component serves the VM's virtual device' pattern is well-trodden.
- **The dregg net-PD + cap-gradation bridge (in-tree: sel4/dregg-pd/net*, sel4/dregg-firmament/)** — Already-built dregg code: a 2-PD net system (driver PD sole-holder of NIC+MMIO+DMA, smoltcp client PD) over sel4-shared-ring-buffer free/used queues + a PPC channel for MAC fetch + an Ed25519 SignedTurn admission gate at the edge (turn_gate.rs: 'DRGT' envelope, verify_strict, bad sigs never reach the heart). Plus dregg-firmament/: one `(target,rights)` handle + a FirmamentRouter dispatching to a LocalBacking (seL4 CNode mint/revoke/call) or a DistributedBacking (real TurnExecutor over a real Ledger), with `granted <= held` enforced at both ends and the n=1 bounds-collapse witnessed.  
  *dregg relevance:* The fulcrum's foundation already exists. The migration path is GENERALIZING this proven pattern: the net-PD's sDDF transport -> every device class's driver/virtualiser; the turn_gate edge -> the dbus->turn shim; the firmament router's two backings -> (local seL4 service | distributed dregg cell) for migrated services. 'Weld, not build' — the census shows the capability already exists, disconnected.

**Design for dregg**

THE FULCRUM RESHAPE: the firmament's executor-PD heart sits beside a Linux COMPATIBILITY VM (libvmm PD), and every device/service is incrementally lifted from 'inside the VM (Linux driver)' to 'native dregg driver-PD on sDDF', with a dbus->turn shim turning Linux service calls into dregg turns. Three layers:\n\n(1) THE COMPAT SUBSTRATE. A libvmm VMM-PD boots an unmodified aarch64 Linux/buildroot guest as a `<virtual_machine>` (guest_ram memory_region, one vcpu, GIC vcpu map). Day one the VM owns the devices via virtio backends the VMM emulates — so a full Linux userspace (dbus, drivers, apps) runs immediately. This is the portability fulcrum: best-on-mac/linux semihost OR real seL4, the SAME Linux compat layer, no Lean-port blocker on the critical path. The VM is a dregg app-PD writ large: it gets ONLY the caps the firmament mints it.\n\n(2) THE sDDF MIGRATION SPINE (the one-by-one extraction). dregg's net-PD is already an sDDF driver/client pair, so generalize it: for each device class add the canonical trio driver-PD (owns the real device) -> virtualiser-PD (multiplexes) -> {native dregg client-PD | the VM's virtio backend}. EXTRACTION ORDER, easiest/highest-leverage first: (a) NETWORK — DONE-shaped: the existing net driver-PD becomes the backend for BOTH the VM's virtio-net AND native dregg ingress, via a net virtualiser; the Linux guest's packets and dregg turns share one NIC. (b) TIMER — trivial sDDF class, needed by smoltcp/the executor; native first. (c) SERIAL/console — sDDF serial + libvmm virtio-console; the VM's tty becomes a native-multiplexed device. (d) BLOCK/STORAGE — the persist-PD becomes an sDDF block driver (redb-over-raw-block-cap, FIRMAMENT.md §3); the VM mounts a virtio-blk backed by it, so guest files land in dregg's verified snapshot/overlay store with the root tooth. (e) INPUT then (f) DISPLAY/framebuffer last (hardest; Kitty keeps graphics in the VM longest — webgpu/servo compositing is the eventual native target). Each extraction is staged-additive-then-cutover: the native PD lands beside the VM's driver, the virtualiser routes to it, then the VM's in-guest driver is disabled.\n\n(3) THE dbus->dregg-cap-IPC SHIM (the semantic bridge). A native dregg 'bus-broker' PD presents a dbus system-bus endpoint to the VM (a vsock/virtio-serial transport the guest's libdbus speaks to). Each (service, client) pair is a badged seL4 endpoint (Genode's pattern) whose badge carries a dregg capability; a dbus method-call on a migrated service becomes an seL4_Call the broker resolves badge->cap->dregg turn through the executor-PD, returning the receipt as the dbus reply. So a Linux app calling org.dregg.Storage.Put runs a real mandate-gated dregg turn. Unmigrated services stay intra-VM dbus; migrated ones are transparently dregg cells — the SAME fluid local<->distributed reach-out the firmament router already embodies, now extended to the Linux/dregg seam. dbus names map to dregg cells via a directory cell (the rbg DirectoryCell, M2, already boots), so the bus's well-known-name table IS a dregg c-list.\n\nThe whole thing is the firmament's n=1 thesis applied to legacy: the Linux VM is just another app-PD; its services migrate from VM-internal to native-dregg one at a time without ever taking the system down, and each migrated service inherits dregg's verified turns, attenuable caps, and snapshot/restore.

**Key primitives**

- libvmm VMM-PD hosting an unmodified aarch64 Linux guest as a Microkit `<virtual_machine>` (guest_ram + vcpu + GIC-vcpu map; the compat substrate, no Lean-port dependency)
- The sDDF driver -> virtualiser -> client trio per device class, over sel4-shared-ring-buffer free/used queue pairs + notification channels — generalizing the EXISTING net-PD transport to every device
- The virtio-backend bridge: libvmm traps the guest's virtio-MMIO on fault and forwards to a native sDDF virtualiser, so the VM is a CLIENT of the same native driver dregg apps use (the sDVF cross-VM-sharing pattern)
- Badged seL4 endpoints as the dbus caller-identity primitive (Genode pattern): badge carries a dregg capability; a dbus method-call -> seL4_Call -> badge-resolve -> dregg turn through the executor-PD
- The dregg `(target, rights)` firmament handle + FirmamentRouter (ALREADY BUILT) extended with a third backing: Local seL4 object | Distributed dregg cell | VM-hosted Linux service (migrating left across the trichotomy)
- A bus-broker PD presenting a dbus system-bus to the guest over virtio-serial/vsock, with the well-known-name table backed by the rbg DirectoryCell (M2, boots) as a dregg c-list
- The Ed25519 SignedTurn admission gate (turn_gate.rs, built) as the template for the VM->dregg trust boundary: a Linux service request is de-enveloped + cap-checked before it reaches the verified executor
- Per-device extraction as staged-additive-then-cutover: native PD lands beside the in-VM driver, the virtualiser cuts traffic over, the guest driver is disabled — one device class at a time, never a flag-day

**First step**

Boot au-ts/libvmm's `examples/simple` Linux guest under the EXISTING dregg toolchain (Microkit SDK 2.2.0, native-macOS, qemu_virt_aarch64) as a new `sel4/dregg-pd/vmm/` PD + a `vm.system` assembly — proving the compat substrate boots beside the firmament's PDs with zero new toolchain. Then make ONE concrete extraction: wire the guest's virtio-net backend to the ALREADY-BOOTING dregg net driver-PD (sel4/dregg-pd/net/) by inserting a minimal net virtualiser-PD between them, so a process inside the Linux VM and the native smoltcp client-PD share the one NIC over the same rx/tx free/used rings net-full.system already declares. Success = a `ping`/`curl` from inside the Linux guest egresses through the dregg-owned virtio-net driver (serial-captured), exactly as the M3 net probe was captured. This is the smallest end-to-end demonstration of the fulcrum: a legacy Linux service (the guest's network stack) running on top of a native dregg driver-PD, with the migration path (replace the guest's stack with native dregg ingress) now a virtualiser-routing change rather than a rewrite. It is additive (does not touch the Lean executor-PD blocker) and reuses the proven net-PD sDDF transport wholesale.

**Open questions**

- Transport for the dbus shim: virtio-serial (simplest, libvmm has virtio-console) vs a vsock-style channel vs a custom virtio device — which gives the guest's libdbus the least-patched path while keeping the badge->cap resolution clean? Does the guest need an unmodified dbus or a dregg-aware dbus broker shim installed in-guest?
- sDDF in C vs dregg's Rust PDs: sDDF/libvmm are C; dregg's PDs are rust-sel4. The net-PD already bridges this (it uses the rust sel4-shared-ring-buffer matching sDDF's queue ABI) — but do the virtualisers get written in Rust against the sDDF queue layout, or do we adopt sDDF's C virtualisers and only write Rust clients? (ABI compatibility of the ring-buffer/queue structs is the load-bearing question.)
- Where does the executor-PD blocker bite? The dbus->turn shim and any 'migrated service runs a real dregg turn' path need the executor-PD, which is still blocked on the Lean ELF/libuv-excision/musl port. Can the early migration steps (network, timer, serial sharing) be PURELY sDDF device-sharing — no turns — so the fulcrum demonstrates value BEFORE the executor-PD lands, with the dbus->turn shim queued behind it?
- Display/input migration is the hardest and the ember-vision endgame (native verified graphics, servo/webgpu). Does the Linux VM keep the GPU via passthrough indefinitely while dregg builds native cap-first compositing, or is there an intermediate (virtio-gpu 2D via sDDF, which sDDF already supports) worth targeting?
- Guest determinism vs the houyhnhnm contract: a Linux VM is NOT houyhnhnm (it has wall-clock, RNG, ambient nondeterminism). How does a non-deterministic compat VM coexist with the firmament's deterministic-replayable contract — is the VM explicitly OUTSIDE the houyhnhnm boundary (a legacy quarantine that migrated services LEAVE as they go native), and is that the right framing for the trust story?
- The semihost/'robigalia emulator' fulcrum: does the mac/linux-native semihost run libvmm-equivalent Linux hosting (nested virt / a user-mode Linux) or does the semihost simply run the dregg PDs natively and SKIP the VM (since the host IS Linux/mac)? I.e. is the compat-VM only meaningful on real seL4, with the semihost using the host OS directly as the compat layer?

## The real seL4 substrate roadmap toward a bootable dregg DESKTOP image (executor-PD hosting → driver PDs → compositor PD → dregg.system desktop assembly), building on the firmament / executor-pd / net-client that exist today.

**Prior art**

- **rust-sel4 sel4-root-task-with-std + sel4-musl (the SDK's hosted-std substrate)** — sel4-musl (~/sel4-sdk/rust-sel4/crates/experimental/sel4-musl) emulates the musl syscall surface inside an seL4 PD; sel4-root-task-with-std (crates/private/support/) is a root-task runtime that gives malloc/pthread/TLS/C++-exceptions. Together they are the documented host for a std/libc payload on seL4.  
  *dregg relevance:* THE exact substrate the executor-PD step 4 needs. The verified Lean executor already runs as a static aarch64-linux-musl ELF (out/dregg-executor.elf, 0 undefined, status:2 ok:1 on a real turn — executor-pd/WALL.md). sel4-musl is the in-PD musl the executor was built against; hosting that object set on root-task-with-std is the single next build that makes the verified executor a booting seL4 PD.
- **rust-sel4 sddf + the http-server multi-PD example (virtio-blk-driver / virtio-net-driver / smoltcp server PDs)** — The seL4 Device Driver Framework (sddf) + sel4-microkit-driver-adapters + sel4-shared-ring-buffer(-smoltcp) + sel4-driver-interfaces (block/net/rtc/serial/timer). The http-server example is a canonical N-PD assembly: a device-driver PD owns the device cap, a client PD speaks to it over shared ring buffers + a notification channel.  
  *dregg relevance:* This IS the pattern dregg's net edge already copied (net-client.rs uses DeviceImpl over sel4-shared-ring-buffer-smoltcp; net.system mirrors the http-server net half). The same driver-PD-owns-the-cap + ring-buffer-to-client pattern is the template for the persist-PD (virtio-blk driver already exists in-SDK) and for the new display/input driver PDs.
- **rust-sel4 drivers/virtio/{blk,net} + virtio-mmio HAL** — In-SDK virtio-blk and virtio-net MMIO drivers with a shared hal-impl. The dregg net driver PD already probes a real QEMU virtio-net-device on the mmio bus (device_type=Network, NIC up with MAC — FIRMAMENT §6).  
  *dregg relevance:* virtio-blk gives the persist-PD its block backend (redb-over-block-cap, SEL4-EMBEDDING §3) for free. CRUCIAL GAP: the SDK ships NO virtio-gpu and NO virtio-input driver, and neither is on the local crates mirror — the compositor's display+input path is genuinely greenfield (must carry the external virtio-drivers crate, or write sddf-style gpu/input driver PDs).
- **LionsOS (lionsos crate in-SDK) — the reference seL4 desktop/IO-system architecture** — The Trustworthy-Systems reference OS built on Microkit + sDDF: a static assembly of driver PDs, virtualiser PDs (multiplex one device across clients), and client PDs, with a serial/console multiplexer and a fault-handler PD.  
  *dregg relevance:* The architectural model for the dregg desktop's later rungs: the compositor PD is exactly a display virtualiser (one virtio-gpu, many app-PD surfaces) and the input PD is an input virtualiser (one virtio-input, routed to the focused app). Adopt LionsOS's driver/virtualiser/client split rather than inventing one.
- **the dregg-firmament cap-gradation crate (sel4/dregg-firmament/)** — One (target, rights) Capability handle that resolves to a local seL4 CNode/endpoint OR a distributed dregg cell through a Router, attenuating on the SAME real dregg is_attenuation (granted ⊆ held) at both ends; Bounds carry the n-parametrized relaxation; the n=1 collapse is witnessed; tests run the real TurnExecutor in the loop.  
  *dregg relevance:* This is the desktop's window/surface-handle model already in code. A compositor surface, an input focus token, and a file handle become firmament Capabilities whose Target is a local seL4 frame/endpoint (the compositor's shared framebuffer) — so the desktop's UI objects ride the existing local↔distributed bridge with zero new authority model.
- **executor-pd ELF excision pipeline (cross-compile-closure / build-leanrt-elf / build-gmp-elf / link-probe)** — Recompiles the 757-facet verified Lean closure to ELF aarch64, rebuilds an ELF Lean runtime bottom-half (leanrt + Init/Std/Lean/mathlib/deps + kernel C++) from lean4@d024af099 with the 10 libuv objects excised + real GMP 6.3.0, links a static musl executor that runs one real verified turn.  
  *dregg relevance:* The hard, banked breakthrough the whole desktop heart stands on. Steps 1-3 are GREEN; the roadmap's first rung is purely step 4 (host the existing object set), NOT re-deriving the runtime. Any desktop image's executor-PD reuses these scripts verbatim.
- **starbridge-v2 (gpui master interface, embedded executor) + its sel4-thin feature** — A gpui-rendered cockpit that EMBEDS the real verified executor (world::World over dregg_turn::TurnExecutor) for a live local dregg world; a sel4-thin feature builds a Lean-free thin HTTP/verifier client for 'the eventual seL4 component' (its own main.rs names docs/SEL4-EMBEDDING.md).  
  *dregg relevance:* The concrete compositor-PD payload target and the proof that the desktop face already speaks two builds: native-full (host, today) and sel4-thin (the seL4 client path). The desktop roadmap's top rung is making the gpui surface render through a dregg compositor PD over a virtio-gpu framebuffer instead of host Metal.
- **Microkit SDK 2.2.0 (native macOS-aarch64) + the .system assembly model** — The seL4 Foundation's native-macOS Microkit: the microkit image tool + prebuilt seL4 kernel ELFs for qemu_virt_aarch64/riscv64. A .system XML declares PDs, memory_regions (incl. device regions by phys_addr), maps, IRQs, and channels; the tool links them into one bootable image. dregg.system already assembles 5 PDs this way.  
  *dregg relevance:* The desktop image IS a grown dregg.system: add display/input driver-PD seats + a compositor-PD + app-PD seats, each with its device/frame caps partitioned. The build/boot recipe (make run-assembly) and the platform ladder (aarch64→riscv64, M5 already boots) are in place; the desktop is more PDs in the same proven harness.

**Design for dregg**

A four-rung seL4 substrate ladder from the verified executor that runs TODAY to a bootable dregg desktop, each rung a real QEMU boot in the existing Microkit harness, each adding ONE PD to a growing dregg.system. The invariant throughout: a driver PD is the SOLE holder of its device cap, clients reach it only over shared ring buffers + a notification channel, and every UI object (surface, focus, file) is a firmament Capability so the desktop never invents authority outside the local↔distributed gradation.

RUNG 1 — THE HEART HOSTS (executor-PD as root-task-with-std). The only blocker between today and a verified-executor-on-seL4 is step 4 of the banked excision: host the existing static-musl executor object set on the SDK substrate. Build musl + sel4-musl for aarch64, wire its syscall handler, link out/dregg-executor.elf's archive set into a sel4-root-task-with-std PD, embed the one-turn driver (scripts/driver.c is the PD-main seed) feeding wideDemoInput, boot under qemu-system-aarch64. Decision §8.3 says start as a root task (simpler, weaker isolation) then graduate to a Microkit-PD musl substrate. Deliverable: the verified execFullForestG prints status:2 ok:1 over serial on seL4 — the firmament's first true heartbeat — and the executor seat in dregg.system stops being a stub. The crypto-floor 8 dregg_* symbols (panic-stubbed in the host probe) are supplied here by the Rust PD wrapper, as WALL.md already specifies.

RUNG 2 — THE DURABLE + EDGE SEATS GO REAL (persist-PD + net 2-PD). Land the persist-PD as the sole holder of a virtio-blk device cap using the in-SDK virtio-blk driver + sel4-driver-interfaces::block, backing redb over a raw block cap (SEL4-EMBEDDING §3; content-addressed Blob=frame means no FS/inode layer needed). Complete the net edge: boot net.system as the real 2-PD system (driver PD owns the NIC + virtio-mmio + DMA paddr + IRQ 79; net-client PD runs the smoltcp DHCP/echo + the Ed25519 SignedTurn admission gate that already exists in turn_gate.rs). Close the M3 walls named in README: align the QEMU virtio-mmio slot (0xa003000 / offset 0xe00, slot 31) and wire the 2-PD ring-buffer channel. Then route the gate's accepted message across the existing ingress→executor channel in dregg.system so a SignedTurn arriving over TCP reaches the Rung-1 executor and commits to the Rung-2 persist store. Deliverable: a real turn submitted over the wire, executed by the verified heart, persisted to a block-cap store — the n>1 invocation path of the cap gradation, end to end, on seL4.

RUNG 3 — THE DISPLAY + INPUT DRIVER PDs (the genuinely greenfield rung). seL4/rust-sel4 ships NO virtio-gpu or virtio-input driver, so this is the real new construction. Write two sddf-pattern driver PDs following the exact net-driver template (device-cap-sole-holder + shared-ring-buffer-to-client): (a) a virtio-gpu driver PD that owns the GPU mmio + DMA region, sets up a 2D scanout, and exposes a shared framebuffer region + a flush notification; (b) a virtio-input driver PD that owns the input device + IRQ and pushes keyboard/pointer events into a shared event ring. Reuse the virtio-mmio HAL from drivers/virtio; carry the external rcore-os virtio-drivers crate (it has gpu+input) into the PDs OR extend sel4-driver-interfaces with display/input traits and write sddf-native drivers (preferred for the steady-state, matches block/net). Boot each driver PD standalone first (its own .system, like net-driver-only.system) against QEMU -device virtio-gpu-device / virtio-keyboard-device, proving the probe + a test pattern on screen, before assembly. Deliverable: pixels on a QEMU window driven by an seL4 driver PD, and keystrokes arriving at a client PD — the desktop's eyes and hands.

RUNG 4 — THE COMPOSITOR PD + THE BOOTABLE dregg.system DESKTOP. Add a compositor PD that is a LionsOS-style display virtualiser: it holds the framebuffer cap from the gpu driver, owns the input focus, and multiplexes many app-PD surfaces. Each app-PD gets a surface as a firmament Capability whose Target is Local{frame} (a shared frame the compositor composites) and an input-focus token (a badged endpoint) — so the desktop's window manager IS the cap fabric (§5 no-ambient-authority falls out: an app can only draw to its granted surface frame, can only receive input when it holds focus). The compositor renders the dregg world the executor PD runs (the starbridge-v2 cockpit content), reached over the executor↔compositor channel. App launch = the factory→seL4_Untyped_Retype port (RBG-TO-SEL4.md) minting a new app-PD's CNode + surface frame. Grow dregg.system to: executor (real, Rung 1) · persist (real, Rung 2) · net driver+client (real, Rung 2) · verifier-stark (already real) · gpu-driver · input-driver · compositor · app-PD(s). Deliverable: make run-desktop boots ONE Microkit image where a houyhnhnm app draws a window, takes input, and every action it takes is a verified replayable turn through the executor PD — the bootable dregg desktop.

Cross-cutting: stay on the aarch64 QEMU platform for all four rungs (M5 proves the riscv64 retarget is a rebuild, not a rewrite — defer it and hardware to after the desktop boots in QEMU). gpui-on-seL4 is the LONG pole behind Rung 4's payload (gpui wants a full GPU/windowing stack); the staged fallback is a thin dregg-native compositor drawing the cockpit's reflective model directly to the framebuffer (the sel4-thin/headless surfaces already exist), with gpui/servo/webgpu compositing as the post-desktop vision target.

**Key primitives**

- root-task-with-std executor host: sel4-musl syscall shim + sel4-root-task-with-std runtime hosting the existing static-musl verified-executor object set (out/dregg-executor.elf) — the ONLY thing between today and verified-execFullForestG-on-seL4; reuses executor-pd/scripts verbatim, crypto-floor symbols supplied by the Rust PD wrapper
- driver-PD-owns-the-cap pattern (the firmament's structural law): every device (block, NIC, GPU, input) lives in exactly one PD that is the sole holder of its mmio region + DMA paddr + IRQ; clients reach it ONLY over sel4-shared-ring-buffer(+notification). Already realized for net; the template for persist (virtio-blk, in-SDK) and the new gpu/input PDs
- virtio-gpu + virtio-input driver PDs (the greenfield rung): NOT in rust-sel4 or the local mirror — write sddf-pattern PDs over the virtio-mmio HAL, either carrying the external rcore-os virtio-drivers crate or extending sel4-driver-interfaces with display/input traits; a shared framebuffer region + flush notification (gpu), a shared event ring (input)
- compositor-PD as a LionsOS-style display virtualiser: holds the framebuffer cap, owns input focus, multiplexes app-PD surfaces; the window manager IS the cap fabric — a surface is a firmament Capability(Local{frame}), focus is a badged endpoint, so no-ambient-authority (FIRMAMENT §5) falls out structurally
- firmament Capability as the desktop's UI-object model (sel4/dregg-firmament/): window surface / input focus / file handle are all one (target, rights) handle attenuating on the real dregg is_attenuation at both local-seL4 and distributed-dregg ends — the desktop invents zero new authority, rides the n=1 collapse for strong local guarantees
- the growing dregg.system assembly + Microkit native-macOS harness: the desktop image is dregg.system with added gpu/input/compositor/app PD seats, each with partitioned device/frame caps; built+booted by make run-assembly→run-desktop on qemu_virt_aarch64, the same proven harness that already boots 5 PDs
- redb-over-raw-block-cap persist backend: the persist-PD as sole virtio-blk device-cap holder, content-addressed (Blob=frame) so no FS/inode layer is needed (SEL4-EMBEDDING §3) — the durable seat made real
- app-launch = factory→seL4_Untyped_Retype (RBG-TO-SEL4.md): minting a new app-PD's CNode + surface frame is the kernel-enforced form of the dregg factory slot-caveat — capability-secure creation becomes a kernel invariant

**First step**

Land Rung 1: host the already-built static-musl verified executor as a sel4-root-task-with-std PD and boot it on qemu_virt_aarch64. Concretely, in sel4/dregg-pd/executor-pd/: (1) build musl for aarch64 + bring in sel4-musl from the SDK (~/sel4-sdk/rust-sel4/crates/experimental/sel4-musl) and wire its syscall handler; (2) link out/dregg-executor.elf's archive set (the ELF closure + ELF leanrt + GMP bottom-halves already produced by scripts/{cross-compile-closure,build-leanrt-elf,build-gmp-elf}.sh) into a sel4-root-task-with-std PD, with scripts/driver.c's one-turn harness as the PD main and the 8 dregg_* crypto-floor symbols supplied by the Rust PD wrapper; (3) feed it the banked wideDemoInput wire and boot via the Makefile's LINK_AND_RUN pattern. Success = the verified execFullForestG prints status:2 ok:1 over serial ON seL4 (not just host-musl), turning the dregg.system executor seat from stub into the real firmament heart. This is pure step-4 hosting — the hard runtime derivation (steps 1-3) is already GREEN — so it is the most tractable high-value build, and it unblocks routing the net edge's accepted SignedTurn into a real on-seL4 execution at Rung 2.

**Open questions**

- Executor-PD runtime shape (FIRMAMENT §8.3, still open): start as a root-task-with-std (simpler, weaker isolation, what sel4-musl supports today) then graduate to a Microkit-PD musl substrate (the steady-state dregg.system shape), or invest in the Microkit-PD musl substrate up front? The SDK's std substrate is experimental + root-task-only.
- virtio-gpu/input driver provenance: carry the external rcore-os virtio-drivers crate (has gpu+input, fast path) vs. write sddf-native drivers extending sel4-driver-interfaces with display/input traits (matches the in-SDK block/net pattern, the steady-state, but more work). Neither gpu nor input exists in rust-sel4 or the local crates mirror today.
- gpui-on-seL4 feasibility (the Rung-4 payload long pole): can the starbridge-v2 gpui cockpit render over a virtio-gpu framebuffer in a PD, or is the realistic v0 a thin dregg-native compositor drawing the cockpit's reflective model directly to the framebuffer (gpui needs a full GPU/windowing stack; servo/webgpu is the stated post-desktop vision, not the first desktop)?
- GMP strategy for the executor on the bare target (FIRMAMENT §8.2): real GMP 6.3.0 is already cross-built and GREEN, but is a fixnum-only shim viable — does any verified turn ever exceed 63-bit fixnums? If not, the shim deletes a whole C dependency from the desktop heart (needs a kernel numeric-range check).
- Persist crash-consistency on a raw block cap: redb assumes mmap; over a raw block cap (no POSIX file) does the existing crash-consistent commit-log design (persist/src/commit_log.rs) hold, or does the block-cap backend need its own journaling? The verified-FS line (BilbyFs/Cogent) is explicitly external/adopt-not-build and beyond the desktop horizon.
- Checkpoint/restore wiring (FIRMAMENT §4, quarter+): the design marries a PD checkpoint to a dregg snapshot, but PD-checkpoint machinery on Microkit is unbuilt — is it on the desktop critical path (for app suspend/resume) or deferrable until after the desktop boots?
- Multi-app input/display multiplexing semantics: when the compositor virtualises one virtio-gpu + one virtio-input across many app-PDs, what is the focus/Z-order/clipping authority model in firmament-Capability terms — is a per-app surface frame + a single focus token sufficient, or does it need a richer compositor protocol (damage rects, cursor ownership)?

## VERIFIED-GRAPHICS / TRUSTED-PATH — the compositor as a verified dregg cell, output-integrity as an unfoolability property, and the trusted path against UI-spoofing (the future seL4 north star; the most research-y facet)

**Prior art**

- **EROS Trusted Window System (EWS) — Shapiro, Vanderburgh, Northup, Chizmadia (USENIX Security 2004)** — A from-scratch trusted window system for the EROS capability OS. Window-system objects (windows, the session, the event source) ARE capabilities a client holds; the window system mediates ALL user input and output. Its headline property is 'robust traceability of user volition' — the system guarantees a keystroke/click is delivered ONLY to the program the user intended (a password typed at app A cannot leak to app B), and it can enforce mandatory access control / information-flow on the display. A double-buffered display design 'largely eliminates covert channels from the display system.' Under 4,500 LOC — a factor of 10 smaller than Trusted X — explicitly so it is small enough to evaluate for high assurance.  
  *dregg relevance:* The closest ancestor: a capability OS where the window system is a small, evaluable, capability-mediated component, and the guarantee is phrased as 'user volition is traceable.' This is EXACTLY dregg's unfoolability/anti-ghost cast onto the I/O path: an input event must provably reach the cell the user chose, and an output pixel must provably come from the cell that owns it. 'Traceability of user volition' is the input-side dual of dregg's output-integrity. The covert-channel-free double-buffered display is a concrete mechanism the dregg compositor-cell should adopt. dregg's win over EWS: the window-system component is not just small-and-auditable but Lean-verified-by-construction, and its object model IS the cell forest the rest of dregg already proves about.
- **Nitpicker secure GUI server — Feske & Helmuth (ACSAC 2005); the Genode GUI stack** — An ~1,500-LOC secure GUI server. Clients open a GUI session and create 'views' onto a shared pixel buffer; views are grouped into 'domains' with independent policy, and each domain occupies a fixed layer in a global view stack. Trusted path is achieved by (a) per-domain LABELING — a colored frame + textual watermark naming the owning client, placed at the topmost position where it stays visible — so a Trojan that mimics a trusted app's pixels is exposed by its label; (b) TINTING — untrusted content is blended with a domain color unless it currently holds input focus, making the focused (trusted-path) surface visually unambiguous; (c) FOCUS-MEDIATED INPUT — keyboard/mouse go only to the focused domain, with an external focus-policy component able to override the default click-to-focus. The server is the entire TCB for GUI; it reports only the screen region/size each domain may see (no inference of hidden UI).  
  *dregg relevance:* The directly-portable mechanism set. Nitpicker's 'domain' = a dregg cell; its 'view' = a cell's surface; its per-view label = the cell's authenticated identity (the CellId / state-root the user can verify); its focus-mediated input = the secure-attention/trusted-path gate. The labeling+tinting trick is the minimal usable form of output-integrity: the user is told, unspoofably, WHICH cell produced each surface and WHICH surface their input reaches. dregg upgrades the label from 'the server asserts this client's name' to 'this label is bound to the cell's verified state-root' — the label becomes an unfoolability claim, not a server courtesy. Nitpicker's 1,500-LOC minimality is the existence proof that a trusted-path compositor TCB can be small enough to verify.
- **seL4 Device Driver Framework (sDDF) — Parker, Heiser et al.; LionsOS** — A framework for running device drivers as unprivileged user-level seL4 protection domains. A driver PD's authority is 'reduced to just the ability to access the control registers of the device it drives' (a frame cap to the MMIO region, an IRQ cap), and it talks to clients over an asynchronous shared-memory transport (lock-free ring buffers in shared frames) — no kernel involvement per operation, yet performance competitive with Linux. The bounded authority + clear shared-memory protocol gives a small, well-defined verification scope; LionsOS is driving these components toward functional-correctness proofs (PISTIs-V, SMT-based Microkit-component verification). Current verified seL4 configs do NOT yet cover IOMMU/SystemMMU address translation for DMA devices.  
  *dregg relevance:* The blueprint for the GPU/display/input drivers in the seL4 north-star. A verified GUI needs a display driver and an input driver as PDs; sDDF says each is a PD holding ONLY its device-register frame cap + IRQ, exchanging framebuffers over shared-ring transport. This is the firmament cap-gradation made physical for hardware: a 'frame cap to the display MMIO' is just another point on the same (target, rights) handle. The honest caveat sDDF surfaces — DMA/IOMMU is NOT yet verified — is dregg's exact frontier-vs-now line: a GPU doing DMA can scribble any framebuffer unless an IOMMU cap confines it, and that confinement is currently unverified everywhere. dregg should adopt sDDF's transport for the compositor↔driver edge so the verified scope stays bounded.
- **Secure Attention Key / Sequence (SAK/SAS) — classical trusted-path primitive (Windows Ctrl-Alt-Del, Linux SAK)** — A key combination that ONLY the kernel can intercept (the kernel holds an exclusive channel to the keyboard driver; any process trying to grab that combo is denied). Pressing it suspends all userspace — including a login-spoofer masquerading as the login screen — and hands control to a trusted path the user knows is genuine. Defeats login-spoofing because the trust anchor is the one component (the kernel) that physically owns the input device.  
  *dregg relevance:* The hardware root of the trusted path, and the one piece that CANNOT be a normal cell — it must be a kernel/root-task-mediated reserved gesture. On seL4 the 'kernel owns the keyboard' becomes 'the root-task (or a tiny trusted-path PD) holds the SOLE keyboard-IRQ/input cap and recognizes a reserved secure-attention gesture'; on the semihosted-mac/linux emulator it becomes a reserved chord the host shell delivers only to the trusted-path component. This is what lets the user summon the unspoofable 'which cell am I really talking to?' overlay (the cipherclerk/palette in starbridge-v2 is the natural home). Without a SAK, output-integrity labeling can itself be overpainted; the SAK is what makes the labels checkable on demand.
- **Display/output-integrity verification (immersive-web trusted UI 2020; functional-safety display-integrity patents; HDCP/HACP content-path signatures)** — A body of work on guaranteeing that what reaches the physical display matches what an application intended: trusted indicators users can rely on when untrusted content is composited (overlay/clickjacking defense), and functional-safety schemes that sample the framebuffer / sign the content-path connection-state and authenticate that the pixels emitted equal the pixels intended (ASIL display integrity; signed attach-point + display-device path).  
  *dregg relevance:* The modern, web-forward framing of output-integrity and the closest external statement of dregg's goal for the COMPOSITING case (untrusted web content + servo/webgpu). Two transferable ideas: (1) trusted indicators for overlay/clickjacking defense map onto dregg's per-cell labels + tinting of unfocused surfaces; (2) content-path signing maps onto binding the scanned-out framebuffer to the producing cell's verified state-root — the strongest form of dregg output-integrity (a 'frame attestation': this pixel region = projection of cell C at state-root R). The patents confirm the hard part is the LAST hop (framebuffer→panel over an untrusted GPU/DMA path), which is precisely dregg's research frontier.

**Design for dregg**

THE CORE REFRAME: output-integrity IS unfoolability applied to the display path. dregg already proves (AssuranceCase.lean `unfoolability_guarantee`) that a light client checking only `verify root = true` learns the whole history evolved correctly — it cannot be fooled by 'the pale ghost' (a server lying about protocol state). The verified-graphics facet asks the SAME question one hop further out: can the HUMAN at the glass be fooled? The pale ghost on the display is a UI that paints pixels claiming 'cell C holds balance B / you are signing transfer T' when C does not, OR steals the keystroke meant for C and routes it to attacker A. So the property is: (output) every surface region is the genuine projection of its owning cell's verified post-state; (input) every input event is delivered only to the cell the user demonstrably chose. Both are anti-ghost teeth on the I/O boundary.\n\nTHE COMPOSITOR IS A VERIFIED DREGG CELL. Model the compositor as one cell whose state is the scene graph: an ordered list of SURFACES, each surface = (owningCellId, regionRect, contentDigest, sourceStateRoot, zLayer, focusFlag). Compositing is a turn: a cell submits a 'present(region, contentDigest @ myStateRoot)' effect against the compositor cell; the executor's caveat gate enforces the security invariants AS the existing per-effect gates do for transfers. The invariants become anti-ghost teeth proved with the existing EffectCommit machinery: (T1 NON-OVERLAP / no-overpaint) a cell may write only regions its capability authorizes — granted⊆held on screen rectangles, the SAME is_attenuation lattice the firmament uses, just with Rights = a region-set; a surface overpainting another cell's region is UNSAT. (T2 LABEL-BINDING) every surface carries its owningCellId and sourceStateRoot, and the rendered label is a function of those — a surface whose claimed label ≠ its owning cell is UNSAT (the Nitpicker label upgraded to a verified binding). (T3 FOCUS-EXCLUSIVITY) at most one surface has focusFlag=true; input is routed to it; a turn that delivers input to a non-focused cell, or sets two focus flags, is UNSAT (EROS 'traceability of volition' as a state invariant). (T4 NO-INFERENCE) a cell's present() can read only its own region's prior contents (the Nitpicker screen-size-per-domain rule = a capability-scoped read). The compositor cell's state-root then commits the WHOLE scene; the existing `state_root()` / Ledger::root tooth machinery applies unchanged, so 'the scene the user sees' inherits the same root-binding + conservation story as the ledger.\n\nTHE TRUSTED PATH = SAK + the verified label overlay. The one thing that cannot be a normal cell is the secure-attention gesture: a tiny trusted-path PD (or the root task) holds the SOLE input cap and the SOLE top-zLayer surface cap. A reserved gesture (seL4: a kernel-routed key combo via the trusted PD's exclusive keyboard-IRQ cap; semihosted emulator: a reserved host chord) suspends normal input and draws an unspoofable overlay — owned by the trusted PD, at a zLayer no cell can reach — that shows, for the focused surface, the genuine (owningCellId, sourceStateRoot) read straight from the compositor cell state, NOT from the app. This is the cipherclerk/Cmd-K palette in starbridge-v2 promoted to a trust anchor: 'who am I really talking to, and is its state-root the one my light client verified?' Because the trusted PD's surface cap is at a layer outside every cell's region grant (T1), no cell can overpaint it.\n\nWHAT'S VERIFIABLE NOW vs THE FRONTIER. NOW (pure Lean + the existing toolkit, no new hardware trust): the compositor cell's scene-graph invariants T1–T4 as anti-ghost teeth via EffectCommit/VerificationToolkit (an AppSpec whose slots are surfaces, whose admit-predicate is the non-overlap+focus-exclusivity invariant) — this is a real verified-by-construction result with zero new axioms, runnable on the semihosted emulator TODAY and in starbridge-v2's embedded executor. The firmament handle extends cleanly: a surface cap is a (Target::Surface{compositorCell, regionId}, Rights=regionSet) point on the same gradation — local seL4 framebuffer-frame-cap vs distributed present()-turn, same verbs. FRONTIER (named hardware-trust assumptions, the honest seams): (F1) the LAST HOP — binding the scanned-out framebuffer to the cell's contentDigest requires either a verified display driver that hashes what it scans out (a frame attestation: 'panel showed digest D for region R') or trust in the GPU; with an untrusted DMA-capable GPU this is a named assumption exactly as sDDF flags IOMMU as unverified. (F2) IOMMU/DMA confinement — a display/GPU PD doing DMA can scribble arbitrary framebuffer regions unless an IOMMU cap confines it to its granted regions; seL4 does not yet verify SystemMMU translation, so 'T1 holds against a malicious driver' is currently an assumption, not a theorem (this is THE crypto-floor-equivalent named primitive for graphics). (F3) verified GPU/servo/webgpu compositing — out of reach; the near-term honest stance is a SOFTWARE compositor cell (CPU blit, à la EROS/Nitpicker) where T1–T4 are real, with GPU acceleration as an explicitly-trusted fast path. The design discipline mirrors the rest of dregg: drive every seam to ONE worthwhile semantics, hold it at the layer that can bear it (the compositor cell holds the scene invariants; the driver/IOMMU holds the named hardware assumptions), and label the frontier seams as severe-problems-with-closure-lanes, never walls.

**Key primitives**

- SURFACE CELL / scene-graph cell: the compositor modeled as one verified dregg cell whose state is an ordered list of surfaces = (owningCellId, regionRect, contentDigest, sourceStateRoot, zLayer, focusFlag); compositing = a present() turn through the executor
- REGION-SET as Rights on the firmament lattice: screen rectangles become a point on the SAME is_attenuation (granted⊆held) lattice as AuthRequired — a surface cap is (Target::Surface, Rights=regionSet); no-overpaint is attenuation on screen real estate
- OUTPUT-INTEGRITY TOOTH (anti-ghost on the display): a surface whose claimed (label/owner/state-root) ≠ its owning cell's genuine values is UNSAT — the existing EffectCommit anti-ghost teeth applied to the scene; binds the seen pixels to the producing cell's verified post-state
- FOCUS-EXCLUSIVITY INVARIANT (traceability of volition): at-most-one-focus + input-routed-only-to-focus as a state invariant proved over the compositor cell; delivering input to a non-focused cell is UNSAT (EROS's user-volition property as a dregg keystone)
- SECURE-ATTENTION PD + reserved gesture: a tiny trusted-path component holding the SOLE input cap and the SOLE top-zLayer surface cap; a kernel/host-routed gesture summons an unspoofable overlay reading owner+state-root straight from compositor-cell state (the SAK, the trust anchor, the only non-cell piece)
- VERIFIED LABEL = state-root binding: the per-surface label upgraded from Nitpicker's server-asserted name to a binding on the cell's verified state-root, so the label is an unfoolability claim a light client can independently check
- FRAME ATTESTATION (frontier): a verified display-driver PD that hashes what it scans out, binding panel-output digest to region+cell — the strongest output-integrity, gated on a named display-driver/IOMMU trust assumption (the graphics crypto-floor)
- DRIVER-AS-PD + sDDF transport: display and input drivers as unprivileged PDs holding only device-register/IRQ caps, exchanging framebuffers over async shared-ring transport — bounded verification scope; the firmament cap-gradation made physical for hardware

**First step**

Build the SOFTWARE compositor cell as a verified AppSpec in Lean, NOW, with no new hardware trust and runnable in starbridge-v2's embedded executor + the semihosted emulator. Concretely: define a `Compositor` cell whose state is a small fixed list of surfaces (owningCellId, regionRect as a few field-element slots, contentDigest, focusFlag, zLayer); write a `present(region, digest, ownerProof)` effect; instantiate the existing VerificationToolkit AppSpec so the admit-predicate is the conjunction (T1 non-overlap among distinct owners + T3 at-most-one-focus + label==owner). The toolkit then gives FOR FREE: app_commit_iff_admit (a present commits IFF it respects the scene invariants), app_violation_rejected (the anti-ghost tooth — an overpaint or a label-spoof or a double-focus is UNSAT), and the conservation/no-amplify keystones re-exported. Add #guard teeth that bite (a surface overpainting another cell's region REJECTS; a surface claiming a label ≠ its owner REJECTS; two focused surfaces REJECT) to prove non-vacuity. This is a self-contained, axiom-clean, verified-by-construction result that makes 'output-integrity = unfoolability on the scene' a theorem the same day, mirrors exactly how the 45+ effects were welded, and gives starbridge-v2 a real (not decorative) trusted scene model to render — the cipherclerk overlay reads its labels from this proven state. It explicitly defers F1/F2/F3 (the hardware last-hop) as named, lane-tracked frontier seams rather than blocking on seL4 graphics.

**Open questions**

- The last-hop binding (F1): can we get a frame attestation without a fully-verified GPU? Options — a verified CPU-blit display-driver PD that hashes its scanout (real but slow/no-accel), trust-the-GPU-with-an-IOMMU-cap (named assumption, needs seL4 IOMMU verification which doesn't exist yet), or a hardware display-controller that signs the content path (HDCP-like, needs vendor support). Which is the worthwhile semantics to drive toward, and what's the honest interim assumption?
- Region-set as a Rights lattice: is a rectangle-set the right object for is_attenuation, or do we need a Merkleized tile/quadtree so the in-circuit non-overlap check stays cheap (echoing the capability-crown's sorted-Poseidon2 Merkle reshape)? What's the proving cost of the non-overlap tooth at realistic surface counts?
- Continuous vs turn-based compositing: a 60fps display wants per-frame presents, but a turn+proof per frame is absurd. Does the proving-modality dial (prove-on-demand / checkpoint / eager, task #169) apply — prove the scene invariants only at trust boundaries (when the SAK overlay is summoned, or when a NEW surface/owner appears), and run unproven fast blits in between? What exactly must be proven at the moment the user looks vs continuously?
- Input-event provenance the other direction: T3 routes input to the focused cell, but can we BIND an input event to the cell's turn so the cell's receipt proves 'this turn was authorized by a user gesture at state-root R' (closing the loop — user volition becomes a witnessed premise in the cell's own proof, not just a compositor invariant)? Is this the input-side dual of frame attestation?
- The emulator vs seL4 trust gap: on the semihosted-mac/linux emulator the SAK is a host-delivered chord and the 'driver caps' are host abstractions — what part of the trusted-path guarantee is REAL there vs only-modeled, and how do we label that honestly (the same n=1-collapse discipline: the emulator gives the verified MODEL, seL4 gives the verified MECHANISM)?
- Covert channels in the shared compositor (EROS's concern): a shared scene graph + shared framebuffer can leak across cells via timing/contention/overlap-probing. EROS's double-buffering 'largely' eliminated display covert channels — what is the dregg analogue, and is it a provable information-flow property over the compositor cell or a named residual?

## SEMIHOSTED-SEL4 / the "robigalia emulator" — run the SAME seL4 cap-first programming model NATIVELY on mac/linux, with the rust-sel4 API as the same-code-targets-real-seL4 contract.

**Prior art**

- **gVisor (Sentry app-kernel)** — A user-space process (the Sentry, in Go) that re-implements the entire Linux syscall surface internally and never passes a guest syscall to the host kernel. Guest syscalls are redirected to the Sentry via three interchangeable 'platforms': ptrace, KVM, or Systrap (seccomp-based). The Sentry IS both guest kernel and VMM; from the host it is an ordinary process.  
  *dregg relevance:* THE closest architectural analog and the load-bearing pattern: a faithful re-implementation of a privileged ABI running as a host userspace process, with a pluggable 'platform' for how the guest's privileged ops are trapped. The semihost is exactly this for the seL4 syscall/IPC surface instead of Linux. gVisor's platform abstraction validates the design's central seam: the SAME guest code runs against different trap mechanisms (host-emulated vs real). dregg's win over gVisor: dregg already HAS the formal semantics (the executor is the verified Lean), so the emulator can be a faithful model rather than a from-scratch reimplementation.
- **User-Mode Linux (UML, the `um` architectural port)** — Linux ported to a NEW architecture (`um`) whose 'hardware' is the host's own syscall interface. The same kernel source compiles to `um` (boots as a host process, mmap's a big anon block as its 'physical RAM', builds its own page tables, intercepts guest syscalls via SKAS) OR to a real ISA (boots on metal). One codebase, two backends, selected at the arch/port layer.  
  *dregg relevance:* The cleanest precedent for the design's headline requirement — SAME-CODE-TARGETS-BOTH. UML proves a capability/OS layer can have a 'host' backend that is a first-class compile target alongside the real-hardware backend, with the seam drawn at a single well-defined port (the arch). For dregg the 'port' is the rust-sel4 API: a `semihost` cfg is the `um`-equivalent target, the real seL4 the metal target. UML's SKAS (separate-kernel-address-space, no host patching) is the model for isolating emulated PDs as host threads/processes without root.
- **rust-sel4 `sel4-musl` syscall shim** — An IN-TREE rust-sel4 crate (crates/experimental/sel4-musl) that installs a Rust `SyscallHandler` fn into musl's `__sysinfo` indirect-syscall hook, so musl's libc syscalls are serviced by Rust code running INSIDE the seL4 PD (no host kernel). `set_syscall_handler` / `set_raw_syscall_handler` are the exact interception points; Syscall::parse decodes the (sysnum, args).  
  *dregg relevance:* A direct, in-repo precedent for the interception mechanism AND an inversion of the same idea — sel4-musl emulates Linux syscalls inside a real seL4 PD; the semihost emulates seL4 syscalls on a real host. The `set_syscall_handler` pattern (a single function-pointer indirection that re-points the whole syscall floor) is precisely the seam the semihost installs at the `sel4::sys` layer. It is also the substrate the executor-PD already targets (WALL.md step 4), so the semihost and the real-seL4 executor-PD share a runtime shape.
- **seL4 CAmkES SIMULATION flag + capDL loader** — CAmkES builds each component to a separate ELF, packs them into a CPIO, links them against the capDL-initializer root task + the kernel, and the SIMULATION=1 flag emits a script that boots that image under QEMU. The capDL loader statically allocates all objects/caps from a declarative spec at boot.  
  *dregg relevance:* The NEGATIVE result that proves the niche is unfilled: seL4's own 'simulation' still runs the REAL verified kernel under full-system QEMU — there is NO host-native shim of the cap/IPC surface. So the robigalia emulator is genuinely new. BUT capDL is the model for the boot/init spec: the semihost's emulated-CSpace initial layout should be a capDL/DreggDL-shaped declarative spec (the firmament's RBG-TO-SEL4 §6 bridge), so the SAME spec instantiates caps on the host emulator and on real seL4.
- **WASM Component Model + WASI 0.2 (Preview 2)** — Components are isolated by SFI; a component can touch only resources the host explicitly grants (capability-based). WASI 0.2 redesigned all OS interfaces (clocks/files/sockets) as capability handles described in WIT, and wasmtime mediates inter-component calls through a typed canonical ABI (an RPC-like fabric over interface types + resource handles).  
  *dregg relevance:* The modern cap-IPC fabric analog and a design template for the emulated IPC layer: 'resource handles' = caps in a CSpace; the canonical-ABI call = an endpoint pp_call; 'no ambient authority, only granted handles' = exactly the firmament's §5 houyhnhnm contract. WIT-typed inter-component calls are a reference for typing the emulated endpoint message surface (MessageInfo + the IPC buffer) so host-side PD-to-PD calls are checked, and for the eventual same-code story where an app's required caps are a declared import list.
- **CHERI / Morello capability machines (+ hybrid-CHERI-on-seL4)** — ISA extensions that make pointers into unforgeable hardware capabilities carrying bounds + permissions + a validity tag; loads/stores are checked in hardware. Recent work runs hybrid-CHERI userspace under seL4 and plans to extend the seL4 endpoint cap with a permission gating whether CHERI caps may travel via IPC.  
  *dregg relevance:* The hardware ground-truth for 'unforgeable, attenuable, bounded reference' — the exact abstraction the firmament unifies (seL4 cap ≈ dregg cap ≈, ultimately, CHERI cap). Informs the emulator's cap representation (a tagged, bounds+rights-carrying handle that CANNOT be forged from raw bits in host memory — enforce this with a typed opaque handle + a generation/epoch tag, since host RAM has no tag bits) and points at the long-horizon target where the gradation extends DOWN to hardware caps, not just OUT to distributed cells.

**Design for dregg**

THE ROBIGALIA EMULATOR = a host-native backend for the rust-sel4 API surface, so one PD source tree runs (a) on the host emulator today and (b) on real seL4 unchanged. The contract (the 'port', UML-style) is the `sel4-microkit` API the dregg PDs ALREADY code against: the `#[protection_domain]` entry, the `Handler` trait (notified/protected/fault), `Channel` (notify / pp_call / irq_ack), `MessageInfo` + the IPC buffer, and `memory_region_symbol!`. Below that sits the lower seam, `sel4::sys` (the asm `svc` syscall helpers + CNode/Untyped invocations). The design draws the cut at BOTH layers and selects the backend by cfg.\n\nFOUR LAYERS.\n\n(1) THE EMULATED MICROKERNEL (`semihost-kernel`, a host library crate). A single in-process Rust object that owns: a set of CSpaces (one per emulated PD) each a slot-table of typed Caps with a derivation (mint/revoke) tree — this is LocalBacking from dregg-firmament, PROMOTED from a stub to the real CSpace model; Endpoint objects (a rendezvous queue: a blocked sender parks a MessageInfo + IPC-buffer payload until a receiver arrives, faithfully synchronous like seL4); Notification objects (a badge-word accumulator + a waiter); Untyped regions (a byte budget that Retype carves typed objects from). The kernel exposes exactly the invocations seL4 has — Call/Send/Recv/Reply/NBSendRecv, CNode Mint/Copy/Revoke/Delete, Untyped_Retype, Notification Signal/Wait — and enforces the SAME cap-rights and granted-subset-held checks (reusing dregg_cell::is_attenuation, as the firmament already does). It runs on macOS and Linux with std only; no privileged anything.\n\n(2) PDs AS HOST EXECUTION CONTEXTS. Each emulated PD is a host OS thread (v0; later optionally a process for real memory isolation). The Microkit run-loop (Handler::run, which today calls ipc::recv/reply_recv/nb_send_recv) is re-targeted: under the `semihost` cfg, `ipc::*` calls the emulated kernel's blocking endpoint/notification ops on this thread instead of issuing `svc`. The PD thread blocks on a host condvar exactly where it would block on seL4 Recv — preserving the event model (Notified / Protected / Fault) byte-for-byte. The `#[protection_domain]` macro under cfg(semihost) registers the init fn + heap with the emulator instead of emitting the bare-metal entrypoint.\n\n(3) THE IPC/CAP FABRIC ON THE HOST. Channels map to emulated Notification caps (cross-PD signal = condvar wake + badge-OR) and Endpoint caps (pp_call = synchronous rendezvous carrying a MessageInfo + the message-register slice). Shared memory regions (`memory_region_symbol!`) map to host shared buffers (a thread sees a real &mut [u8]; a process-backed PD sees an mmap/shm_open region) — so the net-client's smoltcp-over-shared-ring-buffer code runs against host shared rings unchanged. The fabric carries badges (the scope/membership values RBG-TO-SEL4 maps to) so the membership-bounded discovery and factory slot-caveat semantics hold identically.\n\n(4) THE FIRMAMENT AS THE LOCAL-DISTRIBUTED GRADATION, EXTENDED. dregg-firmament's Router already unifies LocalBacking (the CNode stub) with DistributedBacking (the real executor turn). The emulator IS the production-strength LocalBacking: the same `(target, rights)` handle, a `Target::Local{slot}` now resolves against the emulated kernel's real CSpace (synchronous, immediate-revoke — the strong n=1 bounds), and `Target::Distributed{cell}` still routes to the executor→net path. The verified executor (the WALL.md aarch64 musl ELF) hosts inside an emulated executor-PD: on the host it links the ordinary macOS/Linux Lean runtime (no libuv excision needed off-seL4), so the emulator gets a REAL verified heart immediately — the executor-PD blocker that gates real-seL4 does NOT gate the emulator. This is the portability fulcrum: best-on-mac/linux now (real executor + emulated cap fabric), seL4-native later (same PD code, the WALL.md musl port for the heart).\n\nTHE SAME-CODE SEAM (the contract). A PD crate depends on `sel4-microkit` and is `#![no_std]` for the real target; for the emulator we provide a `sel4-microkit`-API-compatible facade (a shim crate exporting protection_domain/Handler/Channel/MessageInfo/debug_println/memory_region_symbol with the SAME signatures) selected by a Cargo feature/cfg, std-backed. The PD's `fn init() -> impl Handler` and its notified/protected bodies are LITERALLY the same text. The acceptance bar: m0-hello, the rbg DirectoryCell PD (M2), and the net-client smoltcp PD compile AND run on the emulator and on real seL4 from one source, differing only by the backend cfg — proven by a shared integration test that boots the same PD set both ways.\n\nWHY THIS IS RIGHT, NOT A TOY. The honest-bounds discipline carries over: the emulator advertises Bounds::LOCAL (n=1: immediate revoke, synchronous commit, consistent checkpoint) and these are REAL on the host (a host thread's revoke IS synchronous), so the emulator is a faithful n=1 firmament, not a lossy mock. The cap checks are the genuine dregg attenuation. The one deliberate non-fidelity (host threads share an address space, so 'no ambient authority' is by-construction in the API, not MMU-enforced as on seL4) is exactly the gap the process-backed-PD upgrade and the real-seL4 target close — and it is the SAME gap UML's traced-thread→SKAS evolution faced, with the same fix.

**Key primitives**

- EmulatedKernel — the in-process host object owning all CSpaces/Endpoints/Notifications/Untypeds; the single source of truth, std-only, runs on mac+linux
- CSpace + Cap derivation tree — slot-indexed typed caps with mint(reduced-rights)/copy/revoke(transitive)/delete; PROMOTE dregg-firmament's LocalBacking from stub to full model; reuse dregg_cell::is_attenuation for the granted-subset-held gate
- Endpoint object — synchronous rendezvous queue carrying (MessageInfo, message-registers); a Call blocks until a Recv arrives (faithful seL4 synchrony) — backs Channel::pp_call
- Notification object — badge-word accumulator + waiter; Signal = badge-OR + wake, Wait = block-until-nonzero; backs Channel::notify and the Notified event
- Untyped region + Retype — a byte/object budget that mints exactly the declared object type; the kernel-enforced form of the factory slot-caveat (RBG-TO-SEL4 DirectoryFactory→Untyped_Retype)
- PD execution context — a host thread (v0) / process (v1, real isolation) running the Microkit run-loop; blocks on a host condvar exactly where seL4 Recv would block
- sel4-microkit API facade — a std-backed shim exporting protection_domain/Handler/Channel/MessageInfo/memory_region_symbol/debug_println with identical signatures, cfg-selected; THE same-code contract
- ipc backend shim — recv/reply/reply_recv/nb_send_recv re-targeted from asm-svc to the emulated kernel's blocking ops under cfg(semihost)
- Shared-memory region backend — memory_region_symbol! → a host shared buffer (thread: &mut[u8]; process: mmap/shm), so smoltcp-over-shared-ring-buffer code is unchanged
- Badge — the word carried on endpoint/notification caps encoding scope/membership/fault-flags (the firmament-boundary discriminator; matches Microkit's IS_ENDPOINT/IS_FAULT badge bits)
- DreggDL/capDL boot spec — the declarative initial cap layout instantiated identically by the host emulator and real seL4 CapDL (the same-spec-both-targets bridge)
- FirmamentRouter (existing) — the (target,rights) dispatcher whose Local backing becomes the emulated kernel and whose Distributed backing stays the real executor turn (the gradation, unchanged)

**First step**

Build the minimal semihost scaffold as a new standalone crate `sel4/semihost/` (its own workspace, like dregg-firmament, path-depending on dregg-cell/turn/types) with TWO pieces that together prove the same-code seam end-to-end:\n\n(A) `semihost-kernel`: lift dregg-firmament's `LocalBacking` into a real `EmulatedKernel` that adds, beside the CNode slot-table it already has, a synchronous `Endpoint` (rendezvous: Call parks until Recv) and a `Notification` (badge-OR + condvar wake). Keep it std-only; reuse `dregg_cell::is_attenuation` for mint. ~1 file, ~300 lines.\n\n(B) a `sel4-microkit` API facade exposing `protection_domain!`/`Handler`/`Channel`/`MessageInfo`/`debug_println` with the EXACT upstream signatures, backed by (A), where `Channel::notify` signals an emulated Notification, `Channel::pp_call` does an emulated endpoint Call, and the run-loop blocks on the kernel's Recv.\n\nTHE PROOF: take `dregg-pd/m0-hello/src/main.rs` and a TWO-PD slice (one PD `notify`s a second over a Channel; the second's `notified()` handler fires) and run that SAME source against the facade on the host — `cargo test` on macOS, no QEMU, no nightly, no build-std. Assert: the second PD's notified handler observed the badge; a `Channel::pp_call` round-trips a MessageInfo; and a CNode mint+revoke on the shared kernel enforces granted-subset-held (reusing the firmament's existing assertions). That single green test — the same PD text booting on the host emulator that today only boots under QEMU — is the keystone landed: it establishes the contract (the facade signatures) and the fabric (emulated endpoints/notifications) in one move, and everything else (process-backed PDs, the smoltcp shared-ring PD, hosting the verified executor-PD, the capDL boot spec) is additive against that base. Gate it with a shared integration test that later also boots the SAME PDs on real seL4, so the same-code claim is continuously checked, not asserted once.

**Open questions**

- Where to draw the PRIMARY seam: the high `sel4-microkit` API (fast, clean, but only same-code for Microkit-style PDs) vs the low `sel4::sys` syscall floor (covers root-task PDs + the sel4-musl-hosted executor too, but must emulate the IPC-buffer + invocation-decode wire format). Likely BOTH — microkit-facade for app PDs, a sel4::sys backend for the executor-PD — but which lands first and whether they share one EmulatedKernel.
- PD isolation model on the host: threads (v0, fast, but shared address space defeats the 'no ambient authority' MMU guarantee — fidelity gap) vs processes with shm_open/mmap regions (real isolation, faithful to seL4, but heavier IPC marshalling). When does the gradation/contract REQUIRE the process model, and is a thread-model emulator honestly labeled (Bounds say n=1-local but isolation is API-only not hardware)?
- How to make emulated caps genuinely UNFORGEABLE in host RAM (which has no CHERI tag bits): an opaque typed handle + generation/epoch + a kernel-side validity table, so a PD cannot fabricate a cap by writing raw bytes — vs accepting that the host emulator trusts in-process PD code and only the real-seL4/process backends enforce unforgeability in the substrate.
- Fault/exception fidelity: seL4 delivers a Fault to a PD's fault handler (the Handler::fault arm). What host event maps to it — a panic-catch turned into a synthetic fault MessageInfo? A guard-page SIGSEGV under the process model? How faithful must fault delivery be for the firmament's houyhnhnm-contract enforcement to be meaningful on the host.
- Time/RNG and houyhnhnm determinism on the host: the §5 contract says an app gets time/randomness only as a cap-mediated turn input. On the host, who supplies the emulated clock/entropy cap, and does the emulator enforce determinism (a recorded, replayable clock) or merely expose the real host clock (breaking replay)? The emulator is the natural place to make determinism real for free.
- Whether the emulated kernel should be DERIVED from / checked against the seL4 formal model (the kernel has an Isabelle spec; rust-sel4's invocation semantics are the de-facto contract) so the emulator is a faithful model of seL4's IPC/cap semantics rather than a hand-rolled approximation — i.e. is there a 'differential test the emulator against real seL4 under QEMU' loop (the model-finds-the-bug pattern), and is that the acceptance gate for 'faithful'.

# Pyana: ZK Objcap Tokens with Offline Cross-Silo Delegation

## Vision

A post-quantum zero-knowledge authorization system where capability tokens
are attenuated chains (each step can only narrow, never expand), verified
via recursive STARKs. The core insight: capability delegation *is* IVC —
each attenuation is a fold step proving valid narrowing.

Holders prove predicates ("I can do X") without revealing identity, chain
depth, issuing authority, or other capabilities. Fully offline. No liveness
dependency on the issuer. Post-quantum.

## Architecture (6 layers)

```
┌─────────────────────────────────────────────────────┐
│ 6. Joining Protocol                                 │
│    Capability introduction via bridge chains.       │
│    Two silos interop the instant someone provides   │
│    a bridge connecting their trust roots.           │
├─────────────────────────────────────────────────────┤
│ 5. Federation Attestation (Morpheus)                │
│    BFT consensus over revocation roots.             │
│    Adaptive: leaderless (quiescent) ↔ leader DAG.   │
│    Produces signed (root, timestamp) pairs.         │
├─────────────────────────────────────────────────────┤
│ 4. Revocation (Monotonic Accumulator)               │
│    Append-only per-authority Merkle tree.            │
│    Non-membership proof folded into presentation.   │
│    Revoker doesn't enumerate subtree; proof system  │
│    enforces CDT transitivity.                       │
├─────────────────────────────────────────────────────┤
│ 3. Authorization Circuit (STARK)                    │
│    Prove valid Datalog derivation trace offline.    │
│    Prover runs evaluation; circuit checks trace.    │
│    ~1-2k constraints/step for fixed-arity Datalog.  │
├─────────────────────────────────────────────────────┤
│ 2. Token Chain (IVC / Recursive STARK)              │
│    Each attenuation = fold step.                    │
│    State = Poseidon commitment to (facts, rules).   │
│    Final token = (commitment, proof, pub inputs).   │
├─────────────────────────────────────────────────────┤
│ 1. Policy Model (pyana-token, exists now)           │
│    AuthRequest, TokenClearance, Attenuation, etc.   │
│    Macaroon caveats + Biscuit Datalog semantics.    │
│    Reference evaluator whose traces we prove.       │
└─────────────────────────────────────────────────────┘
```

## What Needs Building

### pyana-circuit (the hard part)

The STARK circuit that proves:
1. **Chain validity:** IVC over attenuation steps. Step function:
   `F(state, caveat) → state'` where state is Poseidon(facts ∪ rules).
   Must prove new state is a valid narrowing of old state.

2. **Authorization:** Given committed (facts, rules), prove a Datalog
   derivation trace leads to `allow` for a given request. The trace is
   the witness; the request is public input. Per-step: unification check +
   Merkle membership over fact set.

3. **Non-revocation:** For each ancestor in the chain, prove non-membership
   in the revocation accumulator. Accumulator root is public input (freshness
   bound enforced by verifier policy).

### PQ threshold signatures (replace hints)

Morpheus needs threshold attestation but `hints` uses BLS12-381. Options:
- Hash-based multisig (SPHINCS+ variants?)
- Lattice-based threshold (academic, immature)
- STARK-based: each signer produces a STARK proof of "I signed with my key",
  aggregate via recursive composition. Heavy but definitely PQ.

### Federation tree aggregation

Authorities form trust hierarchies. A federation node collects member roots
into an aggregate Merkle root. Verifier tracks one root per federation.
Cross-federation = bridge chain (itself a folded proof).

## Key Design Decisions (Open)

### STARK backend choice
- **Plonky3**: Fast, flexible, Rust-native. Actively developed.
- **Binius**: Binary fields — good for hashing/equality. Very new.
- **RISC Zero / SP1**: Compile reference evaluator to RISC-V, prove execution.
  Easiest dev path but large circuits.
- **Custom AIR**: Write the Datalog step directly as an AIR. Most efficient
  but most work.

Leaning: custom AIR on Plonky3 for the step function. The Datalog evaluation
is structured enough that a zkVM is overkill.

### Hash function for commitments
- Poseidon: ~250 constraints in R1CS, well-studied. Algebraic.
- Rescue Prime Optimized (RPO): Similar, used in Miden.
- For STARKs over binary fields (Binius): different tradeoffs.

### Proof size vs proving time
- Final presentation proof target: <500KB (network-transmissible token)
- Proving time target: <10s on commodity hardware (acceptable for offline)
- Verification: <50ms (must be fast for verifier)

### Revocation freshness model
- Verifier supplies `max_age` parameter (e.g., 30s, 1h, 24h)
- Prover includes signed `(root, timestamp)` from their federation
- Circuit checks timestamp > now - max_age (now is public input)
- Tradeoff is explicit and per-deployment

## Analogous Systems to Study

- **Semaphore**: Merkle-tree anonymity set, ZK group membership. Curve-based
  but the pattern (prove membership without revealing identity) applies.
- **Mina**: Recursive SNARK chain. Curve-based but the recursive composition
  pattern is exactly what we want (STARK version).
- **StarkNet recursive proofs**: STARK-verifying-STARK in production today.
- **Coconut / ACL**: Anonymous credentials with selective disclosure.
  Not composable across issuers (our key differentiator).
- **Certificate Transparency**: Append-only Merkle logs with gossip.
  Same model for revocation propagation.
- **Robigalia CDT**: The original objcap revocation problem this solves.

## The Objcap Framing

In a traditional objcap system, cross-silo delegation has three bad modes:
1. Live relay (issuer must be online → availability dependency)
2. Bearer token (verifier sees full chain → authority graph leaked)
3. Revocation callback (phone home → liveness + tracking)

Pyana gives mode 4: **offline, private, composable delegation.**
- Present a proof of capability without revealing the chain
- Issuer doesn't know you used it
- Verifier doesn't learn the chain
- Revoker doesn't need to enumerate subtree
- Two silos interop via bridge chains, no prior coordination
- All post-quantum

This is the first system combining: PQ + composable attenuation +
cross-domain anonymous delegation + effective revocation.

## Immediate Next Steps

1. **Sketch the step function AIR** — what does F(state, caveat) → state'
   look like concretely? Which Plonky3 primitives does it map to?
2. **Prototype Datalog trace verification** — write the reference trace
   checker in plain Rust, then figure out how to arithmetize it.
3. **Evaluate STARK recursion maturity** — can Plonky3 verify its own proofs
   today? What's the overhead? Is SP1/RISC Zero more practical near-term?
4. **Design the accumulator** — sparse Merkle tree with Poseidon? What's
   the non-membership proof cost?
5. **PQ threshold sig research** — is there anything usable today or do we
   need STARK-based attestation from day one?

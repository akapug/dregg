## self-hosting-loop

The SELF-HOSTING LOOP is deos developing dregg inside itself: a source edit made in the firmament editor is a verified ledger turn, and the deos terminal's toolchain compiles that very edit. The view that mounts both halves is `starbridge_v2::self_hosting::SelfHostingView`; the headless bake is `--render-self-hosting` (two halves, one view) and `--render-self-hosting-full` (the full single loop).

What it shows: two real panes side by side. The EDITOR pane (`dock::editor_surface::EditorPane::firmament_over`) edits a sovereign *cell*, not a disk file; a save sets the buffer and commits a cap-gated `SetField` turn through the verified executor, leaving a genuine `TurnReceipt` — the status line's save count IS the live on-ledger receipt count, read off the same ledger the cockpit's cell inspector reads. The TERMINAL pane (`dock::terminal_surface::TerminalPane::spawn`) runs a real child on a real alacritty PTY; its genuine stdout lands in the cell grid.

How the FULL loop closes: the editor saves to CELLS (ledger turns) while `cargo`/`rustc` read DISK. The FirmamentFs↔disk dual-write bridges them — after the verified turn commits (the cell is the durable, receipted source of truth), `save` ALSO writes the decoded content to an optional disk mirror (`<root>/<path>`), a derived read-mirror the legacy disk-reading toolchain compiles from. A mirror write error is fail-loud, so disk can never silently desync from the ledger.

The proof is in the bake, not the description: `--render-self-hosting-full` hard-asserts THREE facts (non-zero exit otherwise) — the save grew the live `TurnReceipt` count (a real verified turn), the cell's new `v2` content was dual-written to `<dir>/main.rs`, and a live `sh` PTY ran `rustc main.rs -o prog && ./prog` over the mirrored file and the compiled program printed `v2`. The captured frame (`starbridge-v2/self-hosting-loop-full.png`, 3200×2000) shows line-numbered syntax-highlighted code beside the terminal that compiled it, under the live image header reading `6 receipts · on-ledger`. (Runbook: `docs/deos/SELF-HOSTING-LOOP.md`; reproduce index: `docs/deos/DEOS-RUNS.md`.)

## zed-workspace

The ZED WORKSPACE surface is the full Zed IDE running over the dregg cell ledger: a real `workspace::Workspace` instantiates over a `Project` whose `Fs` IS the cell ledger, so the IDE's own panels browse cells-as-files and a save through the IDE is a verified ledger turn. It lives in the `deos-zed-full` crate, a separate workspace that links Zed unchanged (both it and starbridge-v2 pin the same `zed` rev → one `gpui`/`dregg-sdk`); starbridge-v2 is unaffected.

What it shows: Zed's OWN `project_panel` + `outline_panel` + `terminal_panel` each `Panel::load` + dock + resolve (all `Some`); the project panel lists the ledger's cells as files; a cell opened as a `language::Buffer`, edited, and `project.save_buffer`'d through the WORKSPACE fires a verified `TurnReceipt` — a save from the whole IDE is a turn. Nested-dir expansion works (a 2-deep cell tree `/proj/src/inner/deep.rs` materializes via the real worktree scan + click-to-expand), and a real OS PTY shell opens in the docked `TerminalPanel` (Zed's own `add_center_terminal`). A confined Hermes agent docks as a real `workspace::Panel` (`HermesPanel`) alongside the others — an agent over the cell-ledger inside the IDE, every tool-call cap-gated through `HermesGateway::admit_call`.

How you use it / what it proves: this is the IDE-shaped face of the self-hosting story — project + outline + terminal PTY + a working confined agent, all over the verified ledger. The proof is the green `full-zed` test suite (`cd deos-zed-full && cargo test --features full-zed`): the workspace instantiates, the panels resolve, a save fires a real receipt. A subtle trap surfaced and was fixed here — every cell reporting `inode:0` made Zed's worktree scanner treat nested cells as symlink cycles; a distinct stable per-path inode fixed the recursion. (This surface carries no committed screenshot; the demonstration is the test suite, not a bake.)

## web-deos

WEB DEOS is the same gpui cockpit as native, bundled for and booted in a browser tab: the real `gpui` element-tree renderer on the `gpui_web` platform backend (wasm32 + WebGPU canvas), driving the same in-browser verified executor the native desktop drives. It is not a lesser "web skin" — it is one renderer, one model, two platforms (native windowing vs. a browser canvas).

What it shows: the wasm entrypoint `boot_cockpit()` (`starbridge-v2/web/src/cockpit_web.rs`) seeds the same `world::demo_genesis` the native cockpit boots, boots a single-threaded web gpui app, registers the cockpit fonts, calls `gpui_component::init(cx)`, opens a window (creating the WebGPU canvas), and mounts the REAL `starbridge_v2::cockpit::Cockpit` — the exact `Cockpit::with_node(world, anchors, focus, None, seed)` the native `run_window` constructs, with the in-tab executor as the data plane. Every affordance click is a real cap-gated turn through the verified executor in-tab (the `World::commit_turn` seam), the post-state re-reflected on the next frame.

What it proves (honestly): the bundle (`web/pkg-gpui/`, via `build-gpui.sh` → `wasm-bindgen --target web`, exporting `boot_cockpit`) loads in headless Chrome, `boot_cockpit()` is invoked, the `gpui_web` platform initializes WebGPU successfully, and the run-loop (rAF + ResizeObserver) PAINTS a real cockpit frame on a live WebGPU canvas at full device-pixel resolution — the verification reports `canvas backing max : 2560x1640 (PAINTED a real frame)` and `run-loop reentrancy : false`. The captured frame is `starbridge-v2/web/cockpit-gpui-web-painted.png`. The earlier first-paint ceiling (`closure invoked recursively or after being dropped`, canvas stuck 1×1) was a lifetime bug — `Application::run` dropped the owning `Rc<AppCell>` when it returned (web `run` is fire-and-forget) while the browser still held the `WebWindow`'s rAF/ResizeObserver closures; the fix leaks the owning `Rc` on wasm so the app lives for the page's lifetime.

The honest distance: the paint fix currently lives in a LOCAL `emberian/zed` fork checkout wired via a `[patch]` in `starbridge-v2/Cargo.toml`; the cutover is to push the one-line fix upstream, bump the rev, and drop the `[patch]`. The native-resource backends (terminal PTY, editor Fs, web-shell servo) each still need their web wire per the per-app backend map. (Doc: `docs/deos/WEB-DEOS.md`.)

## mud

The MUD MULTI-INHABITANT surface is a small multiplayer world where the PHYSICS is the proof: 3 inhabitants across 2 connected rooms, every rule enforced by the real embedded executor's own gates via real receipts, not by Rust bookkeeping. It is captured by the `mud` test suite (9 tests, ~76s of real proving) — `cd starbridge-v2 && cargo test --features native-full mud`.

What it shows (four executor-enforced properties): movement is cap-gated — `move_through` writes the destination room cell, so it passes `check_cross_cell_permission`; without a door cap the move is refused with `CapabilityNotHeld` ("a door is a cap you lack"). An item is conserved across a multi-hop trade and exactly one contender wins (no duplication; the `mint_needs_held_factory` grant/revoke). An authority-amplification on a `give` is refused (`gen_conferral_is_attenuation` — you cannot hand someone more than you hold). Value conserves: `Effect::Transfer` keeps Σδ=0, an overdraft is refused.

What it demonstrates: that a believable interactive game world built on dregg gets its "rules of physics" for free from the protocol's existing teeth — conservation, attenuation, cap-gated writes — and that the refusals are genuine executor verdicts on real receipts. (Seam: presence today is the door-cap-gated write — entry cap-gating is proven; a fuller presence-token move/grant is future. This surface carries no committed screenshot; the demonstration is the test suite.)

## membrane

The MULTIPLAYER MEMBRANE is the fork-and-stitch primitive — the killer move where a *message* is a cap-bounded world-fork you drive and stitch back. It is captured by the `shared_fork` suite (21 tests on the real embedded executor) — `cd starbridge-v2 && cargo test --features native-full shared_fork`.

What it shows: ONE minted `MembraneFrustum` (a cap-bounded `World::fork` cull — the screenshot-of-the-moment) is carried over the postcard wire shape (the `MembraneEnvelope.snapshot` bytes), then rehydrated into TWO independent real `World`s under TWO distinct principals — both `frustum_root`-matched, which is the anti-substitution check (you cannot swap in a different world behind the frustum). Each principal drives a real verified `commit_turn` (one overlapping conflict + one disjoint private edit). The two are stitched via `Stitch::settle`: the disjoint case is a clean pushout-merge; the overlapping divergence resolves by the linear Dead-wins join TRANSPARENTLY (not a silent last-writer-wins); an over-authorized confer is REFUSED as a lossy drop (`SettleOutcome::Refused`). Σδ=0 and authority stay sound throughout.

What it demonstrates: distributed time-travel with consent — branch a bounded view of the world, let two parties evolve it independently, and merge with conflicts surfaced as first-class objects rather than overwritten, while conservation and the attenuation lattice hold across the whole fork→drive→stitch arc. (This surface carries no committed screenshot; the demonstration is the test suite. The same `MembraneEnvelope` also round-trips byte-intact through a real Matrix homeserver — see the chat run in `docs/deos/DEOS-RUNS.md`.)

## federation

The FEDERATION surface is a real multi-node `dregg-node` deployment over QUIC gossip + blocklace consensus: two nodes sharing a 2-validator genesis form an n=2 committee, and a turn submitted on one node finalizes on the other. Nothing here is a mock — both nodes run the verified Lean producer, exchange real Ed25519-signed blocks, and converge to a byte-identical DAG.

What it shows: the committee is established by a SHARED `genesis.json` carrying both validators' public keys (the same committee → the same federation_id → the same participants). With both nodes running `--federation-mode full` and cross-pointing gossip ports, a faucet turn on node A gossips to B; within ~5s both `/api/blocklace/blocks` return the IDENTICAL 6-block DAG, `latest_height=1` on both (the turn block super-ratified cross-node), `dregg_consensus_attested_total 1` on BOTH (consensus-wide Attested), and the recipient's balance = 5000 on both nodes — the effect applied through consensus finalization, not locally.

Robustness: each node runs a peer reconnect prober (`blocklace_sync::spawn_peer_prober`) so a peer DOWN at boot (or one that drops and returns) still converges without an operator restart, and a bounded 3s initial dial means a down peer no longer stalls startup. Discovery is authenticated gossip-of-peers: a node shares cryptographically-verified `peer-identity → listen address` bindings, and a receiver accepts an address ONLY for a key already in its genesis-trusted committee — so the mesh forms transitively from a single seed while a forged Sybil binding is rejected.

How you reproduce it: the full staged recipe (mint a 2-validator genesis, build two data dirs, run both nodes, faucet on A) is in `docs/deos/DEV-NODE-RUNBOOK.md`; the reconnect / discovery / quorum paths are pinned by named `net`/`node` tests (`late_join_prober`, `gossip_of_peers_transitive`, `gossip_of_peers_accepts_committee_rejects_forged`, `quorum_crossing_is_reported_on_whichever_vote_is_second`). (This surface carries no committed screenshot; the demonstration is the runbook + the tests.)

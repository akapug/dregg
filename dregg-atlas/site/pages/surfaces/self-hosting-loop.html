SELF-HOSTING LOOP surface

Develop dregg INSIDE deos: the firmament editor (a save is a cap-gated SetField ledger turn) beside a live alacritty PTY running real cargo/rustc — one image. The bake hard-asserts the receipt grew AND the toolchain ran; src `starbridge-v2/self-hosting-loop-full.png`.

The SELF-HOSTING LOOP is deos developing dregg inside itself: a source edit made in the firmament editor is a verified ledger turn, and the deos terminal's toolchain compiles that very edit. The view that mounts both halves is starbridge_v2::self_hosting::SelfHostingView; the headless bake is --render-self-hosting (two halves, one view) and --render-self-hosting-full (the full single loop).

What it shows: two real panes side by side. The EDITOR pane (dock::editor_surface::EditorPane::firmament_over) edits a sovereign *cell*, not a disk file; a save sets the buffer and commits a cap-gated SetField turn through the verified executor, leaving a genuine TurnReceipt — the status line's save count IS the live on-ledger receipt count, read off the same ledger the cockpit's cell inspector reads. The TERMINAL pane (dock::terminal_surface::TerminalPane::spawn) runs a real child on a real alacritty PTY; its genuine stdout lands in the cell grid.

How the FULL loop closes: the editor saves to CELLS (ledger turns) while cargo/rustc read DISK. The FirmamentFs↔disk dual-write bridges them — after the verified turn commits (the cell is the durable, receipted source of truth), save ALSO writes the decoded content to an optional disk mirror (<root>/<path>), a derived read-mirror the legacy disk-reading toolchain compiles from. A mirror write error is fail-loud, so disk can never silently desync from the ledger.

The proof is in the bake, not the description: --render-self-hosting-full hard-asserts THREE facts (non-zero exit otherwise) — the save grew the live TurnReceipt count (a real verified turn), the cell's new v2 content was dual-written to <dir>/main.rs, and a live sh PTY ran rustc main.rs -o prog && ./prog over the mirrored file and the compiled program printed v2. The captured frame (starbridge-v2/self-hosting-loop-full.png, 3200×2000) shows line-numbered syntax-highlighted code beside the terminal that compiled it, under the live image header reading 6 receipts · on-ledger.

THE UNIFIED BOOT is one window holding three panes at once: a LIVE pane attached to a real running dregg-node over the wire, the FirmamentFs editor, and a live PTY terminal — the cockpit's own panes standing alongside a real node, not a mock. The view is starbridge_v2::unified_boot; the headless bake is --render-unified-boot in main.rs, which wires all three over a node listening on :8775 and captures the frame deos-unified-boot.png.

What it shows (three panes, one frame): the LIVE NODE pane is a real crate::client::LiveNode::sync snapshot of the attached node — its /status (lean producer, height, DAG), every live cell from /api/cells, and the latest receipt from /api/receipts, each projected through the SAME crate::live_node::LiveReflection uniform reflective model the cockpit's own live-node strip uses. That is what makes the attach demonstrably LIVE rather than embedded: the cells and the receipt shown are the node's, pulled over HTTP at build time. The EDITOR pane (dock::editor_surface::EditorPane::firmament_over) edits the cockpit's LOCAL World; a save is a cap-gated SetField turn on that local ledger. The TERMINAL pane runs a real child on a live alacritty PTY inside deos.

The honest seam (reported by the bake, not papered over): an editor save lands on the cockpit's LOCAL World — EditorPane::firmament_over commits through WorldSpine → World::commit_turn — while the --node attach is read-only-synced. The bake settles this empirically: it drives a real editor save, then re-reads the node's receipt count and finds it unchanged (node receipts 1→1), so the save did NOT reach the node's ledger. To make a save a real over-the-wire self-hosting write-back, route the FirmamentFs save through NodeClient::submit_turn (the designed-pending write surface) with local key custody to sign; the shape is WIRE, not build (the submit surface already exists; the editor's LedgerSpine would gain a node-routing arm). (Named in HORIZONLOG.md under the unified-boot landing, commit ff7a0da0c.)

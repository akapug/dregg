WEB DEOS (in-browser) surface

The same gpui cockpit bundled to wasm32 + WebGPU, painting a real frame in a browser tab over the same in-tab verified executor — one renderer, two platforms. src `starbridge-v2/web/cockpit-gpui-web-painted.png` (headless Chrome, canvas 2560×1640).

WEB DEOS is the same gpui cockpit as native, bundled for and booted in a browser tab: the real gpui element-tree renderer on the gpui_web platform backend (wasm32 + WebGPU canvas), driving the same in-browser verified executor the native desktop drives. It is not a lesser "web skin" — it is one renderer, one model, two platforms (native windowing vs. a browser canvas).

What it shows: the wasm entrypoint boot_cockpit() (starbridge-v2/web/src/cockpit_web.rs) seeds the same world::demo_genesis the native cockpit boots, boots a single-threaded web gpui app, registers the cockpit fonts, calls gpui_component::init(cx), opens a window (creating the WebGPU canvas), and mounts the REAL starbridge_v2::cockpit::Cockpit — the exact Cockpit::with_node(world, anchors, focus, None, seed) the native run_window constructs, with the in-tab executor as the data plane. Every affordance click is a real cap-gated turn through the verified executor in-tab (the World::commit_turn seam), the post-state re-reflected on the next frame.

What it proves (honestly): the bundle (web/pkg-gpui/, via build-gpui.sh → wasm-bindgen --target web, exporting boot_cockpit) loads in headless Chrome, boot_cockpit() is invoked, the gpui_web platform initializes WebGPU successfully, and the run-loop (rAF + ResizeObserver) PAINTS a real cockpit frame on a live WebGPU canvas at full device-pixel resolution — the verification reports canvas backing max : 2560x1640 (PAINTED a real frame) and run-loop reentrancy : false. The captured frame is starbridge-v2/web/cockpit-gpui-web-painted.png. The earlier first-paint ceiling (closure invoked recursively or after being dropped, canvas stuck 1×1) was a lifetime bug — Application::run dropped the owning Rc<AppCell> when it returned (web run is fire-and-forget) while the browser still held the WebWindow's rAF/ResizeObserver closures; the fix leaks the owning Rc on wasm so the app lives for the page's lifetime.

The honest distance: the paint fix currently lives in a LOCAL emberian/zed fork checkout wired via a [patch] in starbridge-v2/Cargo.toml; the cutover is to push the one-line fix upstream, bump the rev, and drop the [patch]. The native-resource backends (terminal PTY, editor Fs, web-shell servo) each still need their web wire per the per-app backend map. (Doc: docs/deos/WEB-DEOS.md.)

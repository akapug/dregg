The ZED WORKSPACE surface is the full Zed IDE running over the dregg cell ledger: a real workspace::Workspace instantiates over a Project whose Fs IS the cell ledger, so the IDE's own panels browse cells-as-files and a save through the IDE is a verified ledger turn. It lives in the deos-zed-full crate, a separate workspace that links Zed unchanged (both it and starbridge-v2 pin the same zed rev → one gpui/dregg-sdk); starbridge-v2 is unaffected.

What it shows: Zed's OWN project_panel + outline_panel + terminal_panel each Panel::load + dock + resolve (all Some); the project panel lists the ledger's cells as files; a cell opened as a language::Buffer, edited, and project.save_buffer'd through the WORKSPACE fires a verified TurnReceipt — a save from the whole IDE is a turn. Nested-dir expansion works (a 2-deep cell tree /proj/src/inner/deep.rs materializes via the real worktree scan + click-to-expand), and a real OS PTY shell opens in the docked TerminalPanel (Zed's own add_center_terminal). A confined Hermes agent docks as a real workspace::Panel (HermesPanel) alongside the others — an agent over the cell-ledger inside the IDE, every tool-call cap-gated through HermesGateway::admit_call.

How you use it / what it proves: this is the IDE-shaped face of the self-hosting story — project + outline + terminal PTY + a working confined agent, all over the verified ledger. The proof is the green full-zed test suite (cd deos-zed-full && cargo test --features full-zed): the workspace instantiates, the panels resolve, a save fires a real receipt. A subtle trap surfaced and was fixed here — every cell reporting inode:0 made Zed's worktree scanner treat nested cells as symlink cycles; a distinct stable per-path inode fixed the recursion.
